# E4.4 — Leaderboards

## Purpose

Deliver the optional live leaderboard: a per-quiz, faculty-enabled ranking of participants that
refreshes in real time as answers are scored during the session (PRD §4.4 "Real-time Leaderboard
(optional)", PRD §7 "Leaderboards refresh live"). Ranking state is Redis-backed
(`leaderboard:{quizId}`, folder-structure §9) so it survives horizontal scaling and stays cheap to
read at broadcast frequency (PRD §12 "Live leaderboard updates (optional)").

## Business Value

Live leaderboards add the engagement dimension that distinguishes an interactive quiz session from
a silent exam — a classroom-energy feature faculty can opt into for formative or gamified
assessments, and switch off for high-stakes exams. Because it is strictly optional per quiz, it
adds value without compromising assessment settings where ranking pressure is unwanted (PRD §4.4).

## Dependencies

- E4.1: connections, rooms, and the event catalog — leaderboard updates are broadcast to the quiz
  room.
- E4.2: session lifecycle — the leaderboard exists only within a live session window.
- Phase 3: objective auto-grading — live ranking requires scores computable at answer time; only
  objectively graded questions (MCQ, multiple select, true/false, numerical — PRD §10) can
  contribute during the session. Descriptive answers are manually evaluated later and cannot rank
  live.
- Phase 2 quiz settings: the per-quiz leaderboard on/off toggle. **Assumption:** this toggle is a
  quiz-configuration field owned by `modules/quizzes/` (PRD §4.4 marks the feature "optional" but
  does not say where it is configured).
- Redis (`server/cache/`) for the ranking structure (folder-structure §9).

## Acceptance Criteria

- On quizzes where the leaderboard is enabled, every participant and the faculty dashboard see
  rankings update live as answers are submitted and auto-graded, without page refresh (PRD §7).
- On quizzes where it is disabled, no leaderboard data is computed, broadcast, or visible to any
  role (PRD §4.4).
- Leaderboard state is Redis-backed under the cataloged `leaderboard:{quizId}` key and consistent
  across server instances (folder-structure §9, PRD §16 Scalability).
- Leaderboard reads/writes are tenant- and quiz-scoped; a student never sees another quiz's
  leaderboard (PRD §16 Security).
- **Needs clarification:** scoring/ranking rules are unspecified in the PRD — score only? speed
  tiebreak? penalty for wrong answers? This must be decided before implementation (tracked in
  F4.4.1 Open Questions).
