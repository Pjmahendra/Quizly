# E4.1 — Real-Time Infrastructure

## Purpose

Stand up the Socket.IO foundation everything else in Phase 4 depends on: a bootstrapped socket
server with session-authenticated, tenant-scoped connections (`server/socket/`), per-quiz room
management with reconnection recovery (PRD §7 "Connection recovery is supported"), and the single
canonical event catalog with typed payloads in `modules/socket/events/` (folder-structure §5, §9).
This epic delivers infrastructure and vocabulary only — domain behavior (session orchestration,
monitoring, integrity) is built on top of it in E4.2–E4.4.

## Business Value

The persistent live connection is the PRD's primary differentiator (PRD §7): without it there is
no live monitoring, no faculty control, no integrity enforcement, and no live leaderboard. Getting
authentication, tenancy, rooms, and recovery right once — behind clean seams — lets every later
feature be a handler over trusted infrastructure instead of re-solving connection problems.
Reliable recovery directly protects assessment fairness: a dropped connection must never cost a
student their attempt (PRD §16 Reliability).

## Dependencies

- Phase 1: authentication/session plumbing (`server/auth/`) and tenant model — the handshake
  reuses them; no separate socket credential is introduced.
- Phase 3: attempt lifecycle — room membership is keyed to an active attempt on a quiz.
- Redis (`server/cache/`) for cross-instance socket state and the `socket-room:{quizId}` /
  `active-users:{quizId}` keys (folder-structure §9); required for horizontal scale (PRD §16).
- **Blocking decision:** realtime hosting — Vercel (PRD §17) cannot host long-lived Socket.IO
  connections. Must be resolved before F4.1.1 implementation (see phase README, Risks).

## Acceptance Criteria

- A logged-in participant's client establishes an authenticated socket connection tied to their
  existing session; unauthenticated or cross-tenant connection attempts are rejected at the
  handshake (PRD §16 Security).
- Each live quiz has its own room; participants join and leave it correctly, and membership
  survives server horizontal scaling via the Redis adapter (PRD §16 Scalability).
- A student who disconnects and reconnects mid-quiz is restored to the correct room with resynced
  state (current question, answers so far, remaining time) without losing attempt progress
  (PRD §7).
- Every event name used anywhere in the codebase resolves to the catalog in
  `modules/socket/events/`; payloads are typed and validated (folder-structure §5, §9).
- Disconnect/reconnect signals are kept distinct from integrity (focus-loss) signals end to end
  (PRD §13).
