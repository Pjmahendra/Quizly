# E4.2 — Live Execution & Faculty Control

## Purpose

Turn Phase 3's request/response quiz into a live, faculty-directed session. This epic delivers the
live session lifecycle (`quiz-started` → `quiz-ended`) with server-authoritative timer
synchronization and late-join handling (PRD §7), the full set of faculty in-quiz controls — pause,
resume, end, extend time, lock/unlock, remove student (PRD §4.3 "During Live Quiz") — and
broadcast announcements to all participants (PRD §4.3, §7). Every control action propagates to
affected students in real time over the E4.1 infrastructure.

## Business Value

Live control is what makes Quisly a classroom instrument rather than a form submitter: faculty can
respond to real conditions (a fire alarm, a projector question, a disruptive participant) without
abandoning the assessment. Server-authoritative timing guarantees fairness — every student sees
the same clock regardless of device or network — and instant propagation of controls gives faculty
confidence that what they see is what students experience (PRD §7, §16 Performance).

## Dependencies

- E4.1 in full: authenticated connections (F4.1.1), rooms and recovery (F4.1.2), and the event
  catalog (F4.1.3) — all lifecycle and control events travel through them.
- Phase 3 attempt lifecycle: start/submit/auto-submit services in `modules/attempts/` are invoked
  by lifecycle transitions (e.g., quiz end forces submission of open attempts).
- Phase 2 quiz scheduling/publishing: a session can only go live for a published, scheduled quiz.
- Redis for authoritative session state (pause/extend/lock) shared across instances
  (folder-structure §9, PRD §16).

## Acceptance Criteria

- Starting a quiz emits `quiz-started` to the quiz room; ending it emits `quiz-ended` and closes
  open attempts through the Phase 3 auto-submit path (PRD §7, §8).
- The server broadcasts `timer-sync` so all participants converge on the same remaining time; a
  late joiner or reconnecting student receives the correct remaining time immediately (PRD §7).
- Pause, resume, end, extend time, lock, unlock, and remove-student each take effect for the
  affected students in real time, and each action is attributable to the acting faculty member
  (PRD §4.3, §16 Audit Logs).
- A removed student cannot rejoin the session; a locked quiz admits no new joiners while allowing
  existing participants to continue (PRD §4.3). **Assumption:** lock blocks new entry only —
  the PRD does not define lock semantics; confirm before implementation.
- Faculty announcements reach every connected participant instantly and are visible to students
  without disrupting their in-progress answers (PRD §4.3, §7).
- All control paths enforce faculty ownership of the quiz and tenant scope — no cross-quiz or
  cross-organization control is possible (PRD §16 Security).
