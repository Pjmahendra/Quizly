# F4.2.1 — Live Session Orchestration & Timer Sync

**Epic:** E4.2 — Live Execution & Faculty Control · **Complexity:** Large

## Description

Orchestrate the live session lifecycle over the E4.1 infrastructure: when a quiz goes live,
`quiz-started` is broadcast to the quiz room; when it ends (faculty action, scheduled end, or time
expiry), `quiz-ended` is broadcast and open attempts are closed through the Phase 3 auto-submit
path (PRD §7, §8). The server is the single time authority: it broadcasts `timer-sync` so every
participant converges on the same remaining time regardless of local clock drift (PRD §7 "Timer
synchronization occurs automatically"). Late joiners — students joining after start, within
whatever the quiz's join window allows — receive the current session state and correct remaining
time on entry. Domain logic lives in `modules/socket/handlers/quiz-events` orchestrating
`modules/quizzes/` and `modules/attempts/` services (folder-structure §5).

## Dependencies

- F4.1.1–F4.1.3 (connections, rooms/recovery, event catalog).
- Phase 3: attempt start/submit/auto-submit services; Phase 2: quiz schedule/duration settings.
- Redis for authoritative session state (phase, started-at, end-at) shared across instances.

## Database Changes (conceptual)

- The quiz session's lifecycle facts (actual start time, actual end time, how it ended) must be
  durably recorded — conceptually an extension of existing quiz/attempt records rather than a new
  domain. **Assumption:** live-session runtime state (current phase, adjusted end time) lives in
  Redis with durable anchors persisted; exact persistence shape is an implementation decision.

## Backend Changes

- `modules/socket/handlers/quiz-events`: start/end orchestration — validates the transition,
  updates session state, broadcasts lifecycle events, triggers Phase 3 forced submission on end.
- Timer authority service (in `modules/quizzes/` services): computes remaining time from
  server-side anchors (start, duration, pause/extension adjustments from F4.2.2) and drives
  periodic `timer-sync` broadcasts. **Needs clarification:** broadcast cadence is unspecified;
  PRD §16 requires "low-latency" without numbers.
- Late-join flow: join handler returns session snapshot (phase + remaining time) on entry.
- Time-expiry watchdog: server ends the session when time runs out even if faculty is absent.

## Frontend Changes

- Student quiz surface (`app/quiz/` shell; components in the owning module per folder-structure
  §3/§4): reacts to `quiz-started`/`quiz-ended`, renders the countdown from `timer-sync` (never a
  purely local clock), and transitions to the submitted state on end.
- Faculty session view: start control, live session phase, and the same synchronized clock.

## Socket Events

- `quiz-started` — broadcast to the quiz room on session start.
- `quiz-ended` — broadcast on session end (any cause; payload carries the reason).
- `timer-sync` — periodic authoritative remaining-time broadcast; also sent on join/reconnect.
- `student-connected` — late joiners announce presence via the existing F4.1.2 flow.
- No proposed additions in this feature.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Start live session (faculty) — transition a published, scheduled quiz to live.
- Fetch current session state — snapshot for page load/refresh before the socket attaches.
  **Assumption:** start is a server action/endpoint that then triggers the broadcast, keeping
  routes thin per folder-structure §3.

## Permissions

- Start/end restricted to the owning faculty (and org admin per RBAC catalog); students receive
  lifecycle events only. All checks via `server/permissions/` capabilities.

## Validation Rules

- Start allowed only for published quizzes within their scheduled window; double-start rejected.
- End is idempotent; time computations never trust client timestamps.
- Late join allowed only while the session is live and the quiz is not locked (F4.2.2).

## Edge Cases & Failure Scenarios

- Faculty disconnects mid-session → session continues; timer watchdog still ends it on expiry.
- All students disconnect → session stays live until end conditions are met.
- Clock skew across server instances → time anchors kept in Redis; single source of truth.
- Student's device sleeps through `quiz-ended` → on wake/reconnect, resync shows the ended state
  and the attempt was already force-submitted server-side.
- Broadcast lost to one client → next `timer-sync`/resync corrects drift; end state enforced
  server-side regardless.

## Acceptance Criteria

- Start and end broadcast to all participants instantly; open attempts are auto-submitted on end.
- All connected participants display remaining time within one sync interval of the server value,
  including after reconnect and late join (PRD §7).
- Session end by time expiry works with no faculty present.
