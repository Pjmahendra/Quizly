# F4.1.1 — Socket Server & Authenticated Connections

**Epic:** E4.1 — Real-Time Infrastructure · **Complexity:** Large

## Description

Bootstrap the Socket.IO server with authenticated, tenant-scoped connections. Bootstrap, adapter,
auth middleware, and room primitives live in `server/socket/` (folder-structure §5). The handshake
validates the caller's existing Phase 1 session (`server/auth/`) — no separate socket credential —
and attaches server-derived identity (user, role, organization) to the connection so handlers never
trust client claims (PRD §16 Security; CLAUDE.md multi-tenancy). A Redis adapter is wired from the
start for horizontal scale (PRD §16 Scalability).

**CRITICAL blocking decision:** PRD §17 names Vercel for deployment, but Socket.IO needs
long-lived stateful connections Vercel serverless does not host. A dedicated realtime host or
managed alternative must be chosen before implementation. This spec keeps `server/socket/`
host-agnostic so the decision changes deployment, not code placement.

## Dependencies

- Phase 1: sessions, RBAC, tenant model (`server/auth/`, `server/permissions/`).
- Redis in local Docker (CLAUDE.md); the hosting decision above (**Needs clarification** — blocking).

## Database Changes (conceptual)

None. Connection state is ephemeral (Redis). **Assumption:** no persistent connection-log table
here; durable connect/disconnect history arrives with F4.3.4's audit trail.

## Backend Changes

- `server/socket/`: bootstrap, Redis adapter, auth middleware resolving session → user + org,
  lifecycle hooks, room join/leave primitives (consumed by F4.1.2).
- `config/socket.ts`: typed, env-derived configuration (path, origins, timeouts).
- Unauthenticated/expired/cross-tenant handshakes refused with typed errors (`lib/errors.ts`),
  logged via `server/logger/`; connection-attempt rate limiting (PRD §16).

## Frontend Changes

- Shared client socket provider/hook (`src/providers/` + `src/hooks/`) opening the authenticated
  connection and exposing status. **Assumption:** shared placement since multiple portals use it.

## Socket Events

- `student-connected` — emitted to the quiz room when a participant's connection is established.
- `student-left` — emitted on disconnect. No proposed additions in this feature.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- None new. The handshake reuses the existing session; realtime-host health checks are a
  deployment concern (out of planning scope).

## Permissions

- Connecting requires an authenticated session; role/org attached server-side. Registered via
  `modules/socket/permissions.ts` into `server/permissions/`: students connect to enrolled
  quizzes; faculty to quizzes they own or run.

## Validation Rules

- Handshake payload Zod-validated; unknown fields rejected; origins restricted to app origins
  (`config/security.ts`); identity fields always server-derived — client values ignored.

## Edge Cases & Failure Scenarios

- Session expires mid-connection → socket terminated; client re-auths without losing attempt data
  (attempt state is server-side, Phase 3).
- Redis adapter down → degrade to single-node with alerts, or fail closed? **Needs clarification.**
- Duplicate connections (two tabs) → reject, replace, or allow? **Needs clarification.**
- Connection storm at quiz start → rate limits must not lock out legitimate joiners.

## Acceptance Criteria

- Authenticated students/faculty establish sockets tied to their sessions; unauthenticated and
  cross-tenant handshakes are rejected before any room access.
- Handlers receive server-derived user, role, and organization on every connection.
- Two app instances sharing Redis both serve the same quiz's connections correctly.
