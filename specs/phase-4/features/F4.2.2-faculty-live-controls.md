# F4.2.2 — Faculty Live Controls

**Epic:** E4.2 — Live Execution & Faculty Control · **Complexity:** Large

## Description

Give faculty the full in-quiz control surface from PRD §4.3 "During Live Quiz": pause, resume,
end, extend time, lock, unlock, and remove student. Every control propagates to affected students
in real time; every action is attributable and audit-logged (PRD §16 Audit Logs). Pause freezes
the authoritative timer and student interaction; resume restores both; extend time shifts the
server-side end anchor (F4.2.1) and re-syncs everyone; lock prevents new joiners while current
participants continue; remove ends one student's participation and blocks rejoin. Control handling
lives in `modules/socket/handlers/faculty-events`, orchestrating `modules/quizzes/` and
`modules/attempts/` services (folder-structure §5). End reuses the F4.2.1 end path.

## Dependencies

- F4.2.1 (session state and timer authority — pause/extend mutate its anchors).
- F4.1.1–F4.1.3; Phase 3 attempt services (remove → attempt closure).
- F4.3.4 audit trail for durable control-action records (can land in parallel; audit hooks here).

## Database Changes (conceptual)

- Control actions need durable, attributable records (who, what, when, prior/new values — e.g.,
  old and new end time for an extension) — conceptually part of the quiz audit log owned by
  `modules/audit/` (PRD §13, §16). A removed student's attempt must record removal as its
  terminal reason, distinct from auto-termination and normal submission.

## Backend Changes

- `modules/socket/handlers/faculty-events`: one handler path per control; each verifies faculty
  authority, applies the state change via services, broadcasts the corresponding event, and writes
  the audit record.
- `modules/quizzes/` session service: pause/resume/extend/lock state transitions with validity
  rules (e.g., resume requires paused); state in Redis for cross-instance consistency.
- `modules/attempts/`: remove-student path closes the attempt with a "removed by faculty" outcome.
  **Needs clarification:** does removal auto-submit answered questions (like PRD §13 termination)
  or void the attempt entirely? PRD §4.3 lists the control without defining the outcome.

## Frontend Changes

- Faculty monitoring surface (module components rendered under `app/faculty/`): control bar with
  confirmation dialogs for destructive actions (end, remove), disabled states reflecting session
  phase, and per-student remove affordance in the roster.
- Student quiz surface: paused overlay (interaction blocked), resume restoration, extended-time
  timer update, lock/unlock feedback for joiners, and a terminal "removed" screen.

## Socket Events

- `quiz-ended` — from the catalog, for the end control (via F4.2.1 path).
- `timer-sync` — re-broadcast after pause/resume/extend so clocks converge.
- **Proposed addition to catalog:** `quiz-paused` — students must freeze instantly (PRD §4.3).
- **Proposed addition to catalog:** `quiz-resumed` — counterpart to pause.
- **Proposed addition to catalog:** `quiz-time-extended` — notify participants of the new end.
- **Proposed addition to catalog:** `quiz-locked` / `quiz-unlocked` — join-gate state change.
- **Proposed addition to catalog:** `student-removed` — targeted to the removed student and the
  faculty dashboard.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Issue a control action (pause/resume/end/extend/lock/unlock/remove) — server action(s) that
  validate, mutate session state, and trigger the broadcast; thin transport per folder-structure.
  **Assumption:** controls are actions/endpoints (not client-emitted socket events) so RBAC and
  audit run through the standard server path; the socket is the propagation channel.

## Permissions

- All controls require quiz-ownership capability (owning faculty; org-admin override per the RBAC
  catalog). Registered in `modules/quizzes/`/`modules/socket/` `permissions.ts`; no permission
  strings at call sites (folder-structure §4).

## Validation Rules

- State-machine validity: pause only when live; resume only when paused; extend only while live or
  paused with a positive duration; lock/unlock idempotent; remove only for an active participant.
- Extension durations bounded (Zod-validated); all IDs tenant-scoped server-side.

## Edge Cases & Failure Scenarios

- Student offline during pause/extend → F4.1.2 resync snapshot must reflect current control state
  on reconnect.
- Concurrent conflicting controls (double-click resume, two faculty) → serialized via session
  state versioning in Redis; stale transitions rejected.
- Remove issued for a student who just submitted → no-op with clear feedback.
- Pause near time expiry → expiry watchdog must respect paused state (no end while paused).

## Acceptance Criteria

- Each of the seven controls takes effect for affected students in real time (PRD §4.3).
- Paused students cannot answer; removed students cannot rejoin; locked quizzes admit no new
  joiners while existing participants continue.
- Every control action is recorded with actor, timestamp, and effect in the audit trail.
