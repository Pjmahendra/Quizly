# F4.2.3 — Faculty Announcements Broadcast

**Epic:** E4.2 — Live Execution & Faculty Control · **Complexity:** Small

## Description

Let faculty broadcast announcements to all participants during a live quiz (PRD §4.3 "Broadcast
announcements"; PRD §7 lists "Faculty announcement" among real-time events). An announcement is a
short, one-way, faculty-to-room text message ("10 minutes remaining", "Question 4 has a typo —
answer B and C both accepted") delivered instantly to every connected student and displayed
non-disruptively — a student's in-progress answer must never be lost or blocked by an incoming
announcement. Handling lives in `modules/socket/handlers/faculty-events` (folder-structure §5).
This is distinct from classroom announcements (PRD §5) and notifications (PRD §14), both Phase 5.

## Dependencies

- F4.1.1–F4.1.3 (connections, rooms, catalog); F4.2.1 (announcements only within a live session).

## Database Changes (conceptual)

- Announcements should be durably recorded with the session (who sent, text, when) so the session
  record is complete and late joiners / reconnecting students can see announcements they missed.
  **Assumption:** persistence for replay is included; the PRD only mandates the live broadcast, so
  the replay depth (all vs. most recent) is an open question.

## Backend Changes

- `modules/socket/handlers/faculty-events`: announcement path — validate, persist, broadcast to
  the quiz room, and write the audit entry.
- Announcement service in `modules/quizzes/` (session-scoped); F4.1.2 resync snapshot extended to
  include missed announcements for reconnecting/late-joining students.
- Rate limiting to prevent announcement flooding (PRD §16 Security/Rate Limiting).

## Frontend Changes

- Faculty monitoring surface: compose-and-send announcement input with delivery feedback.
- Student quiz surface: non-blocking announcement display (toast/banner via shared
  `components/feedback/`, feature wiring in the module) plus a reviewable list of session
  announcements so a missed toast is not lost.

## Socket Events

- **Proposed addition to catalog:** `faculty-announcement` — PRD §7 names "Faculty announcement"
  as a real-time event, but the current catalog has no name for it. Payload: announcement id,
  text, sender display name, sent-at.
- No other events; delivery uses the existing quiz room from F4.1.2.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Send announcement (faculty, per live session) — server action validating and persisting before
  broadcast. **Assumption:** same action-then-broadcast pattern as F4.2.2 controls.
- List session announcements — supports the reviewable list and reconnect replay.

## Permissions

- Sending requires the same quiz-ownership capability as F4.2.2 controls; students are receivers
  only. Registered through the module `permissions.ts` into `server/permissions/`.

## Validation Rules

- Text required, length-bounded, Zod-validated; sanitized against XSS before render (PRD §16).
- Only within a live (or paused) session of a quiz the sender controls; tenant-scoped IDs.

## Edge Cases & Failure Scenarios

- Student disconnected at send time → announcement appears via the resync/replay list on
  reconnect, not silently lost.
- Announcement sent as the session ends → delivery is best-effort; persistence still records it.
- Flooding (many announcements in seconds) → rate limit with faculty-facing feedback.
- Very long text → rejected at validation, not truncated silently.

## Acceptance Criteria

- An announcement reaches every connected participant in real time and is displayed without
  interrupting answer entry (PRD §4.3, §7).
- Reconnecting and late-joining students can see announcements sent while they were absent.
- Announcements are attributable and recorded with the session.

  catalog addition reflected in folder-structure §5 if approved.

## Open Questions

- Replay depth for late joiners; whether announcements can be pinned or expire; any student
