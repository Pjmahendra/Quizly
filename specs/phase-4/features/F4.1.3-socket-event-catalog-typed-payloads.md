# F4.1.3 — Socket Event Catalog & Typed Payloads

**Epic:** E4.1 — Real-Time Infrastructure · **Complexity:** Medium

## Description

Establish `modules/socket/events/` as the single event vocabulary for the entire application
(folder-structure §5, §9): every event name used at any emit or listener site — server or client —
resolves to this catalog; string literals at call sites are forbidden. Each cataloged event gets a
typed payload shape (`modules/socket/types/` + Zod schema in `modules/socket/validators/`) so both
ends of the wire share one contract. The catalog ships with the twelve current names
(`student-connected`, `student-left`, `quiz-started`, `quiz-ended`, `answer-submitted`,
`timer-sync`, `faculty-selected-student`, `window-focus-lost`, `warning-issued`,
`student-auto-submitted`, `resource-added`, `discussion-message`) plus the additions Phase 4
features propose. `resource-added` and `discussion-message` are cataloged with typed payloads but
have no Phase 4 handlers — their behavior is Phase 5 scope.

## Dependencies

- None hard; F4.1.1/F4.1.2 consume the catalog, so this lands first or alongside them.
- Approval of proposed catalog additions (listed below) — each requires updating
  folder-structure §5 (the catalog's governing document).

## Database Changes (conceptual)

None.

## Backend Changes

- `modules/socket/events/`: the catalog — one canonical constant per event, grouped by domain
  (presence, lifecycle, control, monitoring, integrity, leaderboard).
- `modules/socket/types/` + `modules/socket/validators/`: payload type per event and matching Zod
  schema; server-side listeners validate inbound payloads before any handler runs.
- `modules/socket/handlers/` skeleton (`join-room`, `quiz-events`, `faculty-events`,
  `monitoring-events`) wired to catalog names only (folder-structure §5).
- Lint/review guardrail expectation: emits/listens must import from the catalog (enforcement
  mechanism decided at implementation time).

## Frontend Changes

- Client socket layer imports the same catalog and payload types (single source — the module's
  `index.ts` is the only entry point, folder-structure §4), so client and server can never
  disagree on names or shapes.

## Permissions

- The catalog itself carries no permissions; per-event authorization lives in handlers (emitting
  features). The catalog records, per event, which roles may emit and who receives it, as
  documentation for handler authors. **Assumption:** this per-event role annotation is
  documentation, not runtime enforcement — enforcement stays in handlers/permission catalog.

## Validation Rules

- Every inbound event payload is Zod-validated against its cataloged schema; invalid payloads are
  rejected with a typed error and logged, never partially processed.
- Unknown event names are ignored server-side and logged.

## Edge Cases & Failure Scenarios

- Payload schema evolves while old clients are connected → payload shapes must be versioned or
  additive-only during a session. **Needs clarification:** compatibility policy for mid-session
  deploys is undefined.
- An event needed by a future feature is missing → the process is a catalog addition (with
  folder-structure §5 update), never an inline string.

## Acceptance Criteria

- A single module exports every event name and payload type; server and client both import it.
- Grep for event-name string literals outside `modules/socket/events/` returns nothing.
- Inbound payloads failing validation never reach domain handlers.
- Proposed additions are explicitly approved (and folder-structure §5 updated) or replaced by
  in-catalog alternatives before dependent features start.
