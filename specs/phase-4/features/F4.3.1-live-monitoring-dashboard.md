# F4.3.1 — Live Monitoring Dashboard

**Epic:** E4.3 — Live Monitoring & Integrity · **Complexity:** Large

## Description

The faculty live monitoring dashboard: for an active quiz, faculty see the live participant count
and a per-student roster showing connection status, current question, questions answered,
completion percentage, and remaining time — all updating in real time without refresh (PRD §12;
PRD §4.3 "Monitor every student in real time", "View question-wise progress"). The dashboard is a
consumer of the event stream: presence from F4.1.1/F4.1.2, progress from attempt events, timing
from F4.2.1. It also hosts the F4.2.2 control bar and F4.2.3 announcement composer, and surfaces
integrity toasts from F4.3.3. Server-side aggregation lives in `modules/socket/handlers/
monitoring-events` orchestrating `modules/attempts/` reads; the page shell is `app/faculty/`
(routes stay thin — folder-structure §3).

## Dependencies

- F4.1.1–F4.1.3, F4.2.1 (event stream and session state).
- Phase 3 attempts: progress derives from attempt state (answered count / total, position).
- Redis `active-users:{quizId}` for participant counts (folder-structure §9).

## Database Changes (conceptual)

None new. The dashboard reads live state (Redis + socket events) and Phase 3 attempt data; durable
history is F4.3.4's concern.

## Backend Changes

- Faculty joining the monitoring room: join handler admits the owning faculty to the quiz room (or
  a faculty-facing sub-room — **Assumption:** faculty receive a superset of student room traffic;
  exact room topology is an implementation decision inside `modules/socket/`).
- `modules/socket/handlers/monitoring-events`: enriches raw events into roster updates (e.g.,
  `answer-submitted` → recompute that student's completion %).
- Initial-state endpoint/service: full roster snapshot on dashboard load, after which socket
  events keep it current — no polling (PRD §16 "Fast dashboard updates").
- Progress signal for navigation-without-answer ("current question of every student", PRD §4.3):
  requires a client-emitted progress event — see proposed addition below.

## Frontend Changes

- Dashboard components in the owning module (feature UI stays in the module — folder-structure
  §4): roster table with live cells, participant-count header, session timer, connection-status
  badges (connected / reconnecting / left — never conflated with integrity flags).
- Uses shared `components/tables/` and `components/feedback/` primitives; handles hundreds of
  concurrent row updates efficiently (PRD §1 scale; batched rendering expected).

## Socket Events

- `student-connected`, `student-left` — roster presence.
- `answer-submitted` — per-student progress/completion updates.
- `timer-sync` — dashboard clock.
- `quiz-started`, `quiz-ended` — session framing.
- **Proposed addition to catalog:** `student-reconnected` (shared with F4.1.2) — distinct
  reconnect status per PRD §12.
- **Proposed addition to catalog:** `student-progress` — student client reports current-question
  navigation so faculty can "see current question of every student" (PRD §4.3) even before an
  answer is submitted; no existing catalog event carries this.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Fetch monitoring snapshot for a live quiz (roster + counts + session state) — initial load and
  refresh-recovery only; live updates flow over the socket.

## Permissions

- Dashboard restricted to the quiz's owning faculty (org-admin visibility per RBAC catalog —
  **Needs clarification:** PRD §12 speaks only of faculty). Students never receive roster or
  other students' progress data.

## Validation Rules

- Quiz ID tenant-scoped; snapshot only for quizzes the caller may monitor.
- Inbound `student-progress` payloads Zod-validated; question index must belong to the quiz.

## Edge Cases & Failure Scenarios

- Faculty dashboard reconnects → full snapshot re-fetch, then resume event stream (no stale rows).
- Event bursts at quiz start (everyone joins at once) → UI batching; server aggregation must not
  drop events.
- Student with no events for a long period → shown via last-activity timestamp (drill-in
  inactivity logic is F4.3.2).
- Two faculty windows open → both receive identical live state.

## Acceptance Criteria

- Faculty see joins, disconnects, reconnects, current question, answered count, completion %, and
  remaining time per student, live (PRD §12, §4.3).
- Dashboard state after a refresh matches live reality (snapshot + stream converge).
- No student-facing surface can access monitoring data.
