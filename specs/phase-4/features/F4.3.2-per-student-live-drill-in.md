# F4.3.2 — Per-Student Live Drill-In

**Epic:** E4.3 — Live Monitoring & Integrity · **Complexity:** Medium

## Description

From the monitoring dashboard, faculty select any student to open a focused live view (PRD §4.3
Student Monitoring): current progress, which questions are answered, completion percentage,
remaining time, inactivity detection, and reconnect-event tracking. Selecting a student emits
`faculty-selected-student` (catalog); the focused view then renders that student's live state and
recent activity. Inactivity is derived server-side from the student's last activity timestamp
(answers, progress, presence events) crossing a threshold — it is an attentiveness signal for
faculty, explicitly NOT an integrity violation (PRD §13 governs focus loss only). Handler logic
lives in `modules/socket/handlers/monitoring-events`; the view is a module component rendered
within the F4.3.1 dashboard.

## Dependencies

- F4.3.1 (dashboard host, roster, snapshot service).
- F4.1.2 (reconnect events to track); Phase 3 attempts (per-question answered state).

## Database Changes (conceptual)

None new. The drill-in reads live state plus the same activity records F4.3.4 persists; reconnect
history within a session comes from those records.

## Backend Changes

- `modules/socket/handlers/monitoring-events`: handle `faculty-selected-student` — validate the
  selection, return the focused snapshot (per-question answered map, completion %, remaining time,
  last-activity timestamp, reconnect count/history for this session).
- Inactivity derivation: last-activity tracking per attempt (Redis) with a threshold marking the
  student inactive. **Needs clarification:** the inactivity threshold is unspecified — fixed,
  configurable per quiz, or per organization?
- **Assumption:** `faculty-selected-student` is a server-consumed event used to authorize and
  scope the focused stream/snapshot; it is not broadcast to students (the PRD gives no indication
  students should know they are being watched).

## Frontend Changes

- Drill-in panel (module component): per-question answered/unanswered map, completion and time
  readouts, inactivity indicator with duration, and a reconnect-events strip for the session.
- Selection interaction from the roster; deselect/switch-student without losing dashboard state.

## Socket Events

- `faculty-selected-student` — emitted by the faculty client on selection (from the catalog).
- `answer-submitted` — drives per-question map updates for the selected student.
- **Proposed addition to catalog:** `student-progress` (shared with F4.3.1) — current-question
  changes for the focused view.
- **Proposed addition to catalog:** `student-reconnected` (shared with F4.1.2) — reconnect-event
  tracking (PRD §4.3 "Track reconnect events").

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Fetch focused-student snapshot (per-question state + activity summary) — initial drill-in load;
  live updates then arrive over the socket.

## Permissions

- Same monitoring capability as F4.3.1: only the quiz's owning faculty may select and view a
  student; selection of a student outside the quiz/tenant is rejected server-side.

## Validation Rules

- Selected student must hold an attempt on the monitored quiz within the faculty's organization.
- Answered-question detail exposes progress state only — not the student's answer content.
  **Assumption:** live drill-in shows which questions are answered, not answer text; PRD §4.3
  says "See answered questions" without specifying content visibility. **Needs clarification.**

## Edge Cases & Failure Scenarios

- Selected student disconnects → view shows reconnecting/left status live rather than freezing.
- Selected student is removed (F4.2.2) or auto-terminated (F4.3.3) → view shows the terminal
  state immediately.
- Rapid switching between students → prior focused subscriptions cleaned up; no leaked streams.
- Selection during an event burst → snapshot + stream converge, same pattern as F4.3.1.

## Acceptance Criteria

- Faculty can select any participant and see progress, answered map, completion %, remaining time,
  inactivity state, and reconnect history, live (PRD §4.3).
- Inactivity indication appears after the threshold and clears on renewed activity.
- Reconnect events for the session are visible and never presented as integrity violations.
