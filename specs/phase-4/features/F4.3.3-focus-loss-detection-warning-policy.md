# F4.3.3 — Focus-Loss Detection & Warning Policy Enforcement

**Epic:** E4.3 — Live Monitoring & Integrity · **Complexity:** Large

## Description

Implement PRD §13 end to end: the student client monitors browser visibility during an active quiz
(tab switch, minimize, app switch, focus loss) and immediately reports each occurrence over the
socket (`window-focus-lost`). The server counts warnings per attempt against the quiz's configured
policy (from F2.3.4: enabled/disabled + threshold, default 3), issues escalating warnings
(`warning-issued` → faculty toast + student notice), and at the threshold auto-terminates the
attempt: remaining questions submitted unanswered via the Phase 3 auto-submit path, submission
marked **"Auto Terminated due to Quiz Integrity Policy"**, faculty notified immediately, activity
log preserved (PRD §13 Warning Policy). Scope per the PRD's scope statement: browser activity
monitoring only — **no webcam monitoring, screen recording, or eye tracking**. Disconnects are
never focus losses (PRD §7 recovery vs. §13 integrity); the two pipelines share nothing but the
socket. Enforcement is server-side in `modules/socket/handlers/monitoring-events` orchestrating
`modules/attempts/` and `modules/audit/`; warning counts live in Redis (`warnings:{attemptId}`).

## Dependencies

- F4.1.1–F4.1.3 (trusted connection + catalog); F4.2.1 (active-session window).
- Phase 2 F2.3.4: per-quiz integrity policy configuration (PRD §13 "configurable proctoring
  policy").
- Phase 3: auto-submit path and a terminal attempt status for integrity termination.
- F4.3.4 persists the events this feature emits (can land in parallel; hooks defined here).

## Database Changes (conceptual)

- The attempt needs a terminal outcome distinguishing integrity termination from normal
  submission and faculty removal, carrying the PRD's exact marking. Focus-loss and warning events
  need durable records (F4.3.4 owns the timeline/audit shape). Live counters stay in Redis;
  durable truth in PostgreSQL.

## Backend Changes

- Server-side policy engine: on `window-focus-lost`, atomically increment the attempt's warning
  count (Redis, race-safe for horizontal scale), evaluate against the quiz's threshold, and either
  emit `warning-issued` (with count/threshold, e.g., "Warning 2 of 3" — PRD §13) or trigger
  termination emitting `student-auto-submitted`.
- Termination flow: force-submit via `modules/attempts/`, set the integrity marking, flag the
  submission for review, write audit entries, notify faculty in real time.
- Policy disabled for the quiz → events are still recorded for the timeline but produce no
  warnings. **Assumption:** recording continues when enforcement is off; PRD says the policy is
  configurable but not whether logging stops. **Needs clarification.**

## Frontend Changes

- Student quiz surface: visibility/blur detection during the active session only; immediate
  report over the socket; prominent warning display on `warning-issued`; terminal integrity
  screen on termination.
- Faculty dashboard (F4.3.1): real-time toast per focus-loss/warning (PRD §13 Faculty Alerts) via
  shared `components/feedback/`, and flagged status in the roster.

## Socket Events

- `window-focus-lost` — student client → server report (from the catalog).
- `warning-issued` — server → student (warning notice) and faculty room (alert).
- `student-auto-submitted` — server → room on threshold termination.
- No proposed additions in this feature.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- None new for enforcement (socket-driven). Policy configuration endpoints belong to F2.3.4;
  post-quiz review surfaces belong to F4.3.4.

## Permissions

- Only the attempt's owner (from server-side socket context) can produce focus events for it;
  warning/termination decisions are entirely server-side. Faculty alerts scoped to quiz owners.

## Validation Rules

- Focus events accepted only for the caller's own in-progress attempt during a live session;
  Zod-validated payloads; counts never client-supplied.
- Events during pause: **Needs clarification** — PRD does not say whether focus loss while paused
  counts. Planned default: not counted while paused (student interaction is already blocked).
- Client-side detection is evadable (dev tools, event suppression) — accepted MVP limitation
  consistent with the PRD scope statement; note in documentation.

## Edge Cases & Failure Scenarios

- Disconnect vs. focus loss: a dropped socket must never increment warnings; a focus loss followed
  by disconnect counts once (the report may arrive on reconnect — **Needs clarification:** are
  queued offline reports accepted, given they cannot be server-verified?).
- Rapid blur/focus flapping → debounce so one real event does not count multiple times.
- Threshold reached simultaneously on two instances → Redis atomic increment guarantees exactly
  one termination.
- Threshold changed mid-session → **Needs clarification:** apply immediately or at next session?

## Acceptance Criteria

- Focus loss produces an immediate faculty toast, timeline entry, audit record, and review flag
  (PRD §13 Faculty Alerts).
- Warnings escalate exactly per the configured policy; default matches PRD §13 (2 warnings, then
  termination on the third).
- Terminated attempts are marked "Auto Terminated due to Quiz Integrity Policy" with remaining
  questions unanswered, faculty notified, and the log preserved.
- Disconnect/reconnect cycles never produce warnings.
