# F4.3.4 — Activity Timeline & Integrity Audit

**Epic:** E4.3 — Live Monitoring & Integrity · **Complexity:** Medium

## Description

Make the live session's integrity story durable and reviewable: a per-student activity timeline
(joins, disconnects, reconnects, focus losses, warnings, submissions, faculty actions affecting
the student), flagged-for-review submissions, and quiz audit log entries (PRD §13 Faculty Alerts:
"the affected student's activity timeline is updated… the event is recorded in the quiz audit
log… the submission is flagged for review"; "The activity log is preserved for later review").
This feature is the persistence-and-review layer over events produced by F4.1.2, F4.2.2, and
F4.3.3: it subscribes to the same domain happenings, writes durable records through
`modules/audit/` (folder-structure §4), and surfaces them in a timeline view (live during the
session and read-only afterwards) plus a flagged-submissions review list for faculty.

## Dependencies

- F4.3.3 (integrity events to persist), F4.1.2 (presence events), F4.2.2 (control actions).
- `modules/audit/` as the audit domain home; Phase 3 submissions (the flag attaches to them).

## Database Changes (conceptual)

- Durable per-attempt activity records: typed event, timestamp, actor, session context — the
  substrate for both the timeline and the audit log. A review flag on submissions with the reason
  (integrity event, auto-termination). Records are append-only and retained after the quiz ends
  (PRD §13, §16 Audit Logs). **Needs clarification:** retention period is unspecified.

## Backend Changes

- `modules/audit/`: audit writer consumed by socket handlers and control actions — one write path
  so no live feature persists audit records ad hoc; repository per folder-structure §8.
- Timeline read service: per-student chronological activity for a session; flagged-submissions
  query per quiz.
- Flagging: focus-loss/warning/termination events mark the related submission for review
  (PRD §13); flag visible in Phase 3's submission/review surfaces.
- **Assumption:** audit writes are asynchronous-safe (never block the live event path) but must
  not be lossy for integrity events; ordering by server timestamp.

## Frontend Changes

- Timeline panel inside the F4.3.2 drill-in (live) and on the post-quiz submission view
  (read-only): chronological entries with clear visual distinction between connectivity events,
  integrity events, and faculty actions.
- Flagged-for-review list on the faculty quiz results surface, linking each flag to its timeline.

## Socket Events

- Consumes (persists) events already defined elsewhere: `student-connected`, `student-left`,
  `window-focus-lost`, `warning-issued`, `student-auto-submitted`, plus the proposed
  `student-reconnected` and F4.2.2 control events once approved.
- Emits none of its own. No proposed additions in this feature.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Fetch a student's activity timeline for a quiz session (faculty, live and post-quiz).
- List flagged-for-review submissions for a quiz (faculty).

## Permissions

- Timeline and flag data restricted to the quiz's owning faculty and org-level roles per the RBAC
  catalog; students do not see their own integrity timeline. **Needs clarification:** the PRD is
  silent on student visibility into their own warnings history post-quiz (they do see warnings
  live via F4.3.3).

## Validation Rules

- All reads tenant-scoped and quiz-scoped; timeline queries paginated (guidelines pagination
  rule).
- Audit records immutable once written — corrections are new entries, never edits.

## Edge Cases & Failure Scenarios

- Audit store briefly unavailable during a live session → integrity events buffered and flushed,
  never silently dropped; monitoring alert on failure (PRD §16 Reliability).
- Event bursts (mass join) → batched writes must preserve per-student ordering.
- Timeline requested mid-session → combines persisted records with in-flight state consistently.
- Duplicate delivery of the same event → idempotent persistence (dedupe key).

## Acceptance Criteria

- Every focus-loss event results in a timeline entry, an audit log entry, and a review flag on
  the submission (PRD §13).
- Faculty can review a complete, ordered per-student timeline during and after the quiz,
  including for auto-terminated attempts.
- Flagged submissions are listable per quiz and traceable to their triggering events.
- Connectivity events and integrity events are visually and semantically distinct.
