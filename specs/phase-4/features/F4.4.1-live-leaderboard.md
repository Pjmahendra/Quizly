# F4.4.1 — Live Leaderboard

**Epic:** E4.4 — Leaderboards · **Complexity:** Medium

## Description

An optional, per-quiz live leaderboard that refreshes in real time during the session (PRD §4.4
"Real-time Leaderboard (optional)"; PRD §7 "Leaderboards refresh live"; PRD §12 "Live leaderboard
updates (optional)"). When enabled on a quiz, each auto-graded answer (objective types only —
MCQ, multiple select, true/false, numerical, PRD §10) updates the student's live score in a
Redis-backed ranking structure (`leaderboard:{quizId}`, folder-structure §9), and updated
standings are broadcast to the quiz room for students and the faculty dashboard. When disabled,
nothing is computed or shown. Ranking logic lives in the quizzes domain services with the socket
handler in `modules/socket/handlers/quiz-events`. **Assumption:** the leaderboard service is owned
by `modules/quizzes/` (it is quiz-session state, not post-hoc analytics); confirm against
`modules/analytics/` ownership before implementation.

## Dependencies

- F4.1.1–F4.1.3 (room + catalog), F4.2.1 (session window).
- Phase 3 objective auto-grading — scores must be computable at answer-submission time.
- Phase 2 quiz settings: per-quiz leaderboard enable/disable toggle (**Assumption:** exists as a
  quiz-configuration field; PRD marks the feature optional without naming its configuration home).
- Redis (`server/cache/keys.ts` cataloged key).

## Database Changes (conceptual)

- The enable/disable choice persists with the quiz configuration. Live standings are ephemeral in
  Redis; final standings derive from persisted Phase 3 results, so no separate durable leaderboard
  record is required. **Assumption:** post-quiz result views recompute from attempts rather than
  freezing the Redis snapshot.

## Backend Changes

- Leaderboard service: on auto-graded answer, update the student's score in the Redis sorted
  ranking; read top-N plus the requesting student's own rank.
- `modules/socket/handlers/quiz-events`: broadcast standings updates to the room, throttled so
  answer bursts do not flood clients (batch/interval broadcasting).
- Lifecycle hooks: initialize on `quiz-started` (when enabled), clean up after `quiz-ended`.
- Ranking rules implemented behind one function so the pending scoring decision (below) changes
  one place.

## Frontend Changes

- Student quiz surface: optional leaderboard panel (top-N + own rank), updating live, dismissible
  so it never obstructs answering; hidden entirely when disabled.
- Faculty dashboard (F4.3.1): leaderboard panel when enabled.
- **Needs clarification:** anonymized display names vs. real names for students — PRD §4.4 peer
  visibility is "subject to organization privacy policies".

## Socket Events

- `answer-submitted` — the input trigger (from the catalog) for score updates.
- `quiz-started` / `quiz-ended` — lifecycle initialize/teardown.
- **Proposed addition to catalog:** `leaderboard-updated` — server → room broadcast carrying
  updated standings; PRD §7 requires live leaderboard refresh but the catalog has no event for it.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Fetch current standings for a live quiz — initial render and refresh recovery; live updates via
  the socket.

## Permissions

- Standings visible to the quiz's participants and owning faculty only, tenant-scoped; the
  enable/disable setting is editable only by whoever may edit the quiz (Phase 2 permissions).

## Validation Rules

- Score updates computed server-side from Phase 3 grading only — never from client-reported
  scores.
- Updates accepted only while the session is live and the feature is enabled for that quiz.
- Manually graded (descriptive) answers contribute nothing live; their exclusion must be explicit
  in faculty-facing copy.

## Edge Cases & Failure Scenarios

- Redis loss mid-session → standings rebuildable from persisted attempt answers (recovery path
  required, PRD §16 Reliability).
- Ties → display rule needed (shared rank vs. tiebreak) — part of the scoring decision below.
- Auto-terminated or removed students → **Needs clarification:** dropped from the board or frozen
  at current score?
- Answer changed before submit (if Phase 3 allows revisions) → score update must reflect the
  latest graded state, not double-count.

## Acceptance Criteria

- With the toggle enabled, standings update live for students and faculty as objective answers are
  graded (PRD §7); with it disabled, no leaderboard is computed or visible (PRD §4.4).
- Standings survive participant reconnects and are consistent across server instances.
- A student always sees their own rank even when outside the top-N.
