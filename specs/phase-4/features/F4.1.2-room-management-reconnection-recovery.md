# F4.1.2 — Room Management & Reconnection Recovery

**Epic:** E4.1 — Real-Time Infrastructure · **Complexity:** Large

## Description

Give every live quiz its own room and make dropped connections recoverable without losing attempt
progress (PRD §7 "Connection recovery is supported"; PRD §16 "Automatic reconnect"). Room
primitives live in `server/socket/`; room naming and the `join-room` handler live in
`modules/socket/handlers/` (folder-structure §5). Membership is tracked in Redis
(`socket-room:{quizId}`, `active-users:{quizId}` — folder-structure §9) for cross-instance
consistency. On reconnect, the server restores the participant to their room and resyncs state —
current question, saved answers, remaining time — from the Phase 3 attempt record. A disconnect is
a connectivity fact, never an integrity event (PRD §13): it must not feed the warning pipeline.

## Dependencies

- F4.1.1 (authenticated connections, room primitives); Phase 3 attempts (resync source of truth);
  Redis key catalog (`server/cache/keys.ts`).

## Database Changes (conceptual)

None. Membership is ephemeral in Redis; attempt state already persists (Phase 3). Reconnect events
become durable history via F4.3.4, not here.

## Backend Changes

- `server/socket/`: room join/leave primitives with tenant-checked room names.
- `modules/socket/handlers/join-room`: verifies an active attempt (student) or session ownership
  (faculty) before joining; emits presence events. Room-naming utility is the only place quiz-room
  names are constructed.
- Recovery: on reconnect within the session window, rejoin room and return a state-resync snapshot
  (attempt progress + remaining time) as the join acknowledgment.
- Grace period distinguishing transient drops from truly leaving. **Needs clarification:**
  duration unspecified in the PRD.

## Frontend Changes

- Client socket layer: automatic reconnect with backoff; apply the resync snapshot (position,
  saved answers, timer) before re-enabling interaction.
- "Reconnecting…" status indicator in the student quiz surface (`app/quiz/` shell; component owned
  by its module per folder-structure §4).

## Socket Events

- `student-connected` — first join to the quiz room.
- `student-left` — student leaves or grace period expires.
- `timer-sync` — sent to the reconnecting client during resync (F4.2.1 owns cadence).
- **Proposed addition to catalog:** `student-reconnected` — PRD §7/§12 name "Student reconnected"
  as a distinct event and F4.3.2 tracks reconnects; the catalog has no name for it.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- None new. Resync rides the socket acknowledgment; attempt endpoints exist from Phase 3.
  **Assumption:** no REST fallback for resync this phase.

## Permissions

- Room join requires an in-progress attempt (student) or quiz ownership (faculty), enforced in the
  join handler via the permission catalog; room names never accepted raw from the client.

## Validation Rules

- Join payload (quizId, attemptId) Zod-validated; both tenant-scoped to the caller's organization.
- Students join only quizzes where they hold an in-progress attempt; resync snapshots are built
  entirely from server-side state.

## Edge Cases & Failure Scenarios

- Reconnect after quiz end → no room join; client directed to post-quiz state.
- Reconnect after auto-termination (F4.3.3) or faculty removal (F4.2.2) → join refused; terminal
  state shown.
- Disconnect/reconnect flapping → presence events debounced for the dashboard; every transition
  still reaches the audit trail (F4.3.4).
- Server instance dies → clients reconnect elsewhere; Redis-backed membership recovers the room.

## Acceptance Criteria

- Participants of one quiz never receive another quiz's room traffic.
- A student who drops mid-quiz reconnects to the same question with saved answers and correct
  remaining time (PRD §7); faculty see the disconnect and reconnect live (PRD §12).
- No focus-loss warning is ever produced by a disconnect/reconnect cycle (PRD §13).
