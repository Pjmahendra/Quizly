# F1.4.5 — Profile Self-Management

**Epic:** E1.4 — User Management & Audit · **Complexity:** Small

## Description

Every user manages their own profile (PRD §4.4 "Profile Management", applicable to all roles):
edit display name, change password (current password required), and set personal preferences.
**Profile images are explicitly deferred:** they require the file-storage platform
(Azurite/Azure Blob, PRD §9), which arrives with the Phase 5 storage work — this feature ships
image-less and notes the seam where an avatar attaches later. Password change is distinct from
reset (F1.1.3): it is an authenticated action requiring the current password, and it revokes
other sessions as a safety measure (**Assumption** — mirrors F1.1.3 behavior; confirm).

## Dependencies

- E1.1 (authenticated account, credential rules, session revocation).
- F1.4.4 (audit for password change).

## Database Changes (conceptual)

- Extends the user concept in prose with self-editable attributes: display name, a small
  preferences bag (e.g. notification/email preferences as they become meaningful — nothing
  invented beyond what Phase 1 surfaces need). No image fields this phase.

## Backend Changes

- `modules/users/services/` self-profile read/update and password-change orchestration (password
  change delegating to `modules/auth` credential rules via its `index.ts`).
- Self-scope only: the acting user is always the target — taken from server context, never a
  client-supplied user ID (guidelines §1.1/§8.2).
- Password change audited (event, never values) and rate-limited (guidelines §8.8).

## Frontend Changes

- `app/settings/` profile surface (thin shell) composing `modules/users/components/`: name and
  preferences form with inline field validation; password-change form (current + new) with
  server-driven errors; sits alongside F1.1.4's sessions surface for one coherent settings area
  (DoD §14 Product Consistency).

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Read own profile; update own name/preferences.
- Change own password (current password required).

## Permissions

- Authenticated self-service; no administrative capability required. Editing anyone else's
  profile is F1.4.3's capability-gated territory — the two must not share endpoints.

## Validation Rules

- Name required and bounded; preferences validated against a typed allow-list of keys/values
  (guidelines §8.4); new password meets the same server-side policy as registration; current
  password verified before any change.

## Edge Cases & Failure Scenarios

- Wrong current password on change: inline error, rate-limited, no session disturbance.
- Password change with other devices signed in: other sessions revoked per the assumption above;
  current session continues with feedback.
- Email change is out of scope this phase (identity-bearing; would need re-verification) —
  listed under open questions.
- Concurrent edits from two of the user's own sessions: last write wins with fresh read-back.

## Acceptance Criteria

- A user can update name and preferences and see them reflected across the app immediately.
- A user can change their password; the old one stops working; the change is audited.
- No endpoint in this feature can affect any other user (self-scope proof).
- The settings surface states that profile images arrive later (no dead upload affordance —
  DoD §2).

## Definition of Done

- `rules/definition-of-done.md` gates (esp. §9 Forms, §8 Feedback) plus guidelines §8.3/§8.4.
  Specific: confirm no preference key renders as a raw internal code (guidelines §13).
