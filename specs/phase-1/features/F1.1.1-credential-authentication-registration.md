# F1.1.1 — Credential Authentication & Registration

**Epic:** E1.1 — Authentication & Sessions · **Complexity:** Medium

## Description

Sign-up and sign-in with email + password (PRD §4, §16). Credentials are hashed with a
memory-hard algorithm of the Argon2id class — never stored reversibly or recoverable
(guidelines §8.3). Both flows are rate-limited (PRD §16). Registration creates an unverified
account pending F1.1.2; what a self-registered account may do before org association follows
F1.4.2's pending decision. The model is MFA-ready by design, but no MFA ships this phase
(PRD §16; guidelines §8.3). **Blocking dependency:** the auth provider — Better Auth vs. Clerk —
is TBD (PRD §17); implementation cannot start until it is decided. This spec is
provider-agnostic; the provider lives behind the `server/auth/` adapter (folder-structure §5).

## Dependencies

- Phase 0 scaffold; **auth provider decision (blocking)**.
- F1.1.2 (verification) and F1.1.4 (sessions) complete the lifecycle this feature starts.

## Database Changes (conceptual)

- A user/account entity in prose: identity (email, unique per platform), display name, credential
  hash, verification status, account status, MFA-ready attachment point for future factors,
  timestamps. Exact shape follows the chosen provider's account model — settled at expansion.

## Backend Changes

- `modules/auth/` per the module anatomy: `actions/` (transport only), `services/`,
  `repositories/` (only Prisma call sites), `validators/` (Zod), `dto/`, and `permissions.ts`
  registering auth-related catalog entries.
- `server/auth/` adapter wrapping the chosen provider's credential and session plumbing.
- Rate limiting on both flows (guidelines §8.8); failure messaging never distinguishes "wrong
  password" from "no such account".

## Frontend Changes

- `app/(auth)/` sign-up and sign-in routes as thin shells (folder-structure §3) composing
  `modules/auth/components/` forms; inline field validation, not toasts (guidelines §13).
- Shared form primitives from `src/components/forms` and `ui/`.

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Register a new account with credentials.
- Sign in with credentials, establishing a session (see F1.1.4).
- Sign out the current session.

## Permissions

- Registration and sign-in are unauthenticated by nature; everything after them is default-deny
  (guidelines §8.2). No catalog entries beyond `modules/auth/permissions.ts` self-service needs.

## Validation Rules

- Email format-validated and normalized; password policy enforced server-side — client validation
  is UX only (guidelines §8.4). All inputs bounded and Zod-validated in `modules/auth/validators/`.

## Edge Cases & Failure Scenarios

- Duplicate registration with an existing email: respond without leaking account existence.
- Sign-in against unverified or disabled accounts: fails with non-leaking copy.
- Rate-limit exhaustion: calm, role-framed error with retry guidance (guidelines §13).
- Provider/adapter outage: typed error surfaced as a load failure, not a permission denial.

## Acceptance Criteria

- A new user can register, and after verifying (F1.1.2) can sign in and reach their portal.
- Credential hashes are Argon2id-class; no code path can return a stored password.
- Both endpoints are rate-limited; failure responses never confirm account existence.
- No Prisma call exists outside `modules/auth/repositories/` (folder-structure §8).
