# F1.2.3 — Organization Admin Assignment & Onboarding

**Epic:** E1.2 — Organizations & Multi-Tenancy Core · **Complexity:** Medium

## Description

Super Admin assigns administrators to organizations (PRD §4.1 "Manage Organization Admins";
PRD §4.2 "Each college/organization has its own administrator"), and the assigned admin completes
initial organization setup on first sign-in. Assignment covers both existing users and
not-yet-registered admins (delivered as an invitation via F1.4.1 machinery bound to the Org Admin
role). Onboarding is a light first-run experience — confirm org details, then be pointed at
inviting users (F1.4.1) — not a wizard inventing requirements beyond the PRD.

## Dependencies

- F1.2.1 (organization + membership), F1.2.2 (organizations exist to assign into).
- F1.3.1/F1.3.2 (Org Admin default role template supplies the granted capabilities).
- F1.4.1 invitation machinery for not-yet-registered admins; Resend for the email.

## Database Changes (conceptual)

- No new entity expected: assignment is a membership (F1.2.1) carrying the Org Admin role
  (F1.3.2). In prose: who assigned whom, when, and the membership's status; onboarding completion
  is a lightweight state on the membership or organization.

## Backend Changes

- `modules/organizations/services/` assignment/removal orchestration (validating target org and
  user, composing with `modules/users` and `modules/auth` via their `index.ts` only —
  folder-structure §4).
- Guards: assignment is Super Admin capability; onboarding steps are Org Admin capability scoped
  to their own org via the F1.2.1 context.
- Audit events for assignment and removal (guidelines §8.8).

## Frontend Changes

- `app/admin/`: assign/remove admin actions on the organization detail surface (side-sheet form,
  confirm dialog for removal — guidelines §13).
- `app/organization/`: first-run onboarding state for a newly assigned admin — helpful empty
  states pointing to next actions (DoD §2, §10), replacing dashboards that would otherwise be
  blank.

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Assign an organization admin (existing user or invitation to a new one) — Super Admin.
- Remove/replace an organization admin — Super Admin.
- Read/complete the organization's onboarding state — Org Admin, own org only.

## Permissions

- Admin-assignment actions on the organization resource, registered via
  `modules/organizations/permissions.ts`, granted to Super Admin; onboarding actions granted to
  Org Admin. Capability grants, never role-name checks (guidelines §9).

## Validation Rules

- Target organization must exist and be active; assignee email/user validated; an invitation-based
  assignment follows F1.4.1's validation. Removal of the last admin is guarded (see edge cases).
  All server-side, Zod-validated.

## Edge Cases & Failure Scenarios

- Removing an organization's only admin: blocked or explicitly confirmed with consequences —
  **Needs clarification** which; an org with zero admins is otherwise unmanageable.
- Assigning a user who already belongs to another organization: depends on the multi-org decision
  in F1.2.1 — fail closed until decided.
- Invitation to a mistyped email: revocation path via F1.4.1.
- Assigned admin never completes onboarding: org remains functional; onboarding is guidance, not
  a gate (**Assumption**).

## Acceptance Criteria

- Super Admin can assign an org admin to any organization, by existing account or invitation, and
  remove one — each action audited and visible on the org's trail.
- A newly assigned admin signs in and lands in their organization portal with correct
  capabilities and a helpful first-run state.
- An org admin can never act on any organization other than their own (F1.2.1 scope proof).

## Definition of Done

- `rules/definition-of-done.md` gates (esp. §1 Feature Completeness, §4 Workflow Completion, §10
  Discoverability) plus guidelines §8.2/§8.8/§9. Specific: onboarding copy is role-framed and
  calm (guidelines §13).

## Open Questions

- **Needs clarification:** may an organization have multiple admins? PRD §4.2 says "its own
  administrator" (singular) but doesn't forbid more. This spec assumes multiple are allowed.
- **Needs clarification:** policy for removing the last admin (block vs. confirm).
