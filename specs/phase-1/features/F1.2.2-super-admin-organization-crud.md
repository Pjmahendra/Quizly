# F1.2.2 — Super Admin: Organization CRUD

**Epic:** E1.2 — Organizations & Multi-Tenancy Core · **Complexity:** Medium

## Description

The Super Admin's organization lifecycle management: create, edit, and delete organizations
(PRD §4.1 "Create/Edit/Delete Organizations", "Manage Colleges/Institutions"). This is the
commercial front door — every institution on the platform is born here. All actions are
capability-gated (Super Admin only), audited (F1.4.4), and operate on the entity defined in
F1.2.1. Deletion is a deliberate, guarded act with explicit confirmation given its blast radius.

## Dependencies

- F1.2.1 (organization entity and scoping context).
- F1.3.1/F1.3.3 (catalog entries and guards); F1.4.4 (audit trail for lifecycle mutations).

## Database Changes (conceptual)

- No new entities beyond F1.2.1's organization; adds lifecycle semantics in prose — status
  transitions (active/suspended), deletion semantics (**Assumption:** soft delete with retention,
  given tenant blast radius — confirm), and audit linkage.

## Backend Changes

- `modules/organizations/services/` lifecycle operations with authorization checks;
  `validators/` for create/edit payloads; `repositories/` for persistence.
- Guards from `server/permissions/` on every mutation (default deny, guidelines §8.2).
- Audit events for create/edit/delete/suspend via the audit platform (guidelines §8.8).

## Frontend Changes

- `app/admin/` organizations area (thin shells) composing `modules/organizations/components/`:
  paginated organization list via the shared table + pagination + search components
  (guidelines §13; DoD §7), create/edit forms in side sheets, delete behind an explicit confirm
  dialog naming consequences (guidelines §13).
- Audit trail visible on the organization detail surface (guidelines §8.8).

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List organizations (paginated, searchable) — Super Admin.
- Create, update (details/status), and delete (or suspend) an organization.
- Read a single organization with its audit trail.

## Permissions

- Organization resource actions (list/read/create/update/delete, plus suspend if modeled as a
  capability) registered via `modules/organizations/permissions.ts`; granted only to the Super
  Admin default role template (F1.3.2). No permission strings at call sites (guidelines §9).

## Validation Rules

- Organization name required, bounded; uniqueness policy decided at expansion (**Assumption:**
  unique platform-wide for admin clarity). Payloads Zod-validated server-side; list reads
  paginated and bounded (guidelines §1.2, §8.4).

## Edge Cases & Failure Scenarios

- Deleting an organization with active members: consequences stated in the confirm dialog;
  members fail closed afterwards (F1.2.1 behavior).
- Concurrent edits: last-write semantics decided at expansion; no silent loss without feedback.
- Non-Super-Admin caller on any endpoint: permission denial, distinct copy (guidelines §9, §13).
- Empty state on first run ("no organizations yet") is helpful, with the create action.

## Acceptance Criteria

- A Super Admin can create, edit, suspend/delete, and browse organizations end-to-end.
- Every mutation appears in the organization's visible audit trail with who/what/when.
- All list surfaces paginate through the shared components; search keeps focus (guidelines §13).
- Every endpoint rejects non-Super-Admin callers at the guard layer.
