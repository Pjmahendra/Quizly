# F1.4.4 — Audit Log Platform

**Epic:** E1.4 — User Management & Audit · **Complexity:** Large

## Description

An append-only audit trail for sensitive mutations, built as a platform (PRD §4.1 "Audit Logs";
PRD §16; guidelines §8.8): who acted, what entity/field, when — never the sensitive value itself.
Phase 1 producers: auth lifecycle events (verification, resets, session revocations, reuse
detections), organization lifecycle (F1.2.2), admin assignment (F1.2.3), role and grant changes
(F1.3.2), invitations and approvals (F1.4.1–2), and user administration (F1.4.3). The trail is
surfaced where the affected entity is viewed (org detail, user detail), plus an org-scoped audit
view for admins and a platform-wide view for Super Admin. Later phases (quiz integrity events,
PRD §13) write to this same platform.

## Dependencies

- F1.2.1 (org scoping of entries and viewers), F1.3.1/F1.3.3 (viewing capabilities and guards).
- Producers listed above integrate as they land; the platform itself depends only on E1.1/F1.2.1.

## Database Changes (conceptual)

- An audit-entry concept in prose: actor, organization scope (or platform scope), affected entity
  reference (type + id), action, summary of what changed (field names, never sensitive values —
  guidelines §8.7/§8.8), timestamp. Append-only: no update or delete pathway exists in the
  application. Indexed for by-entity and by-org reads (bounded queries, guidelines §1.2).

## Backend Changes

- `modules/audit/` per folder-structure §4: a recording service with a typed, data-driven event
  vocabulary (no free-string event names at call sites — guidelines §1.3, mirroring the catalog
  discipline of folder-structure §9), repositories as the only write path, query services for
  entity/org/platform reads.
- Producers call `modules/audit` via its `index.ts`; recording failures must not silently drop
  entries for sensitive mutations (policy: fail the mutation or queue-retry — decided in design,
  guidelines §3).

## Frontend Changes

- A reusable audit-trail component (module-owned, composed into entity detail surfaces): entries
  human-readable ("A changed B's role"), never raw keys (guidelines §13).
- `app/organization/` org audit view and `app/admin/` platform audit view: shared
  table/pagination, filter by entity type/actor/date via the shared search platform.

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Read the audit trail for a specific entity (scope-checked).
- Read the org-scoped audit log (Org Admin) and the platform-wide log (Super Admin), paginated
  and filterable.

## Permissions

- Audit-viewing capabilities registered via `modules/audit/permissions.ts` (view org audit, view
  platform audit); write access is internal-only — no endpoint creates audit entries directly.

## Validation Rules

- Event types come from the typed vocabulary; entries validated to carry actor, entity, action,
  and scope; payload summaries screened so no credential, token, or sensitive value is ever
  recorded (guidelines §8.7). Read queries bounded, paginated, filter inputs validated.

## Edge Cases & Failure Scenarios

- Audit store unavailable during a sensitive mutation: per the decided policy, fail or
  queue-retry — never silent loss.
- Entries referencing since-deleted entities: still render meaningfully (retained descriptor).
- High-volume growth: pagination and indexed reads from day one; retention policy is an open
  question below (guidelines §8.7 classification/retention).
- A viewer without the audit capability opening an entity page: the trail section is gated
  (F1.3.3), the rest of the page unaffected.

## Acceptance Criteria

- Every Phase 1 sensitive mutation listed above produces an entry with who/what/when and no
  sensitive values.
- No application code path can modify or delete an entry (append-only proof).
- The trail is visible on organization and user detail surfaces, and in the scoped audit views,
  correctly tenant-scoped.
- Event vocabulary is typed; a grep finds no ad-hoc audit strings at call sites.

## Definition of Done

- `rules/definition-of-done.md` gates (esp. §3 Data Presentation, §7 Lists) plus guidelines §8.7,
  §8.8, and §18 observability boundaries (audit ≠ logging). Specific: the producer checklist
  (which mutations must audit) documented so later phases follow it.
