# F1.3.2 — Role Management

**Epic:** E1.3 — RBAC & Permission Platform · **Complexity:** Large

## Description

Role management as a product feature (PRD §4.1 "Role Management", "Permission Management"): roles
are named bundles of capability grants drawn from the catalog (F1.3.1), edited through a role
editor UI. Default role templates ship as seeds — Super Admin, Org Admin, Faculty, Student — and
are grant sets, not magic identities: renaming a role changes nothing about what it can do,
because enforcement matches grants, never names (guidelines §9). Role assignment attaches a role
to a user's membership (F1.2.1).

## Dependencies

- F1.3.1 (the catalog is the only source of grantable capabilities).
- F1.2.1 (org-scoped role context), F1.3.3 (guards that make grants real), F1.4.4 (audit of role
  changes).

## Database Changes (conceptual)

- A role concept in prose: name, description, its set of capability grants (catalog references),
  scope (platform-level vs. organization-level — see open questions), template/custom origin,
  timestamps. Role-to-user attachment lives on the membership concept from F1.2.1.

## Backend Changes

- `modules/users/` (or a dedicated roles home inside it) services for role CRUD, grant editing,
  and assignment; repositories as the only data call sites; validators for all payloads.
- Seeds for the four default templates (`prisma/seed.ts` home per folder-structure §8) expressed
  as grants against the catalog.
- Role mutations audited: who changed which grants on which role, when (guidelines §8.8) — the
  role-change propagation checklist of guidelines §9 applies to every catalog-affecting change.

## Frontend Changes

- Role editor UI in the Super Admin portal (`app/admin/`), and — if org-scoped custom roles are
  confirmed — an org-scoped editor in `app/organization/`: role list (shared table/pagination),
  grant editor grouped by resource with human labels (never raw keys, guidelines §13), dependency
  hints driven by the catalog's dependency map (authoring aids only; the server grant is the real
  gate, guidelines §9), destructive actions behind confirm dialogs.

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List/read roles with their grants.
- Create/update/delete a role; edit its capability grants.
- Assign/unassign a role to a user's membership.

## Permissions

- Role-management capabilities (manage roles, assign roles) registered in the catalog; granted by
  default to Super Admin, and to Org Admin only per the scoping decision below. One authorization
  check per action (guidelines §9).

## Validation Rules

- Role name required and bounded; grants must reference live catalog entries; dependency map
  respected (a dependent grant without its parent is rejected server-side); deleting a role in
  use follows the reassignment rule below. All server-side, Zod-validated.

## Edge Cases & Failure Scenarios

- Deleting a role still assigned to users: blocked until reassignment, or explicit bulk
  reassignment — **Needs clarification** which; never silently orphan users.
- Editing a default template: allowed but audited; templates re-seed only on fresh installs
  (**Assumption** — no seed-overwrite of admin edits).
- Removing a grant users are actively relying on: takes effect on next authorization evaluation;
  affected UI gates update accordingly (F1.3.3).
- A role granting a retired capability: evaluated as no-op per F1.3.1, surfaced in the editor for
  cleanup — never a dead toggle (guidelines §9).

## Acceptance Criteria

- The four default templates exist after seeding and confer exactly their documented grants.
- Renaming any role provably changes no authorization outcome.
- Creating a custom role and assigning it changes what the user can do (server and UI) without
  any code change.
- Every role/grant/assignment mutation appears in the audit trail.

## Definition of Done

- `rules/definition-of-done.md` gates (esp. §7 Lists, §9 Forms, §14 Consistency) plus guidelines
  §9 in full. Specific: default template grant sets documented; role-editor shows no raw
  permission keys (guidelines §13).

## Open Questions

- **Needs clarification:** can Org Admins create org-scoped custom roles, or is role editing
  Super-Admin-only? PRD §4.1 places role management in the Super Admin portal only; this spec
  scopes the editor to Super Admin until decided.
- **Needs clarification:** policy for deleting an in-use role (block vs. reassign flow).
