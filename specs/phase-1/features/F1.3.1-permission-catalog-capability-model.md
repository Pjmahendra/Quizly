# F1.3.1 — Permission Catalog & Capability Model

**Epic:** E1.3 — RBAC & Permission Platform · **Complexity:** Large

## Description

The single permission vocabulary for the entire platform (PRD §4.1 "Permission Management";
guidelines §9): a central catalog in `server/permissions/` of resources, actions, and capability
flags, into which every module registers its own entries via its `permissions.ts`
(folder-structure §4–§5, §9). Authorization is modeled as structural capability grants
(resource + action as the default; capability flags for switches an action cannot express;
dedicated guards only for conditional gates) — never role-name matching. Permission dependencies
are modeled data-drivenly in a map, not in branching logic. This is the highest-blast-radius
subsystem of the platform; a written design precedes implementation (guidelines §1, §3, §9).

## Dependencies

- None at runtime beyond Phase 0 — the catalog is foundational; F1.2.1 and all guard enforcement
  (F1.3.3) consume it.
- The reviewed written design (guidelines §3) is the true predecessor.

## Database Changes (conceptual)

- The catalog itself is code (declarative registrations), not data. Persistence appears only
  where grants meet roles (F1.3.2) — in prose: roles hold sets of catalog capability references.
  No user-editable permission definitions exist outside the catalog.

## Backend Changes

- `server/permissions/`: catalog structure, registration mechanism, typed capability references,
  the dependency map, and the query surface guards use ("does this scope hold this capability?").
- Each Phase 1 module (`auth`, `users`, `organizations`, `audit`) ships its `permissions.ts`
  registering its resources/actions — no permission strings at call sites (folder-structure §9).
- Retirement path defined: removing a permission edits catalog + guards, never mutates assigned
  role rows in a feature migration (guidelines §9).

## Frontend Changes

- None user-facing in this feature; the catalog feeds the role editor (F1.3.2) and frontend gates
  (F1.3.3). A typed client-side representation of capability references is exposed for those
  features.

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Expose the catalog (resources, actions, flags, dependency map, human labels) to authorized
  admin surfaces for the role editor and gate mirroring.

## Permissions

- Reading the catalog for role editing is itself capability-gated (role-management capability,
  F1.3.2). The catalog defines, but is also governed by, the vocabulary — bootstrap grants come
  from the seeded Super Admin template.

## Validation Rules

- Registrations validated at startup: unique resource/action identifiers, no orphan dependency
  references, human label present for every entry (never show raw keys to users —
  guidelines §13). Duplicate or near-duplicate permission surfaces rejected in review
  (reuse-before-adding, guidelines §9).

## Edge Cases & Failure Scenarios

- Two modules registering the same resource/action: startup-time failure, not silent override.
- A grant referencing a retired catalog entry: ignored safely at evaluation and flagged for
  cleanup — never an open gate.
- Dependency map cycles: rejected at validation.
- Catalog unavailable at guard time: fail closed (default deny, guidelines §8.2).

## Acceptance Criteria

- Exactly one vocabulary: every permission referenced anywhere (guards, role editor, frontend
  gates, seeds) resolves to a catalog entry; a grep finds no permission string literals at call
  sites.
- Adding a module's permissions touches only that module's `permissions.ts` plus catalog wiring —
  data-driven variant flow (guidelines §1.3).
- Dependency relationships evaluate from the map; no `if role ===` logic anywhere.
- The written design is reviewed and captured in `docs/architecture.md`.

## Definition of Done

- `rules/definition-of-done.md` gates plus guidelines §9 in full (single vocabulary, minimum
  enforcement mechanism, retirement rules) and §17 tests covering catalog validation. Specific:
  the propagation checklist (guard → catalog → seeds → gates → denial UX → role editor → docs →
  tests) documented for future permission changes.

## Open Questions

- Granularity of Phase 1 resources/actions (e.g. one `users` resource vs. faculty/students
  split) — settled in the written design, guided by reuse-before-adding (guidelines §9).
- **Assumption:** capability grants are org-scoped via F1.2.1's context, with Super Admin
  platform-wide at the guard layer.
