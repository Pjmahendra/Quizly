# F1.2.1 — Organization Entity & Tenant Scoping Context

**Epic:** E1.2 — Organizations & Multi-Tenancy Core · **Complexity:** Large

## Description

The single most load-bearing feature of the phase: the organization as a first-class tenant
entity, and the trusted server-side scoping context through which every scoped query in every
module — now and in all later phases — derives its organization/user key (guidelines §1.1, §8.2;
CLAUDE.md "Multi-tenancy is foundational"). The context is resolved once per request from the
authenticated session (never from client input), exposed to module services, and enforced at the
repository seam — the only Prisma call sites (folder-structure §8) — giving tenant scoping a
single enforcement point per domain. Defense-in-depth (relational constraints, row-level policies
if adopted) is the last line, not the first (guidelines §1.1).

## Dependencies

- E1.1 — the scope is derived from the authenticated principal.
- Blast radius is maximal: per guidelines §1/§3, a **written design reviewed before
  implementation** is required for the context's shape and the repository contract.

## Database Changes (conceptual)

- An organization entity in prose: identity, name, status, lifecycle timestamps.
- A membership concept in prose linking users to an organization with their role context — the
  source from which the resolver answers "which org, acting as what".
- Every future scoped entity references its organization — a conceptual invariant this feature
  establishes, not a schema enumerated here.

## Backend Changes

- `modules/organizations/` with the standard anatomy; its `repositories/` are the only
  organization Prisma call sites.
- A scope-context resolver (server-side) mapping session → { user, organization, capabilities },
  the sole input to scoped repository calls; Super Admin's platform-wide context is an explicit
  guard-layer construct — not a magic null org (guidelines §9 super-user rule).
- Repository conventions documented so every later module's repositories take scope from this
  context, never from DTOs.

## Frontend Changes

- Minimal: a provider surfacing the active organization context to portal layouts (`app/admin`,
  `app/organization`, …) for display and routing; no org-switching UI unless the multi-org
  question below resolves to "yes".

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Resolve/return the caller's current organization context for portal bootstrapping.
- (Organization CRUD endpoints belong to F1.2.2.)

## Permissions

- `modules/organizations/permissions.ts` registers the organization resource and its actions into
  the central catalog (F1.3.1). The scope context itself is infrastructure — it carries
  capabilities, it is not gated by them.

## Validation Rules

- Any resource ID arriving in a request is verified to belong to the caller's tenant scope before
  acting (IDOR/BOLA, guidelines §8.2). Client-supplied organization identifiers are never trusted
  as scope — at most a hint validated against membership.

## Edge Cases & Failure Scenarios

- Authenticated user with no membership (e.g. pre-approval): a designed "no organization" state.
- Suspended/deleted organization: members fail closed with role-framed copy.
- Cross-tenant ID probing: responds as "not found" — never confirms existence in another tenant.
- Scope resolver failure: request fails closed; nothing proceeds unscoped.

## Acceptance Criteria

- Every scoped repository method demonstrably requires the server-derived scope; a code-review
  greppable convention makes an unscoped query obvious.
- Cross-tenant read and write attempts fail closed, indistinguishable from "not found".
- Super Admin bypass exists only at the guard layer.
- The written design for the context is reviewed and recorded in `docs/architecture.md`.

## Definition of Done

- `rules/definition-of-done.md` gates plus guidelines §1.1/§8.2 verified explicitly; §17 testing
  matrix must include cross-tenant negative tests. Specific: architecture documentation updated
  (CLAUDE.md primary rule) before completion.

## Open Questions

- **Needs clarification:** can a user belong to more than one organization (e.g. an instructor at
  two colleges)? PRD is silent; the membership model must decide before implementation.
- **Assumption:** organization deletion is soft/recoverable given its blast radius — confirmed or
  reversed in F1.2.2.
