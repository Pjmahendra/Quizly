# F1.4.1 — User Invitations

**Epic:** E1.4 — User Management & Audit · **Complexity:** Medium

## Description

Organization admins invite faculty and students into their organization (PRD §4.2 "Invite
Users", "Manage Faculty", "Manage Students"). An invitation is a single-use, expiring, emailed
link (Resend, PRD §12) that carries trusted server-side binding to the target organization and
intended role — so an invitee's registration lands in the right tenant with the right default
role template without trusting any client input (guidelines §1.1). Invitations can be listed,
re-sent, and revoked. The same machinery serves F1.2.3's org-admin invitations issued by Super
Admin.

## Dependencies

- F1.1.1/F1.1.2 (invitees complete registration + verification), F1.2.1 (org binding),
  F1.3.2 (role templates to bind), F1.4.4 (audit), `server/mail/`.

## Database Changes (conceptual)

- An invitation concept in prose: inviter, target email, bound organization, bound role template,
  single-use token semantics, expiry, status (pending/accepted/revoked/expired), timestamps.

## Backend Changes

- `modules/users/services/` invitation issue/resend/revoke/accept orchestration;
  `repositories/`, `validators/`, `dto/` per module anatomy; org scope always from the F1.2.1
  context — an org admin can only ever invite into their own org.
- `server/mail/templates/system/` invitation email; issuance rate-limited (guidelines §8.8).
- Acceptance consumes the token, creates/links the account through F1.1.1's flow, attaches
  membership + role, and audits the event.

## Frontend Changes

- `app/organization/` invitations surface composing `modules/users/components/`: invite form
  (email + role, constrained-choice picker — guidelines §13) in a side sheet; pending-invitations
  list via shared table/pagination/search; resend/revoke actions (revoke behind confirm dialog).
- Invitee landing route under `app/(auth)/` pre-bound to org/role with clear framing.

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Create an invitation (email + intended role) within the caller's organization.
- List/search invitations with status; resend; revoke.
- Accept an invitation (token-bearing registration entry point).

## Permissions

- Invitation capabilities (invite users, manage invitations) registered via
  `modules/users/permissions.ts`; granted to Org Admin (own org) and Super Admin. Which roles an
  inviter may bind is itself capability-constrained (an org admin cannot invite a Super Admin).

## Validation Rules

- Email format-validated; role must be a live, permitted template; duplicate pending invitation
  to the same email in the same org handled explicitly (supersede or block — expansion-time
  choice); token expiry/single-use server-enforced; issuance bounded and rate-limited.

## Edge Cases & Failure Scenarios

- Invitee already has an account: acceptance links membership instead of duplicating identity
  (multi-org behavior gated on F1.2.1's open question — fail closed until decided).
- Expired/revoked token used: designed invalid-invitation state with "contact your admin" copy.
- Inviter's org suspended before acceptance: acceptance fails closed.
- Mistyped email: revoke + re-issue path; the old token dies with revocation.

## Acceptance Criteria

- Org admin invites a faculty member and a student; each receives email, registers, verifies, and
  lands in the correct organization with the correct role's capabilities.
- Revoked and expired invitations provably cannot be accepted.
- Issue/resend/revoke/accept all appear in the audit trail.
- Org admins cannot invite outside their own organization (scope proof).

## Definition of Done

- `rules/definition-of-done.md` gates (esp. §4 Workflow Completion, §7 Lists, §9 Forms) plus
  guidelines §8.2/§8.8/§9. Specific: invitation email contains no sensitive data beyond the
  link; token never logged (guidelines §8.7).
