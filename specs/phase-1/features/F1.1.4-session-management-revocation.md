# F1.1.4 — Session Management & Revocation

**Epic:** E1.1 — Authentication & Sessions · **Complexity:** Medium

## Description

The session model for the platform: short-lived access credentials with rotating refresh, carried
in cookies with the full protection set (HttpOnly, Secure, SameSite), revocable and enumerable
(guidelines §8.3). A user can list active sessions/devices and revoke any; reuse of a rotated
refresh token is treated as compromise and revokes the session family. **Flagged conflict:**
PRD §16 lists "JWT Sessions" while guidelines §8.3 prescribe cookie-carried rotating sessions;
per CLAUDE.md precedence this spec adopts §8.3 and surfaces the PRD wording rather than hiding it
(guidelines §0). The chosen provider (TBD, PRD §17) may constrain mechanics — a second reason the
provider decision blocks this epic.

## Dependencies

- F1.1.1 (sign-in establishes sessions); **auth provider decision (blocking)**.
- F1.4.4 audit platform for session revocation events.

## Database Changes (conceptual)

- A session concept in prose: session/refresh lineage per account with device descriptor,
  created/last-active timestamps, and revocation state — enough to enumerate and revoke. Redis
  may hold active-session state (PRD §17); the authoritative shape follows the provider decision.

## Backend Changes

- `server/auth/` owns token/cookie plumbing: issuance, rotation, verification, and the
  reuse-detection → family-revocation rule.
- `modules/auth/services/` session listing and revocation orchestration;
  `modules/auth/repositories/` as the only data call sites.
- Revocation events (self-service and operator-initiated via F1.4.3) audited (guidelines §8.8).

## Frontend Changes

- A "Sessions/Devices" surface under `app/settings/` composing `modules/auth/components/`:
  session list (shared table component, DoD §7), current-session marker, revoke action behind a
  confirm dialog (guidelines §13), "sign out everywhere" affordance.

## AI Components

## API Endpoints (purpose-level)

- List the current user's active sessions/devices.
- Revoke a specific session; revoke all other sessions.
- Refresh/rotate the session credential (transport mechanics per provider).

## Permissions

- Self-service session management requires only authentication. Operator revocation of another
  user's sessions is capability-gated in F1.4.3, registered via `modules/auth/permissions.ts`.

## Validation Rules

- Session identifiers validated as belonging to the calling user before revocation (IDOR guard,
  guidelines §8.2); all inputs schema-validated.

## Edge Cases & Failure Scenarios

- Rotated-refresh reuse: revoke the family, force re-authentication, audit it (guidelines §8.3).
- Revoking the current session: signs the user out immediately with clear feedback.
- Race between refresh and revocation: revocation wins; a revoked session cannot rotate back in.
- Expired access credential with valid refresh: silent rotation, no user-visible error.

## Acceptance Criteria

- Session cookies carry HttpOnly + Secure + SameSite; access is short-lived; refresh rotates.
- The sessions list shows every active session with device context; revoking one demonstrably
  terminates it (that device's next request fails authentication).
- Refresh-reuse triggers family revocation and an audit entry.
- Sign-out, password reset (F1.1.3), and operator actions (F1.4.3) all revoke through this one
  mechanism — no parallel revocation paths.
