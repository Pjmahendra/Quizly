# F1.3.3 — Authorization Guards & Frontend Gate Mirroring

**Epic:** E1.3 — RBAC & Permission Platform · **Complexity:** Large

## Description

Enforcement, end to end: server-side guards that authorize every request against catalog
capabilities (default deny — authenticate every request, authorize every mutation,
guidelines §8.2), plus a frontend gating layer that mirrors the backend exactly — hiding or
disabling what the API would reject, revealing what it would allow — with the server remaining
the authority (guidelines §9). Permission-denial UX is distinct from load-failure UX: the user is
told it is a permission issue and pointed to who can grant access (guidelines §9, §13).
Enforcement is in depth: transport guard + business-logic check + data-layer scoping (F1.2.1),
with no single layer's failure opening access — while keeping one authorization check per
concern, not dual-gating (guidelines §8.2, §9).

## Dependencies

- F1.3.1 (catalog as the vocabulary), F1.2.1 (scope the grants evaluate in), F1.3.2 (roles supply
  grants).
- E1.1 (an authenticated principal to authorize).

## Database Changes (conceptual)

- None. Guards evaluate existing grant data; denials may emit audit events (policy below).

## Backend Changes

- `server/permissions/` guard enforcement primitives consumed by module services and server
  actions: capability checks, super-user bypass at the guard layer only, typed permission-denied
  error distinct from not-found/failure (`lib/errors.ts` base per folder-structure §9).
- Convention: server actions/route handlers stay thin; authorization lives in module services
  (folder-structure §3–§4) using catalog references — never string literals.

## Frontend Changes

- A gating primitive (hook/component in `src/components`/`src/hooks` as feature-agnostic UI)
  reading the session's capabilities: hide, disable-with-reason, or deny-page per surface.
- Portal layouts (`app/admin`, `app/organization`, `app/faculty`, `app/student`) gate navigation
  so users never see sections the server would reject.
- Distinct denial components: "you lack access, ask <role/admin>" vs. "something broke, retry"
  (guidelines §13 "distinguish 'you lack access' from 'it broke'"; DoD §2, §15).

## Socket Events

None this phase (Socket.IO auth middleware arrives with Phase 2; it will consume these same
primitives — noted for design, not built now).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Expose the current session's effective capabilities for frontend gate mirroring (shape shared
  with F1.3.1's typed references).

## Permissions

- This feature is the enforcement mechanism itself; it introduces no domain permissions beyond
  what modules register. Super-user bypass is implemented here, at the guard layer, never in
  feature code (guidelines §9).

## Validation Rules

- Guards run before business logic on every mutation; capability references are typed catalog
  entries, validated at build/startup (F1.3.1). Effective-capability responses contain grants
  only — no sensitive user data.

## Edge Cases & Failure Scenarios

- Grants changed mid-session (role edited, F1.3.2): server reflects immediately; the frontend's
  mirrored state refreshes on a defined trigger — a stale UI gate must only ever fail toward the
  server's answer.
- Guard evaluation error: fail closed as denial, logged (never as silent allow).
- Direct API calls bypassing the UI: rejected identically — the frontend gate is never the only
  gate (guidelines §8.1 "assume every client is malicious").
- Deep link into a denied surface: renders the permission-denied state, not a crash or blank page.

## Acceptance Criteria

- Every Phase 1 mutation endpoint is guard-protected; an unauthorized call fails with the typed
  permission-denied error, distinct from not-found and load failure.
- For each gated UI surface, the visible/enabled state matches what the server would answer —
  verified across the four default role templates.
- Denial UX copy is role-framed and points to who can grant access; load failures use different
  copy (guidelines §13).
- Super-user bypass demonstrably lives in exactly one place.

## Definition of Done

- `rules/definition-of-done.md` gates (esp. §2 Data States, §8 Feedback, §15 Error Recovery) plus
  guidelines §8.2 and §9; §17 matrix includes authorized/denied test pairs per endpoint.
  Specific: the mirroring convention documented so later phases gate new surfaces the same way.

## Open Questions

- **Assumption:** denials are logged for abuse detection but not written to the entity audit
  trail (denials are not mutations) — confirm desired observability (guidelines §8.8, §18).
- Refresh trigger for mirrored capabilities (on navigation vs. push) — implementation-time
  decision; Phase 2's socket layer may improve it.
