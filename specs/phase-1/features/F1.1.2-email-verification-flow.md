# F1.1.2 — Email Verification Flow

**Epic:** E1.1 — Authentication & Sessions · **Complexity:** Small

## Description

Token-based email verification completing registration (guidelines §8.3 "design account flows for
verification … from the start"). On registration (F1.1.1) the system emails a single-use,
expiring verification link via Resend (PRD §12, §14, §17). Following the link marks the account
verified; unverified accounts cannot sign in to a portal. Re-sending is supported and
rate-limited with the same rigor as login (guidelines §8.3).

## Dependencies

- F1.1.1 (registration creates the unverified account).
- `server/mail/` Resend adapter with system email templates (folder-structure §5).
- Auth provider decision (TBD, PRD §17) — providers ship differing verification primitives; the
  flow below is the product contract regardless of provider.

## Database Changes (conceptual)

- A verification-token concept in prose: single-use token bound to an account, with expiry and
  consumed-at semantics; account verification status flips exactly once. Whether tokens are
  provider-managed or first-party depends on the provider decision.

## Backend Changes

- `modules/auth/services/` verification issue/consume/resend logic; token generation is
  cryptographically random, stored non-reversibly if first-party.
- `server/mail/templates/system/` verification email template.
- Rate limiting on resend; audit event on verification completion (F1.4.4).

## Frontend Changes

- `app/(auth)/` verification landing route (thin shell) with success, already-verified,
  expired-token, and invalid-token states — each a designed state, not a blank panel
  (guidelines §13; DoD §2 Data States).
- "Check your email" interstitial after registration with a resend affordance.

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Consume a verification token and mark the account verified.
- Re-send the verification email for an account, rate-limited.

## Permissions

- Token consumption is unauthenticated (the token is the credential); resend is rate-limited and
  never confirms account existence. No new catalog entries.

## Validation Rules

- Token presence/shape validated; expiry and single-use enforced server-side. Resend input
  validated and bounded (guidelines §8.4).

## Edge Cases & Failure Scenarios

- Expired token: clear copy and a one-click resend path (DoD §15 Error Recovery).
- Token reused after consumption: treated as invalid; no state change.
- Resend for an already-verified or nonexistent account: identical outward response — no
  existence leak.
- Email delivery failure: retry guidance surfaced; failures logged without logging the token
  (guidelines §8.7).

## Acceptance Criteria

- Register → receive email → click link → account verified → sign-in succeeds.
- Tokens are single-use and expire; both properties provably enforced server-side.
- Resend is rate-limited and existence-safe.
- All verification outcomes have designed UI states.

## Definition of Done

- `rules/definition-of-done.md` gates (esp. §2 Data States, §8 Feedback, §15 Error Recovery)
  plus guidelines §8.3/§8.8. Specific: verify no token value ever appears in logs
  (guidelines §8.7).

## Open Questions

- Token lifetime is an implementation-time policy choice — record it in the expanded spec.
