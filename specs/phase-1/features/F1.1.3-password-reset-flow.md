# F1.1.3 — Password Reset Flow

**Epic:** E1.1 — Authentication & Sessions · **Complexity:** Small

## Description

Self-service "forgot password" flow (guidelines §8.3 — reset, never retrieve). The user requests
a reset by email; if an account exists, a single-use, expiring reset link is sent via Resend
(PRD §12). The outward response is identical whether or not the account exists — no
account-existence leak. Completing the flow sets a new password (hashed per F1.1.1) and, as a
safety measure, revokes existing sessions (F1.1.4). Passwords are never recoverable by any actor,
including admins — admin-initiated resets (F1.4.3) reuse this flow's machinery, they do not view
or set passwords directly.

## Dependencies

- F1.1.1 (credential model), F1.1.4 (session revocation on reset), `server/mail/` Resend adapter.
- Auth provider decision (TBD, PRD §17) — reset primitives differ by provider.

## Database Changes (conceptual)

- A reset-token concept in prose: single-use, expiring, bound to an account, consumed exactly
  once; issuance may invalidate prior outstanding reset tokens. Provider-managed vs. first-party
  follows the provider decision.

## Backend Changes

- `modules/auth/services/` request/consume logic; `modules/auth/validators/` for both steps.
- Rate limiting on the request endpoint (guidelines §8.8); constant-shape responses regardless of
  account existence.
- On successful reset: revoke all active sessions for the account and write an audit event
  (F1.4.4) — event recorded without the credential value (guidelines §8.7, §8.8).

## Frontend Changes

- `app/(auth)/` routes: request form ("enter your email"), sent confirmation (same copy whether
  the account exists), and new-password form with inline validation (guidelines §13).
- Expired/invalid token states with a restart path (DoD §15 Error Recovery).

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Request a password reset for an email address (existence-safe, rate-limited).
- Consume a reset token and set a new password.

## Permissions

- Both steps unauthenticated (the token is the proof); no new catalog entries. Admin-triggered
  resets are a separate, capability-gated concern in F1.4.3.

## Validation Rules

- Email format validated on request; new password revalidated server-side against the same
  policy as registration; token expiry and single-use enforced server-side (guidelines §8.4).

## Edge Cases & Failure Scenarios

- Request for a nonexistent email: identical response and timing profile to a real one.
- Expired or already-consumed token: invalid-token state, restart path offered.
- Multiple outstanding requests: only the latest token is honored (**Assumption** — confirm at
  expansion).
- Reset completed while other devices hold sessions: those sessions are revoked (F1.1.4).

## Acceptance Criteria

- Full flow works: request → email → set new password → sign in with it; old password rejected.
- No response, error, or timing signal reveals account existence.
- No code path recovers or displays an existing password, for any role.
- Completing a reset revokes the account's other active sessions and is audited.

## Definition of Done

- `rules/definition-of-done.md` gates plus guidelines §8.3/§8.4/§8.7/§8.8. Specific: security
  review of the token lifecycle; confirm reset emails contain no sensitive data beyond the link.

## Open Questions
