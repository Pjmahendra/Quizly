# F1.4.2 — Registration Approval Workflow

**Epic:** E1.4 — User Management & Audit · **Complexity:** Medium

## Description

Org admins review and approve pending registrations (PRD §4.2 "Approve Registrations"). A
registration that is not invitation-borne enters a pending state attached to an organization;
until approved, the account cannot act inside the org. Admins see a queue of pending
registrations and approve (attaching membership + role) or reject (with the account's fate per
the open policy question below).

**Needs clarification (flagged at phase level):** PRD §4.2 lists both "Invite Users" (F1.4.1) and
"Approve Registrations" without saying how an uninvited registrant becomes associated with an
organization in the first place — open self-registration with org selection? org-specific
sign-up links? invite-only with approval only as a safety net? This product decision shapes the
registration form (F1.1.1), this queue, and tenant-binding trust. **This spec cannot be
implemented until it is resolved**; the structure below covers the queue mechanics common to all
answers.

## Dependencies

- F1.1.1/F1.1.2 (registrations exist and are verified), F1.2.1 (pending org association),
  F1.3.2 (role attached on approval), F1.4.4 (audit).
- The clarification above — blocking for this feature.

## Database Changes (conceptual)

- A pending-registration concept in prose: the account, the organization it seeks to join, the
  requested/assumed role, status (pending/approved/rejected), decided-by/decided-at.

## Backend Changes

- `modules/users/services/` queue listing and approve/reject orchestration; approval composes
  membership creation (F1.2.1) and role attachment (F1.3.2); all org-scoped from server context.
- Decision notifications to the registrant via `server/mail/` (**Assumption:** an email on
  approve/reject is expected; PRD §14 does not enumerate these).
- Approve/reject audited with who/when (guidelines §8.8).

## Frontend Changes

- `app/organization/` pending-registrations queue: shared table/pagination/search, approve/reject
  actions (reject behind confirm dialog with reason field if the policy adopts reasons), helpful
  empty state ("no pending registrations") — guidelines §13; DoD §7.
- Registrant-facing "awaiting approval" state after verification (F1.2.1's "no organization"
  state specialization).

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List pending registrations for the caller's organization.
- Approve a pending registration (attach membership + role); reject one.

## Permissions

- Approval capabilities registered via `modules/users/permissions.ts`; granted to Org Admin (own
  org) and Super Admin. Default deny for everyone else (guidelines §8.2).

## Validation Rules

- Decisions validated against pending status (no double-decide); target must belong to the
  caller's org scope; role attached on approval must be a live, permitted template. Server-side,
  Zod-validated.

## Edge Cases & Failure Scenarios

- Two admins deciding the same registration concurrently: first decision wins; the second gets a
  clear already-decided outcome, not silent overwrite (guidelines §4.5 idempotency).
- Registrant unverified at decision time: approval still requires verification before sign-in
  (F1.1.2 rule holds).
- Rejected registrant re-registers: policy question below.
- Queue growth: paginated and bounded from day one (guidelines §1.2).

## Acceptance Criteria

- An uninvited (per resolved policy) registration appears in the right org's queue; approval
  grants membership + role and the user can act; rejection prevents org access.
- Decisions are audited and visible on the affected user's trail.
- Org admins see and decide only their own org's queue (scope proof).
