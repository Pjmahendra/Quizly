# F1.4.3 — Global & Org User Management

**Epic:** E1.4 — User Management & Audit · **Complexity:** Large

## Description

User administration at two scopes. Super Admin: global user management across all organizations
(PRD §4.1 "Global User Management"). Org Admin: manage their own faculty and students —
view/search, edit administrative attributes, activate/deactivate, change role assignment, and
trigger admin-initiated password resets (PRD §4.2 "Manage Faculty", "Manage Students", "Reset
Passwords"). Admin resets reuse F1.1.3's machinery — an admin sends a reset, never sees or sets
a password (guidelines §8.3). Operators can also revoke a user's sessions via F1.1.4 (the
"operator" case of guidelines §8.3 session enumerability). Every action is capability-gated,
tenant-scoped, and audited.

## Dependencies

- E1.1 (accounts, reset, sessions), F1.2.1 (scope), F1.3.1–F1.3.3 (capabilities + guards),
  F1.4.4 (audit).

## Database Changes (conceptual)

- No new entities expected: operates on the user/membership concepts (F1.1.1, F1.2.1) plus an
  account-status notion (active/deactivated) if not already modeled there. Deactivation semantics
  in prose: a deactivated user cannot sign in and existing sessions are revoked.

## Backend Changes

- `modules/users/` services for both scopes: global operations guarded by Super Admin
  capabilities; org operations bound to the caller's org from server context — the org key never
  arrives from the client (guidelines §1.1).
- Admin reset-trigger and session-revocation orchestration composing `modules/auth` via its
  `index.ts` (folder-structure §4).
- All mutations audited: status changes, role changes, reset triggers, session revocations
  (guidelines §8.8).

## Frontend Changes

- `app/admin/` global users area and `app/organization/` faculty/students areas (thin shells,
  module components): shared table + always-visible pagination + debounced search that never
  loses focus (guidelines §13; DoD §7); user detail with membership, role, status, sessions, and
  the entity's audit trail; edits in side sheets; deactivation and session revocation behind
  confirm dialogs (guidelines §13).

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List/search users — globally (Super Admin) and within the caller's org (Org Admin), paginated.
- Read a user's administrative detail (membership, role, status, sessions, audit trail).
- Update administrative attributes; activate/deactivate; change role assignment.
- Trigger a password-reset email for a user; revoke a user's sessions.

## Permissions

- Distinct capabilities per action tier (view users, manage users, trigger resets, revoke
  sessions, deactivate) registered via `modules/users/permissions.ts`; Super Admin holds them
  platform-wide, Org Admin org-scoped — reusing actions on the one users resource rather than
  minting near-duplicates (guidelines §9).

## Validation Rules

- Every target user ID verified to be within the caller's scope before acting (IDOR,
  guidelines §8.2); role changes restricted to templates the caller may grant (an org admin
  cannot promote to Super Admin); status transitions validated. All list reads bounded and
  paginated (guidelines §1.2).

## Edge Cases & Failure Scenarios

- Admin deactivating themselves or the org's last admin: guarded (ties to F1.2.3's last-admin
  question).
- Deactivating a user with active sessions: sessions revoked as part of the action.
- Super Admin acting on a user in a suspended org: allowed (global scope) but audited.
- Concurrent role change and deactivation: consistent final state, both audited.

## Acceptance Criteria

- Org admin can find, edit, deactivate/reactivate, re-role, reset-trigger, and session-revoke
  their own faculty/students — and provably cannot touch users of another org.
- Super Admin can do the same globally.
- A deactivated user cannot sign in and their sessions are dead.
- Every administrative mutation is on the affected user's visible audit trail.

## Definition of Done

- `rules/definition-of-done.md` gates (esp. §3 Data Presentation, §7 Lists, §15 Error Recovery)
  plus guidelines §1.1/§8.2/§8.3/§8.8/§9; §17 matrix includes cross-tenant negative tests.
  Specific: no admin code path returns or accepts a raw password.

## Open Questions

- **Needs clarification:** may org admins hard-delete users, or only deactivate? PRD says
  "Manage" without specifying; this spec assumes deactivate-only pending decision (data
  retention implications).
- **Assumption:** "Manage Organization Admins" editing (beyond assignment, F1.2.3) happens
  through this feature's global scope.
