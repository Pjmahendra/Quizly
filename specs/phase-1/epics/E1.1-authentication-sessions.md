# E1.1 — Authentication & Sessions

## Purpose

Deliver the complete credential authentication lifecycle — registration, email verification,
sign-in, password reset — and a session model that is short-lived, rotating, enumerable, and
revocable (PRD §4, §16; guidelines §8.3). Everything in later phases assumes an authenticated
principal; this epic creates it.

## Business Value

- No portal (PRD §4.1–4.4) can exist without trustworthy identity.
- Secure-by-default account flows (verification, reset, rate limiting) are PRD §16 security
  requirements and guidelines §8.3 mandates — building them first avoids retrofitting.
- A revocable session model gives institutions the operational control (kick a compromised
  account) that educational customers expect.

## Dependencies

- Phase 0 scaffold; Resend configured for verification/reset email (PRD §12, §14).
- **Blocking decision: authentication provider — Better Auth vs. Clerk (PRD §17). Implementation
  of every feature below is gated on this choice.** Specs are provider-agnostic; the provider is
  isolated behind `server/auth/` (folder-structure §5) so the decision changes an adapter, not
  the domain module.
- **Flagged tension:** PRD §16 lists "JWT Sessions"; guidelines §8.3 prescribe cookie-carried
  sessions (HttpOnly/Secure/SameSite) with short-lived access + rotating refresh tokens.
  Per CLAUDE.md precedence, guidelines govern engineering practice — the epic follows §8.3 and
  records the conflict rather than silently picking (guidelines §0).

## Acceptance Criteria

- A user can register, receive a verification email, verify, sign in, reset a forgotten password,
  and sign out — the full lifecycle of the phase brief.
- Credentials are hashed with a memory-hard algorithm (Argon2id-class); passwords are never
  recoverable, only resettable (guidelines §8.3).
- Auth endpoints (sign-in, registration, verification, reset) are rate-limited (PRD §16;
  guidelines §8.8).
- Reset and verification flows never reveal whether an account exists.
- A user can list their active sessions/devices and revoke any of them; refresh-token reuse is
  treated as compromise and revokes the session family (guidelines §8.3).
- The model is MFA-ready by design without shipping MFA (PRD §16; guidelines §8.3) — adding a
  factor later must not require re-architecting sessions.
- All features pass `rules/definition-of-done.md`.
