# E1.4 — User Management & Audit

## Purpose

Give administrators the tools to populate and govern their tenants — invitations, registration
approval, global and org-level user management, admin password resets — and give the platform an
append-only audit trail for sensitive mutations, surfaced where the affected entities are viewed
(PRD §4.1–4.2, §16; guidelines §8.8). Includes user profile self-management.

## Business Value

- Institutions onboard hundreds of faculty and students; invitations and approvals (PRD §4.2) are
  the funnel that fills every classroom and quiz in later phases.
- Admin password resets and user lifecycle controls (PRD §4.1–4.2) are day-one operational needs
  for org admins.
- Audit logs are both a PRD §4.1 Super Admin feature and a PRD §16 security requirement;
  building the audit platform now means every later sensitive mutation (quiz publishing, grade
  overrides) writes to an existing trail instead of bolting one on.

## Dependencies

- E1.1 (accounts and auth lifecycle), F1.2.1 (tenant scoping — org admins manage only their org),
  E1.3 (capabilities gating every management action).
- Resend for invitation email (PRD §12, §14).
- **Needs clarification (F1.4.2):** PRD §4.2 lists both "Invite Users" and "Approve
  Registrations" — whether self-registration is open, org-gated, or invite-only is a product
  decision that shapes both features.

## Features

| ID | Feature | Complexity |
|---|---|---|
| F1.4.1 | User invitations | Medium |
| F1.4.2 | Registration approval workflow | Medium |
| F1.4.3 | Global & org user management | Large |
| F1.4.4 | Audit log platform | Large |
| F1.4.5 | Profile self-management | Small |

## Acceptance Criteria

- An org admin can invite faculty and students into their organization; invitees land in the
  registration flow already bound to the right tenant and role (PRD §4.2).
- Pending registrations are visible to org admins and can be approved or rejected (PRD §4.2).
- Super Admin has global user management across organizations; org admins manage only their own
  faculty and students, including admin-initiated password resets (PRD §4.1–4.2) — every action
  capability-gated and tenant-scoped.
- Sensitive mutations across Phase 1 (auth events, org lifecycle, role/permission changes, user
  administration) write who/what/when to an append-only trail — never sensitive values — and the
  trail is visible where the affected entity is viewed (guidelines §8.8).
- A user can update their own name, password, and preferences; profile images are explicitly
  deferred to the Phase 5 storage platform.
- All features pass `rules/definition-of-done.md`.
