# E1.3 — RBAC & Permission Platform

## Purpose

Build capability-based authorization as a platform: a single permission catalog of resources and
actions in `server/permissions/` that modules register into, role management over structural
grants (never role-name matching), and server guards mirrored exactly by frontend gates
(PRD §4.1 Permission Management / Role Management; guidelines §9; folder-structure §5, §9).

## Business Value

- Every feature in every later phase gates through this subsystem — it is the
  highest-blast-radius component of the platform (guidelines §3, §9). Building it as a platform
  now means each later module adds catalog entries instead of inventing authorization.
- PRD §4.1 makes permission and role management a Super Admin product feature, not just
  infrastructure — institutions need to shape who can do what.
- Mirrored frontend gates with distinct denial UX (guidelines §9, §13) prevent the two classic
  failures: users seeing actions the server rejects, and "broken page" confusion when access is
  simply not granted.

## Dependencies

- E1.1 (authenticated principal) and F1.2.1 (tenant scope — grants are evaluated within an
  organization context; Super Admin operates platform-wide).
- Because blast radius is maximal, guidelines §3/§6-of-§1 apply: **a written design for the
  capability model precedes implementation** of this epic.

## Acceptance Criteria

- Exactly one permission vocabulary exists — the catalog in `server/permissions/`, populated by
  each module's `permissions.ts` registrations; no permission strings at call sites
  (guidelines §9; folder-structure §4, §9).
- Authorization decisions use structural capability grants (resource + action, plus capability
  flags where an action cannot express the switch) — never human-readable role-name matching
  (guidelines §9).
- Default role templates exist for Super Admin, Org Admin, Faculty, and Student as seeded grant
  sets; renaming a role changes nothing about what it can do.
- Every mutation endpoint is authorized (default deny, guidelines §8.2); super-user bypass lives
  at the guard layer only.
- The frontend hides or disables exactly what the server would reject; permission-denial copy is
  distinct from load-failure copy and points to who can grant access (guidelines §9, §13).
- Permission changes propagate through all layers in one change: guard → catalog → role
  templates/seeds → frontend gates → denial UX → role editor → docs → tests (guidelines §9).
- All features pass `rules/definition-of-done.md`.
