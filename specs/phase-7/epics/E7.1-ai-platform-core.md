# E7.1 — AI Platform Core

## Purpose

Build the platform layer every AI capability must flow through: a swappable AI provider behind a
single interface (`server/ai/` client, `modules/ai/providers/` abstraction, `config/ai.ts` typed
configuration), prompts managed as versioned runtime-loaded assets, and a parser that turns
non-deterministic model output into a validated structure before anything is stored
(folder-structure §5, §7, §11). This epic encodes the guidelines §24 mandates in code: no
privileged infrastructure access from the AI layer, versioned prompts, untrusted input isolated
from instructions, no prompt/response payloads in logs.

## Business Value

- The AI provider decision is currently open (PRD §17 names none) — an abstraction seam lets
  product evaluation of the flagship feature proceed while keeping the provider swappable, the
  same pattern that lets Azurite become Azure Blob (folder-structure §10).
- Versioned prompts make grading behavior explainable and reproducible: an audit record can say
  exactly which prompt version produced a suggestion (guidelines §24; F7.3.1).
- Structured output validation is what makes AI suggestions safe to store next to real grades —
  a hard prerequisite for faculty trust in the flagship feature (PRD §11).
- One platform seam now avoids re-architecting when post-MVP batch evaluation (PRD §11) arrives:
  the queue would call the same provider/prompt/parser stack.

## Dependencies

- Phase 1 permission catalog and audit platform for guarding and auditing callers (F1.3.x,
  F1.4.4) — the AI layer itself acts only on behalf of an authenticated, authorized caller
  (guidelines §24).
- **Blocking decision: AI provider and model selection — unspecified in the PRD (§17).
  Needs clarification.** All features below are specified provider-agnostic; implementation of
  F7.1.1's concrete adapter is gated on this choice.
- No dependency on E7.2/E7.3 — this epic is upstream of both.

## Acceptance Criteria

- Exactly one domain-facing provider interface exists in `modules/ai/providers/`; no caller
  outside `modules/ai/` touches a provider SDK or endpoint directly, and swapping providers
  changes an adapter, never a caller (folder-structure §5, §11).
- The AI layer holds no database, secrets, or storage access; it receives inputs and returns
  outputs through the module boundary with the caller's identity forwarded (guidelines §24).
- Every prompt used at runtime is a versioned asset under a `prompts/` directory, loaded at
  runtime; no prompt text is inlined in code (guidelines §24; folder-structure §11).
- Untrusted content (student answers) is structurally separated from instructions during prompt
  assembly and treated as data (guidelines §24).
- No AI response is persisted or surfaced until it has passed schema validation (score bounds,
  required feedback/reasoning fields); malformed output is retried within a bounded policy and
  then fails typed, never silently (guidelines §7).
- No prompt or response payload appears in any log at any level (guidelines §24, §8).
- All features pass `rules/definition-of-done.md`.
