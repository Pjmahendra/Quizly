# E7.2 — AI Evaluation Workflow

## Purpose

Deliver the faculty-facing flagship workflow of PRD §11: configure per-question grading behavior
(parameters and rubrics), trigger AI evaluation of an individual subjective response from the
manual evaluation workbench, and review the suggested score, feedback, and reasoning — accepting,
modifying, or overriding it before publishing. The epic covers PRD §11 workflow steps 2 and 5–9
for descriptive, short, long-form, and explanatory answers, strictly per-response (PRD §11 "AI
Evaluation Scope"). Domain code lives in `modules/evaluations/` with AI glue under
`modules/evaluations/ai/` and faculty review UI under `modules/evaluations/components/`
(folder-structure §11).

## Business Value

- This is the product's stated flagship feature and core differentiator (PRD §11, §1): grading
  subjective answers is the most expensive faculty activity the platform can reduce.
- Configurable parameters and rubrics (PRD §11) let institutions encode their own grading
  culture — strict or lenient, keyword-driven or semantic — instead of a one-size-fits-all model.
- Keeping faculty in control (review/override before publish) is what makes AI grading adoptable
  in institutions: the AI proposes, the faculty disposes (PRD §11 step 9).
- Graceful failure to the manual path protects the Phase 3 evaluation capability from any AI
  outage (PRD §16 reliability).

## Dependencies

- **E7.1 (all features)** — provider abstraction, versioned prompts, and output validation are
  the machinery this workflow calls.
- **Phase 3** — attempts and stored subjective answers; the manual evaluation workbench (F3.2.2)
  is the exact surface F7.2.2 and F7.2.3 extend, and the existing publish flow is reused.
- **Phase 2** — reference/expected answers per question (F2.2.3), required input to evaluation
  (PRD §11 step 2).
- **Phase 1** — capability guards for the new evaluate-with-AI permission; tenant scoping from
  server context on every query (guidelines §1.1, §9).
- Phase 5 notifications, optionally, for "Evaluation Finished" (PRD §14) — must not be a hard
  dependency.

## Acceptance Criteria

- Faculty can configure, per subjective question: fuzzy matching sensitivity, case sensitivity,
  semantic relevance, keyword importance, partial marking, strict/lenient grading, maximum
  score, custom grading instructions, and a rubric (PRD §11 "Configurable Evaluation
  Parameters").
- From an individual submission, **Evaluate with AI** produces a suggested score, feedback, and
  reasoning derived from the configured parameters and the reference answer (PRD §11 steps 5–8).
- Evaluation state is always explicit (requested / running / succeeded / failed); re-triggering
  is idempotent — no duplicate in-flight evaluations, no orphaned suggestions.
- Every failure path lands the faculty back on a fully functional manual grading flow with a
  distinct, actionable failure message (PRD §16; guidelines §7, §13).
- Nothing is published to a student without explicit faculty action; accept, modify, and
  override are all recorded with their disposition (PRD §11 step 9; consumed by F7.3.1).
- Objective question types never expose an AI evaluation affordance (PRD §10 vs. §11).
- All features pass `rules/definition-of-done.md`.
