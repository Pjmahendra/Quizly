# F7.1.1 — AI Provider Abstraction & Configuration

**Epic:** E7.1 — AI Platform Core · **Complexity:** Medium

## Description

A single, swappable seam between the application and whichever AI provider is chosen: the raw
provider client lives in `server/ai/` (folder-structure §5), `modules/ai/providers/` wraps it
behind one domain-facing interface, and `modules/ai/services/` is the only entry point callers
use (folder-structure §11). Typed, env-derived configuration (model identifier, timeouts, retry
policy, token limits) lives in `config/ai.ts` (folder-structure §7). The AI layer has no
privileged infrastructure access — no database, storage, or secrets beyond its own credential;
it acts only on inputs passed across the module boundary with the caller's identity forwarded
(guidelines §24). The provider/model itself is an open decision: the PRD's stack table (§17)
names none. **Needs clarification** — interface work can proceed; the concrete adapter is
blocked on the decision.

## Dependencies

- **Blocking decision: AI provider and model selection (PRD §17 specifies none). Needs
  clarification.**
- Phase 1 for validated environment plumbing (`lib/env.ts`) and typed error base classes
  (`lib/errors.ts`) (folder-structure §7, §9).

## Database Changes (conceptual)

None owned here. The model identifier used for an evaluation is recorded on the evaluation
record owned by F7.2.2; provider configuration is environment-derived, not stored.
**Assumption:** one platform-wide provider/model in MVP — no per-organization model selection.

## Backend Changes

- `server/ai/` — provider client, credential handling, transport.
- `modules/ai/providers/` — the provider interface (request envelope in, raw completion out),
  one adapter per provider; shared shapes in `modules/ai/types/`.
- `modules/ai/services/` — public entry point: timeout, bounded retry for transient transport
  errors, typed error mapping (provider-unavailable, timed-out, rate-limited, invalid-config)
  per guidelines §7.
- `config/ai.ts` — model id, timeout, retry bounds, max input/output tokens.
- Metadata-only structured logging (duration, model id, outcome) — never prompt/response
  contents (guidelines §24, §18).

## Frontend Changes

None. Faculty-facing surfaces belong to E7.2.

## Socket Events

None — a synchronous internal service with no client-facing real-time behavior.

## AI Components

The provider interface itself; the model identifier surfaced to callers so evaluation records
(F7.2.2) and audit entries (F7.3.1) can attribute output to a model version.

## API Endpoints (purpose-level)

- None exposed. Internal layer; only module services (F7.2.2) call it — no route handler talks
  to a provider directly.

## Permissions

None of its own. Callers (`modules/evaluations/`) authorize before the AI layer is reached; the
caller's identity is forwarded for audit (guidelines §24).

## Validation Rules

- Config validated at boot via the validated-env pattern; missing/invalid AI config fails
  startup loudly, not at first use.
- Request envelope enforces the configured max input size before any provider call.

## Edge Cases & Failure Scenarios

- Provider outage/5xx: bounded retry, then typed provider-unavailable error.
- Provider rate limiting: distinct typed error so F7.2.2 can message "try again shortly".
- Timeout mid-request: abandoned at the configured deadline; caller decides disposition.
- Model deprecated/renamed: config validation catches unknown model at boot.
- Credential exposure: the credential exists only in `server/ai/` and is never logged.

## Acceptance Criteria

- All AI traffic flows through one interface in `modules/ai/`; provider SDK imports exist only
  under `server/ai/` (folder-structure §5, §11).
- Swapping providers is demonstrably an adapter change with zero caller edits.
- Every failure mode maps to a typed error; no raw provider errors cross the module boundary.
- No prompt/response payloads or credentials in any log (guidelines §24).

## Definition of Done

Per `rules/definition-of-done.md`, plus: the provider decision recorded or re-flagged as
blocking; guidelines §24 checklist reviewed for this layer.
