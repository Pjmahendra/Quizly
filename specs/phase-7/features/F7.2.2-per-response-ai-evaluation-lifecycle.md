# F7.2.2 — Per-Response AI Evaluation Lifecycle

**Epic:** E7.2 — AI Evaluation Workflow · **Complexity:** Large

## Description

The core execution path of PRD §11 steps 5–7: from an individual submission in the manual
evaluation workbench (F3.2.2), faculty trigger **Evaluate with AI** for one subjective response.
The service in `modules/evaluations/services/` orchestrates the folder-structure §11 flow — load
response, reference answer (F2.2.3), and resolved configuration (F7.2.1) → prompt build
(F7.1.2, via `modules/evaluations/ai/` glue) → provider call (F7.1.1) → parse and validate
(F7.1.3) → store the suggestion. Each evaluation has an explicit state — requested, running,
succeeded, failed — re-triggering is idempotent, and every failure degrades gracefully to the
untouched manual grading path (PRD §16). Strictly per-response: no batching, no queue (PRD §11
MVP scope).

## Dependencies

- E7.1 complete (F7.1.1–F7.1.3); Phase 3 attempts/answers and workbench (F3.2.2); Phase 2
  reference answers (F2.2.3); F7.2.1 configuration; Phase 1 capabilities and audit platform.
- Optionally Phase 5 notifications for "Evaluation Finished" (PRD §14) — decoupled; the flow
  must not depend on it.

## Database Changes (conceptual)

An AI evaluation record per trigger, attached to a specific subjective response: state,
requester, timestamps, model identifier, prompt version, the configuration snapshot used, the
validated suggestion (score, feedback, reasoning, optional rubric breakdown), and a failure
class when failed. Raw prompts and raw provider output are never persisted (guidelines §24).
**Assumption:** re-evaluation creates a new record superseding the previous one (history kept
for F7.3.1) rather than overwriting in place.

## Backend Changes

- Orchestration service with a per-response concurrency guard: at most one in-flight evaluation
  per response; duplicate triggers return the in-flight evaluation.
- State transitions enforced in code (requested → running → succeeded | failed) — no other paths.
- Timeout around the provider call; failure classes mapped from F7.1.1/F7.1.3 typed errors
  (provider, timeout, rate-limited, malformed-output).
- Audit hook on every trigger and terminal state (consumed by F7.3.1) — metadata only.
- Eligibility checks: subjective type, reference answer present, attempt submitted.

## Frontend Changes

- **Evaluate with AI** action on the individual response in the workbench, visible only for
  eligible responses and permitted users.
- Running state (disabled trigger, progress), success handoff to the F7.2.3 review panel,
  failure state with distinct copy per failure class, retry, and always-available manual
  grading (guidelines §13).

## Socket Events

None. Evaluation completes within the triggering faculty request/response cycle; there is no
multi-client state to broadcast. The optional "Evaluation Finished" notification (PRD §14) goes
through the notifications module, not sockets.

## AI Components

Consumer of the entire E7.1 stack; owns the evaluation-specific glue in
`modules/evaluations/ai/` and the prompt-variable mapping from response + reference answer +
configuration. Student answers enter prompts only through F7.1.2's delimited data slots
(guidelines §24).

## API Endpoints (purpose-level)

- Trigger AI evaluation for one specific response (faculty; idempotent per in-flight evaluation).
- Read the evaluation state/result for a response, including failure class.

## Permissions

- New capability registered via the module's `permissions.ts`: evaluate-with-AI, granted to
  faculty who can already evaluate the attempt; tenant-scoped from server context, never client
  input (guidelines §1.1, §9).

## Validation Rules

- Server-side eligibility: subjective response, submitted attempt, reference answer present,
  requester authorized for the attempt's quiz.
- Triggers rejected while an evaluation for the same response is in flight (idempotency).
- Suggestions stored only after F7.1.3 validation passes — no partial writes.

## Edge Cases & Failure Scenarios

- Empty student answer: no provider call — deterministic handling. **Needs clarification:**
  zero-suggestion vs. prompt-graded minimum.
- Answer exceeding model input limits: typed failure, manual path (truncation policy open,
  F7.1.2).
- Timeout/provider failure mid-run: record moves to failed with its class; retry allowed.
- Double-click or two faculty triggering concurrently: single in-flight evaluation wins.
- Config edited between triggers: each record carries its own snapshot.
- Re-evaluation after a disposition: prior disposition kept for audit; new suggestion requires
  fresh review (F7.2.3). **Needs clarification:** is re-evaluating an already-published
  response allowed?
- Crash after the provider responds but before storage: stale running records are reaped to
  failed by a staleness bound; re-trigger permitted.

## Acceptance Criteria

- From a submission, faculty trigger evaluation and receive a validated suggestion derived from
  the reference answer and configured parameters (PRD §11 steps 5–8).
- State is always explicit and queryable; no evaluation is ever stuck invisible.
- All failure classes degrade to a fully functional manual path with distinct messaging.
- Repeated triggers never create duplicate in-flight evaluations or orphaned records.
- Every trigger and terminal state is audited without payloads (guidelines §24).

## Definition of Done

Per `rules/definition-of-done.md`, plus: failure-mode walkthrough (provider down, timeout,
malformed output, concurrent triggers) reviewed against guidelines §7 and §24.

## Open Questions
