# F7.1.2 — Versioned Prompt Management

**Epic:** E7.1 — AI Platform Core · **Complexity:** Medium

## Description

Prompts are versioned assets, never strings in code (guidelines §24). General prompt assets live
under `modules/ai/prompts/`; evaluation-specific prompts under `modules/evaluations/prompts/`
(folder-structure §11). Each asset carries an explicit version identifier; a loader resolves the
active version at runtime and returns it alongside the rendered prompt so every evaluation
record (F7.2.2) and audit entry (F7.3.1) can state which prompt version produced a result.
Prompt assembly is also where untrusted-input isolation is enforced: platform and faculty
instructions are template content, while the student answer is injected only into clearly
delimited data slots and never interpreted as instructions (guidelines §24).

## Dependencies

- F7.1.1 — the provider interface consumes assembled prompts.
- Phase 2 reference answers and F7.2.1 parameters define the variables evaluation prompts
  render (consumed via `modules/evaluations/`, not owned here).

## Database Changes (conceptual)

None. Prompt assets are files versioned in the repository; the database stores only the prompt
version identifier on evaluation records owned by F7.2.2. **Assumption:** repository-versioned
file assets satisfy guidelines §24 "versioned assets stored as data/files" — no database-backed
prompt store or admin prompt-editing UI in MVP.

## Backend Changes

- Prompt asset format and directory conventions: template body, declared variables, version
  identifier.
- A loader/registry in `modules/ai/` that loads assets at runtime, validates that all declared
  variables are supplied, and returns rendered prompt + version id.
- Assembly utility enforcing instruction/data separation: untrusted values are
  escaped/delimited and cannot terminate their data slot.
- Typed errors for missing prompt, unknown version, unfilled variables (guidelines §7).

## Frontend Changes

None. Prompts are not user-editable in MVP; faculty influence grading through F7.2.1 parameters
and custom instructions, which render into declared variables.

## Socket Events

None — prompt loading is an internal, synchronous concern.

## AI Components

The evaluation prompt family (per-response subjective grading), the loader/registry, and the
version identifiers consumed by F7.2.2 storage and F7.3.1 audit.

## API Endpoints (purpose-level)

- None. Prompts are internal assets; no endpoint reads or mutates them.

## Permissions

None of its own — no user-facing surface; callers are authorized upstream.

## Validation Rules

- An asset without a version identifier fails loading.
- Rendering fails typed if any declared variable is missing or an undeclared one is supplied —
  prompts and call sites cannot drift silently.
- Untrusted-input slots are mandatory-escaped; there is no raw-interpolation path.

## Edge Cases & Failure Scenarios

- Student answer contains injection text ("ignore previous instructions…"): rendered as inert
  data inside its delimited slot; F7.1.3 guards the output side.
- Prompt version bumped mid-flight: each evaluation resolves its version once at trigger time
  and records that identifier.
- Oversized rendered prompt (long answer + rubric) exceeding model input limits: size reported
  so F7.2.2 fails typed rather than truncating silently. **Needs clarification:** truncation
  policy for very long student answers.
- Malformed template asset: caught by loader validation at startup, not at first evaluation.

## Acceptance Criteria

- No prompt text inlined in application code; all runtime prompts resolve from versioned assets
  under a `prompts/` directory (guidelines §24; folder-structure §11).
- Every rendered prompt carries its version identifier, and F7.2.2 stores it.
- Untrusted input demonstrably cannot escape its data slot (adversarial fixtures reviewed).
- Changing a prompt requires a new version identifier — used versions are immutable.

## Definition of Done

Per `rules/definition-of-done.md`, plus: prompt-format conventions documented so future prompt
authors follow the same variable/versioning rules.
