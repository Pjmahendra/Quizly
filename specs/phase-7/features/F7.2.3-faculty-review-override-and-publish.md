# F7.2.3 — Faculty Review, Override & Publish

**Epic:** E7.2 — AI Evaluation Workflow · **Complexity:** Medium

## Description

The faculty-control half of the flagship feature (PRD §11 steps 8–9): the suggested score,
feedback, and reasoning from F7.2.2 are presented inside the manual evaluation workbench
(F3.2.2), and the faculty member **accepts** the suggestion as-is, **modifies** score and/or
feedback, or **overrides** it entirely with their own grade. Nothing is published to a student
without this explicit faculty action — there is no auto-publish path, configurable or otherwise
(PRD §11: "Faculty retains complete control and may override AI-generated scores before
publishing"). Publishing itself reuses the existing Phase 3 result-publication flow; this
feature extends it with AI provenance, never forks it (guidelines §25 "parallel
implementation"). Review UI lives in `modules/evaluations/components/` (folder-structure §11).

## Dependencies

- F7.2.2 — the stored, validated suggestion and its states.
- Phase 3 — the manual evaluation workbench (F3.2.2) and the existing grade/publish flow this
  feature augments.
- F7.3.1 consumes the disposition data recorded here.

## Database Changes (conceptual)

The final grade on a response gains provenance: a reference to the AI evaluation record it was
derived from (when any) and a disposition — accepted, modified, or overridden — plus who acted
and when. The AI suggestion itself is never mutated; the faculty's final grade is a separate
value, so suggestion and outcome remain comparable (feeds override-rate analytics).
**Assumption:** the student-facing published feedback is the faculty-approved text, which may
start from the AI feedback; students are not shown raw AI reasoning. **Needs clarification:**
whether students should ever see that AI assisted the evaluation.

## Backend Changes

- `modules/evaluations/services/` — record disposition atomically with the final grade
  (multi-step write in a transaction, guidelines §6); enforce that publish requires a faculty
  grading action, never a raw suggestion.
- Guard: a suggestion in failed/running state cannot be accepted; a superseded suggestion
  (re-evaluated since viewed) cannot be accepted stale.

## Frontend Changes

- Review panel on the workbench: suggested score, feedback, and reasoning; the configuration
  summary from F7.2.1 for context; clearly labeled Accept / Modify / Override actions.
- Editable score/feedback fields prefilled on Accept/Modify; untouched manual fields on
  Override.
- Provenance labels on graded responses (AI-suggested → accepted/modified/overridden vs. purely
  manual) so the grading trail is legible at a glance.

## Socket Events

None — review and publish are single-user request/response interactions on the workbench;
result-publication notifications remain the existing flow's concern.

## AI Components

None invoked directly; this feature renders and disposes of F7.2.2's validated suggestion.

## API Endpoints (purpose-level)

- Record a disposition (accept/modify/override) with the resulting final grade for a response
  (faculty, tenant-scoped).
- Read suggestion + disposition history for a response (workbench view).

## Permissions

- Reuses the existing Phase 3 evaluation and result-publication capabilities; disposition
  requires the same capability as manual grading of that attempt (guidelines §9). No new
  student-facing permission.

## Validation Rules

- Final score bounded by the question's maximum regardless of the suggestion.
- Disposition must reference a succeeded, current (non-superseded) evaluation record.
- Modified/override actions require the changed values server-side revalidated (guidelines §8 —
  the server is the gate).

## Edge Cases & Failure Scenarios

- Faculty graded manually first, then triggers AI: suggestion arrives as advisory; existing
  grade is not replaced without an explicit action.
- Re-evaluation lands while the review panel is open: stale suggestion blocked at accept with a
  refresh prompt.
- Two faculty review the same response concurrently: last write is rejected on version conflict
  rather than silently overwriting (guidelines §6 concurrency).
- Suggestion accepted but publish fails downstream: grade + disposition persist; publish is
  retryable through the existing flow.

## Acceptance Criteria

- Faculty can accept, modify, or override every suggestion; each path records its disposition
  and actor (PRD §11 step 9).
- No route exists — API or UI — by which an AI suggestion reaches a student without a faculty
  grading action; code review confirms no auto-publish configuration.
- Suggestion values remain immutable and comparable to final grades (override-rate analytics
  verified against known fixtures in F7.3.1).
- Provenance is visible on the workbench for every AI-assisted grade.

## Definition of Done

Per `rules/definition-of-done.md`, plus: the no-auto-publish invariant documented as a
non-negotiable in the module's docs.

## Open Questions

- **Needs clarification:** student visibility of AI assistance (above) — ties into the Phase 3
  result-dispute flow (PRD §4.4).
- **Needs clarification:** may faculty publish AI-derived feedback verbatim without edits by
  default, or should organizations be able to require a modification step?
