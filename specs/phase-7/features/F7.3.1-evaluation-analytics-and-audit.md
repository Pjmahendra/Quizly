# F7.3.1 — Evaluation Analytics & Audit

**Epic:** E7.3 — Evaluation Analytics & Hardening · **Complexity:** Medium

## Description

Makes AI-assisted evaluation accountable and measurable: the "advanced evaluation analytics" of
PRD §18 Phase 3 plus the guidelines §24 audit mandate. Every AI action — trigger, completion,
failure, faculty disposition — is recorded through the Phase 1 append-only audit platform
(F1.4.4) with metadata only: actor, response identity, timestamp, model identifier, prompt
version, outcome. Prompt, answer, feedback, and reasoning payloads are never written to audit or
logs (guidelines §24, §8). On top of the evaluation records from F7.2.2/F7.2.3, analytics
surfaces show override-rate visibility (accepted vs. modified vs. overridden, score delta
between suggestion and final grade) and failure rates by class — per quiz and per question for
faculty, aggregated for org admins (PRD §15). Dashboards live in `modules/analytics/`; audit
recording flows through `modules/audit/` (folder-structure §4).

## Dependencies

- F7.2.2 and F7.2.3 — evaluation records, states, and dispositions are the source data.
- F7.1.1/F7.1.2 — model identifier and prompt version to attribute.
- Phase 1 audit platform (F1.4.4) and analytics-view capabilities (F1.3.x).
- Existing faculty/organization analytics surfaces from earlier phases — extended, not forked.

## Database Changes (conceptual)

No new source-of-truth entities. Audit entries go to the existing append-only audit log;
analytics aggregate the evaluation and disposition records that F7.2.2/F7.2.3 already persist.
**Assumption:** MVP volumes (per-response, faculty-triggered) allow on-read aggregation with
bounded, paginated queries — no precomputed rollups until measurement says otherwise
(guidelines §10, §25 "future flexibility").

## Backend Changes

- Audit emission points in `modules/evaluations/` for trigger, terminal state, and disposition —
  one catalog of evaluation audit event types, no ad-hoc strings (folder-structure §9 spirit).
- Aggregation services in `modules/analytics/` (via repositories): disposition counts and rates,
  suggestion-vs-final score deltas, failure counts by class; per quiz, per question, org-level.
- All queries tenant-scoped from server context; bounded and paginated (guidelines §1.1, §10).

## Frontend Changes

- Faculty: per-quiz evaluation analytics view — AI usage, disposition breakdown, average score
  delta, failure rate; drill-down per question.
- Org admin: aggregated evaluation analytics on the institutional analytics surface (PRD §4.2,
  §15).
- Audit entries visible where the affected entity is viewed, per the Phase 1 audit presentation
  pattern (guidelines §8.8).

## Socket Events

None — analytics and audit are read/report surfaces with no real-time requirement in the PRD.

## AI Components

None invoked; this feature observes AI activity. Attribution fields (model id, prompt version)
come from E7.1 via the evaluation records.

## API Endpoints (purpose-level)

- Read evaluation analytics for a quiz (faculty scope): dispositions, deltas, failures.
- Read organization-level evaluation analytics (org admin scope).
- Audit entries surface through the existing Phase 1 audit read paths — no new audit API.

## Permissions

- Reuses faculty analytics capability for quiz-level views and org-admin analytics capability
  for institutional views (guidelines §9); audit visibility follows the Phase 1 audit
  platform's rules. Students see none of this.

## Validation Rules

- Audit writes validate that no payload-bearing field is present — enforced by the audit entry
  shape, not reviewer vigilance.
- Analytics date-range and grouping inputs validated server-side; unbounded ranges rejected.

## Edge Cases & Failure Scenarios

- Audit write fails during an evaluation action: the action must not silently lose its trail —
  same transactional posture as Phase 1 audit (**Assumption:** audit write participates in the
  action's transaction; confirm against F1.4.4's decision).
- Zero AI usage on a quiz: analytics render empty states, not errors (guidelines §13).
- Superseded evaluations (re-triggers): analytics count the latest suggestion per response for
  rates, while audit retains every action.
- Prompt version bump mid-quiz: deltas remain attributable per version — regression in a new
  prompt version is visible rather than averaged away.

## Acceptance Criteria

- Every AI evaluation action appears in the audit trail with actor, target, time, model id,
  prompt version, and outcome — and demonstrably no payloads (guidelines §24).
- Faculty can see, per quiz and question, how often suggestions were accepted, modified, or
  overridden, and the average suggestion-vs-final delta.
- Failure rates are visible by class (provider, timeout, malformed output).
- All views are tenant-scoped, capability-gated, paginated, and pass empty-state UX rules.

## Definition of Done

Per `rules/definition-of-done.md`, plus: a payload-leak review of every audit/log emission point
added by Phase 7.
