# F7.1.3 — Structured Output Parsing & Validation

**Epic:** E7.1 — AI Platform Core · **Complexity:** Medium

## Description

The gate between a non-deterministic model and the platform's grading data. Raw provider output
is parsed in `modules/ai/parser/` into the canonical evaluation-suggestion structure — suggested
score, feedback, reasoning — and validated with Zod before anything is stored or shown
(folder-structure §11 "structured JSON out"; PRD §17). Output that is not valid JSON, is missing
required fields, or violates bounds (score above the question's maximum) is rejected and the
model re-asked within a bounded retry policy; exhaustion produces a typed malformed-output
failure that F7.2.2 turns into an explicit failed evaluation. Raw model output is never logged
and never persisted un-validated (guidelines §24).

## Dependencies

- F7.1.1 — raw completions come from the provider interface.
- F7.1.2 — prompts instruct the model toward the expected structure; parser and prompt version
  evolve together.
- F7.2.1 — maximum score and rubric shape define validation bounds, passed in by the caller
  (this layer reads nothing from the database, per guidelines §24).

## Database Changes (conceptual)

None. This feature defines the validated in-memory structure; persistence of suggestions
belongs to F7.2.2.

## Backend Changes

- `modules/ai/parser/` — extraction (tolerant of wrapper noise around the JSON), schema
  validation, normalization into the canonical suggestion structure.
- Zod schemas for the suggestion shape: bounded numeric score, non-empty feedback and reasoning
  with length ceilings, optional per-criterion rubric breakdown when a rubric was supplied
  (**Assumption:** rubric evaluations return a per-criterion breakdown; PRD §11 names
  rubric-based evaluation without specifying output granularity).
- Bounded retry orchestration for malformed output (distinct from F7.1.1's transport retries),
  retry counts in config.
- Typed errors: malformed-output, out-of-bounds-score, retry-exhausted (guidelines §7);
  metadata-only logging (outcome, attempt count, duration) — never the payload.

## Frontend Changes

None. The validated structure is rendered by F7.2.3's review UI.

## Socket Events

None — parsing happens inside the synchronous evaluation call.

## AI Components

The canonical suggestion schema; the parse-validate-retry loop; normalization rules — notably,
out-of-range scores are rejected, never silently clamped.

## API Endpoints (purpose-level)

- None. Internal layer invoked by `modules/evaluations/` services.

## Permissions

None of its own — inherits the caller's authorization context.

## Validation Rules

- Score: numeric, ≥ 0, ≤ the caller-supplied maximum; rejected otherwise.
- Feedback and reasoning: required, non-empty, length-capped; unknown fields stripped.
- Rubric breakdown (when present): criteria must match the supplied rubric; per-criterion
  scores within their weights.
- The full structure validates before any storage or display — partial results are discarded.

## Edge Cases & Failure Scenarios

- Prose instead of JSON, or truncated JSON (token ceiling): parse failure → bounded retry →
  typed failure.
- Model refuses to evaluate (safety refusal text): treated as malformed output, not a score.
- Injection artifacts — output echoing instructions or adding fields: unknown fields stripped;
  structural mismatch fails validation (defense in depth behind F7.1.2).
- Boundary scores (exactly 0 or max): valid — must not be rejected.
- Retries return different valid results: last valid result wins; this non-determinism is why
  faculty review (F7.2.3) is mandatory.

## Acceptance Criteria

- No unvalidated AI output can reach storage or a user surface — the parser is the only path
  from provider output to the suggestion structure.
- Malformed output retries within configured bounds, then fails typed; the failure class is
  distinguishable from transport failure (for F7.3.1 failure analytics).
- Out-of-range scores are rejected, never clamped or stored.
- Logs contain outcome metadata only; adversarial review confirms no payload leakage.
