# F7.2.1 — Evaluation Parameter & Rubric Configuration

**Epic:** E7.2 — AI Evaluation Workflow · **Complexity:** Large

## Description

Faculty configure, per subjective question, how AI evaluation should grade (PRD §11
"Configurable Evaluation Parameters"): fuzzy matching sensitivity, case sensitivity, semantic
relevance, keyword importance, partial marking, strict or lenient grading, maximum score, custom
grading instructions (free text, as in the PRD §11 example), and rubric-based evaluation with
weighted criteria. Configuration is authored alongside the question's reference answer (F2.2.3)
in the faculty question editor and consumed at evaluation time by F7.2.2's prompt build.
Parameter/rubric shapes, validators, and defaults are owned by `modules/evaluations/`, with the
authoring component exported through its `index.ts` and composed into the question editor
(folder-structure §4). **Assumption:** ownership in `modules/evaluations/` rather than
`modules/questions/` — the config is meaningless outside evaluation; review at implementation.

## Dependencies

- Phase 2 question authoring and reference answers (F2.2.3) — the surface and data this
  configuration attaches to.
- Phase 1 permission catalog (faculty question-edit capability).
- Downstream: F7.2.2 consumes the configuration; F7.1.2 renders it into prompt variables.

## Database Changes (conceptual)

Each subjective question can carry one evaluation configuration: the scalar parameters above,
an optional ordered rubric of weighted criteria, and optional custom-instruction text. Defaults
apply when absent so AI evaluation works with only a reference answer. **Assumption:** F7.2.2
snapshots the configuration onto the evaluation record at trigger time so later edits don't
silently re-contextualize past suggestions.

## Backend Changes

- `modules/evaluations/validators/` — Zod schemas for parameters and rubrics; `dto/` boundary
  shapes; defaults in `constants.ts`.
- Services to read/write a question's configuration, tenant-scoped from server context, via
  `modules/evaluations/repositories/` (folder-structure §8).
- A "resolved configuration" accessor for F7.2.2: stored values merged with defaults.

## Frontend Changes

- Configuration panel in the question editor (subjective types only): parameter controls,
  rubric builder (criteria, descriptions, weights), custom-instructions field with the PRD §11
  example as guidance, live max-score/rubric-weight consistency feedback.
- Read-only summary of the effective configuration on the evaluation workbench (for F7.2.3).

## Socket Events

None — standard request/response authoring with no real-time behavior.

## AI Components

None directly — this feature produces the inputs F7.1.2 renders into prompt variables. Custom
instructions are faculty-authored template content, distinct from untrusted student answers
(guidelines §24).

## API Endpoints (purpose-level)

- Read a question's evaluation configuration (faculty, tenant-scoped).
- Create/update a question's evaluation configuration (faculty with question-edit capability).

## Permissions

- Reuses the faculty question-edit capability for writes; workbench viewers may read. No new
  student-facing surface (guidelines §9).

## Validation Rules

- Maximum score: positive and consistent with the question's marks. **Needs clarification:**
  may PRD §11's per-question "Maximum score" differ from the question's quiz-level marks?
- Sensitivity/importance values constrained to enums or bounded ranges — no free numerics.
- Rubric: at least one criterion when present; positive weights that reconcile with max score.
- Custom instructions: length-capped, sanitized as data (guidelines §8).
- Configuration attachable only to subjective question types (PRD §11 vs. §10).

## Edge Cases & Failure Scenarios

- Config edited after responses were evaluated: existing suggestions keep their snapshot; the
  workbench marks them as predating the current config; re-evaluation uses the new one (F7.2.2).
- No reference answer: configuration savable, but AI evaluation blocked with clear messaging
  (PRD §11 step 2 makes the expected answer the anchor).
- Contradictory settings (strict grading + generous partial marking): permitted — parameters
  are advisory to the model — but surfaced in the UI summary.
- Rubric weights not matching max score: blocked at save with actionable copy (guidelines §13).

## Acceptance Criteria

- Every PRD §11 parameter is configurable per subjective question and demonstrably alters the
  prompt variables sent at evaluation time.
- Defaults let a question with only a reference answer be AI-evaluated with zero extra setup.
- Invalid configurations cannot be saved; errors are field-specific.
- Objective question types never show the configuration surface.

## Definition of Done

Per `rules/definition-of-done.md`, plus: parameter semantics documented so prompt authors
(F7.1.2) and faculty-facing help stay consistent.
