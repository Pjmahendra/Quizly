# F2.2.1 — Question bank core

**Epic:** E2.2 — Question Bank · **Complexity:** Large

## Description

The foundational question-bank capability (PRD §4.3 "Question Bank"): create, edit, archive, tag,
and search questions independent of any quiz. Defines the type-discriminated question model that
F2.2.2/F2.2.3 fill in per type and that must anticipate the PRD's full type list — including the
future fill-in-blank and file-upload types (PRD §4.3) — without building them.

**Needs clarification:** bank scope — are questions private to the authoring faculty member, shared
across the organization, or per-department? **Assumption for planning:** per-faculty ownership with
org-scoped storage, so a later sharing model is additive.

## Dependencies

- Phase 1: tenancy, RBAC (Faculty role), user identity for ownership.
- Feeds F2.2.2, F2.2.3 (type payloads) and F2.3.1 (quiz composition).

## Database Changes (conceptual)

A question concept scoped to an organization and owned by a faculty member, with: a type
discriminator covering all eight PRD §4.3 types (six authorable now; fill-in-blank and file-upload
reserved, non-authorable); the question text/prompt; a per-type payload that each type feature
defines (options, keys, reference answers); free-form tags; active/archived state; and audit
timestamps. Editing semantics must protect quizzes: once a question is used by a published quiz,
its content as-seen-by-that-quiz must remain stable — **Assumption:** the quiz snapshots or pins
question content at publish (owned by F2.3.1); the bank itself stays freely editable.

## Backend Changes

- New `modules/questions/` module (per `rules/folder-structure.md` §4) with standard anatomy:
  services (CRUD, tagging, search/list with filters), repositories, Zod validators per type
  envelope, DTOs, server actions, `permissions.ts`, `index.ts` public entry.
- Search: filter by type, tag, text match, archived state; always paginated (guidelines).

## Frontend Changes

- Question-bank browser (searchable, filterable table/list) and question editor shell in
  `modules/questions/components/`, surfaced under `app/faculty/` (route shells only).
- Type-specific editor bodies are delivered by F2.2.2/F2.2.3.

## Socket Events

None this phase.

## AI Components

None this phase — but the model reserves the per-question reference-answer slot that Phase 7 AI
evaluation reads (PRD §11 step 2); F2.2.3 populates it for subjective types.

## API Endpoints (purpose-level)

- Create / update / archive / restore a question.
- Get a question with its type payload.
- List/search questions (type, tags, text, archived; paginated).
- Manage tags on a question (**Assumption:** free-form tags, normalized per organization).

## Permissions

- Create/update/archive: owning faculty member (Org Admin oversight — **Needs clarification**).
- Read: owner; wider read depends on the bank-scope clarification above.
- All access tenant-scoped from server-side context.

## Validation Rules

- Question text required, bounded length; type must be one of the authorable six.
- Attempts to author reserved future types are rejected with a typed error.
- Type payload validated by the owning type's Zod schema (F2.2.2/F2.2.3) — no untyped blobs accepted.
- Tags: bounded count and length, normalized (trim/case).

## Edge Cases & Failure Scenarios

- Changing a question's type after creation → **Assumption:** disallowed; author archives and recreates.
- Archiving a question referenced by a draft quiz → composition (F2.3.1) must surface it, not break.
- Search with no results / large banks → empty states and pagination, per shared UI standards.
- Concurrent edits by the same owner in two tabs → last-write-wins with updated-at conflict signal
  (**Needs clarification** on desired behavior).

## Acceptance Criteria

- Faculty can create, edit, tag, archive, and search questions in their bank scope.
- The six MVP types are selectable; the two future types are visible in the model but not authorable.
- Search filters by type, tag, and text, and is paginated.
- No faculty member can read or mutate another organization's questions.

## Definition of Done

- Meets `rules/definition-of-done.md`; module anatomy per folder-structure §4; permissions
  registered centrally; docs updated per `CLAUDE.md` (feature log + architecture entry).
