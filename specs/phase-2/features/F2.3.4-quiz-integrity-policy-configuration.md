# F2.3.4 — Quiz integrity policy configuration

**Epic:** E2.3 — Quiz Authoring & Scheduling · **Complexity:** Small

## Description

Per-quiz configuration of the integrity/activity-monitoring policy defined in PRD §13: tab-switch
detection is "a configurable proctoring policy, not a hardcoded behavior — organizations can
enable, disable, or adjust it per quiz", with a warning threshold configurable at the quiz level
(default **3**, PRD §13 "Warning Policy"). **This phase is configuration only** — focus-loss
detection, faculty alerts, auto-termination, and audit timelines are Phase 4 (PRD §13 enforcement
behaviors); Phase 4 becomes a pure consumer of the policy stored here.

## Dependencies

- F2.3.1: the quiz definition carrying the policy.

## Database Changes (conceptual)

An integrity-policy slot on the quiz concept: enabled/disabled flag and a warning threshold
(default 3 per PRD §13). **Assumption:** the shape is kept extensible (a policy object rather than
two loose fields) because PRD §13 frames it as a policy area with future enhancements, but no
additional knobs are added now. **Needs clarification:** PRD §13 says *organizations* can adjust
the policy — is there an org-level default that seeds each quiz's policy, or only per-quiz
settings? Planning position: per-quiz only this phase; org defaults noted as future.

## Backend Changes

- `modules/quizzes/`: policy fields in validators/DTOs/services alongside the quiz definition;
  no enforcement logic anywhere.

## Frontend Changes

- Integrity policy section in the quiz builder settings panel (`modules/quizzes/components/`),
  under `app/faculty/`: enable toggle, threshold input with the default of 3, and copy explaining
  the Phase 4 behavior it will drive (warnings then auto-termination, PRD §13).

## Socket Events

None this phase — the PRD §13 focus-loss reporting over Socket.IO (`window-focus-lost`,
`warning-issued`, `student-auto-submitted` in the event catalog) is Phase 4.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Covered by F2.3.1's quiz update/get endpoints; this feature defines the policy fields they carry.

## Permissions

- Same as quiz editing (F2.3.1): owning faculty member, tenant-scoped. **Needs clarification:**
  may org admins constrain or mandate policy values org-wide (PRD §13 "organizations can…")?

## Validation Rules

- Enabled flag boolean; **Assumption:** default enabled with threshold 3, matching PRD §13's
  "By default" warning ladder — to be confirmed.
- Threshold: positive integer within a sane bound (e.g., 1–10 — bound itself **Needs
  clarification**); ignored/not-required when the policy is disabled.
- Frozen at publish with the rest of the quiz configuration (F2.3.2 lock policy).

## Edge Cases & Failure Scenarios

- Threshold of 1 → first focus loss terminates in Phase 4; allowed but called out in the UI.
- Policy disabled → Phase 4 must record nothing and warn nobody; the stored shape must make
  "disabled" unambiguous rather than "threshold 0".
- Cloned quizzes copy the policy verbatim (F2.3.2).
- Changing the org-level stance (if adopted later) must not retroactively alter published quizzes.

## Acceptance Criteria

- Faculty can enable/disable the integrity policy and set the warning threshold per quiz.
- The default threshold is 3 (PRD §13) when the policy is enabled.
- The stored policy is part of the published quiz definition and frozen at publish.
- No monitoring, alerting, or termination behavior exists in this phase.

## Definition of Done

- Meets `rules/definition-of-done.md`; policy lives in `modules/quizzes/`; docs updated per
  `CLAUDE.md`, noting Phase 4 as the consumer.

## Open Questions
