# F2.3.3 — Randomization & option shuffle

**Epic:** E2.3 — Quiz Authoring & Scheduling · **Complexity:** Medium

## Description

Per-quiz configuration for randomized question order and answer-option shuffling (PRD §4.3
"Randomized Questions", "Shuffle Options"). This phase delivers the **configuration and the
determinism contract only** — no attempt exists yet to shuffle for. The central design constraint:
randomization must never break attempt reproducibility. Phase 3 must be able to reconstruct
exactly what each student saw (question order and option order), both for dispute review
(PRD §4.4 "Raise Result Disputes") and for correct grading of position-independent answers.

## Dependencies

- F2.3.1: the quiz definition carrying this configuration.
- F2.2.2: stable per-option identity (answers reference option ids, never positions).

## Database Changes (conceptual)

Two flags on the quiz concept: randomize-question-order and shuffle-options (independent toggles).
The contract for Phase 3 (documented now, consumed later): each attempt's presented question order
and per-question option order must be **persisted or deterministically derivable** (e.g., a
per-attempt seed recorded at attempt start). This phase reserves that contract; it stores no
per-attempt data. **Assumption:** shuffling applies uniformly to the whole quiz — no per-question
overrides or question-pool sampling (PRD §4.3 names neither).

## Backend Changes

- A documented pure utility seam in `modules/quizzes/utils/` where Phase 3's deterministic

## Validation Rules

- Both flags boolean, default off (**Assumption** — PRD states no default).
- Configuration editable while the quiz is editable; frozen with the rest of the quiz at publish
  (F2.3.2 lock policy).
- Shuffle-options must be ignorable per question type: true/false options are conventionally not
  shuffled (**Assumption** — avoids a reversed True/False display); numerical and subjective types
  have no options to shuffle.

## Edge Cases & Failure Scenarios

- Quiz with one question / question with two options → shuffling is a no-op; must not error.
- Option shuffle combined with options whose order is meaningful (e.g., "All of the above") →
  authoring guidance issue; flagged in UI copy, not enforced (**Needs clarification** whether to
  offer per-question shuffle opt-out for this).
- Phase 3 failing to persist per-attempt order would silently corrupt grading/disputes — the
  contract here must be written as a hard requirement in the Phase 3 spec, not advisory.
- Reproducibility audit: given an attempt record, ordering must be reconstructable indefinitely,
  including after quiz cloning (clones get fresh randomization, no shared seeds).

## Acceptance Criteria

- Faculty can toggle randomized question order and option shuffle independently per quiz.
- The configuration is part of the quiz definition, frozen at publish with everything else.
- The determinism contract (per-attempt order persisted or seed-derivable) is written into this
  spec's expansion and referenced by the Phase 3 planning docs.
- No client-side-only shuffling appears anywhere in the design.

## Definition of Done

- Meets `rules/definition-of-done.md`; configuration lives in `modules/quizzes/`; the Phase 3
  contract is documented in `docs/architecture.md` per `CLAUDE.md`.
