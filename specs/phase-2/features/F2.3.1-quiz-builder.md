# F2.3.1 — Quiz builder

**Epic:** E2.3 — Quiz Authoring & Scheduling · **Complexity:** Extra Large

## Description

The faculty quiz-authoring surface (PRD §4.3 "Create Quiz", "Edit Quiz", "Draft Support"): create a
quiz, keep it in draft, compose it from banked questions (E2.2), order the questions, and assign
per-question marks. The builder produces the complete quiz definition that lifecycle (F2.3.2),
randomization (F2.3.3), integrity policy (F2.3.4), and invitations (F2.3.5) operate on.

## Dependencies

- F2.2.1–F2.2.3: the question bank and all six MVP types.
- F2.1.2: classrooms (**Assumption:** a quiz belongs to a classroom — PRD §5 lists "Quiz History"
  inside classrooms and PRD §8 invites students, who exist as classroom rosters; the PRD never
  states the association explicitly — **Needs clarification**).

## Database Changes (conceptual)

A quiz concept scoped to an organization, owned by a faculty member, associated to a classroom
(pending clarification): title, description/instructions, duration/time limit (PRD §4.3 timers and
"Extend Time" imply one), lifecycle state (F2.3.2), and configuration slots for randomization
(F2.3.3) and integrity policy (F2.3.4). Composition: an ordered list of quiz-question entries, each
referencing a banked question and carrying per-question marks. **Reproducibility rule (shared with
F2.2.1/F2.3.3):** at publish, the quiz's question content — prompt, options, keys, reference
answers — is snapshotted/pinned so later bank edits cannot change what students saw or what Phase 7
evaluates against.

## Backend Changes

- New `modules/quizzes/` module (folder-structure §4 shows its anatomy) — services (quiz CRUD,
  composition add/remove/reorder, marks assignment, completeness checks), repositories, Zod
  validators, DTOs (author view vs. future taker view), server actions, `permissions.ts`.
- Cross-module reads of questions go through `modules/questions/index.ts` only (no deep paths).

## Frontend Changes

- Quiz builder UI in `modules/quizzes/components/`: quiz metadata form, bank-picker (search/filter
  reusing F2.2.1 search), composition list with reorder and per-question marks, total-marks summary,
  completeness indicators. Routes under `app/faculty/` (shells only).

## Socket Events

None this phase.

## AI Components

None this phase — but the publish-time snapshot must include subjective questions' reference
answers (F2.2.3) so Phase 7 AI evaluation reads the exact reference in force when the quiz shipped
(PRD §11 step 2).

## API Endpoints (purpose-level)

- Create / update quiz metadata (draft).
- Add / remove / reorder questions in the composition.
- Set per-question marks.
- Get full quiz definition (author view) and quiz list for the faculty member (paginated).
- Validate completeness (usable pre-publish check; publish itself is F2.3.2).

## Permissions

- Create: Faculty. Update/compose: owning faculty member; **Needs clarification:** can co-faculty
  of the same classroom edit? Org admin read access per Phase 1 oversight rules.
- All operations tenant-scoped server-side.

## Validation Rules

- Title required, bounded; duration positive if set.
- Marks: positive, bounded per question; composition must have ≥1 question to be publishable.
- Only authorable-complete questions (keys present per F2.2.2, reference answers per F2.2.3) count
  toward publishability.
- A question may appear at most once per quiz (**Assumption**).

## Edge Cases & Failure Scenarios

- A banked question is archived after being composed into a draft → surfaced in the builder;
  publish blocked until removed or restored.
- Reorder conflicts from concurrent edits → deterministic ordering with conflict signaling.
- Deleting a classroom association out from under a draft → quiz remains editable, must be
  re-homed before publish (pending the association clarification).
- Very large quizzes → bounded question count (**Needs clarification** on the bound).

## Acceptance Criteria

- Faculty can create a draft quiz, compose it from banked questions of all six MVP types, reorder
  them, and set per-question marks with a visible total.
- Incomplete quizzes are clearly marked non-publishable with actionable reasons.
- Published-quiz content is immune to subsequent bank edits (snapshot verified conceptually).
- Only authorized faculty can see or edit a quiz; tenant isolation holds throughout.
