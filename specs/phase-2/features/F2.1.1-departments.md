# F2.1.1 — Departments

**Epic:** E2.1 — Departments & Classrooms · **Complexity:** Small

## Description

Organization admins can create, rename, and archive departments within their organization
(PRD §4.2 "Create Departments"). Departments are the grouping dimension organization analytics
later reports on ("Department Performance", PRD §15). This feature delivers department CRUD and
listing only.

**Needs clarification:** the PRD never defines the department↔classroom relationship — whether a
classroom belongs to a department (optional? required?), whether faculty belong to departments, or
whether departments are purely an analytics label. This must be resolved before F2.1.2 finalizes
the classroom concept.

## Dependencies

- Phase 1: tenancy, RBAC (Org Admin role), permission catalog.
- Downstream: F2.1.2 may reference departments depending on the clarification above.

## Database Changes (conceptual)

A department concept scoped to an organization, carrying a name, optional description, an
active/archived state, and audit timestamps. Name uniqueness is per organization. No relationship
to classrooms is modeled until the open question above is resolved.

## Backend Changes

- **Assumption:** departments live in `modules/organizations/` (org-structure concern); no
  `departments` module exists in `rules/folder-structure.md` §4, and creating a new module requires
  the reuse check (guidelines §23).
- New service, repository, Zod validators, DTOs, and server actions in
  `modules/organizations/{services,repositories,validators,dto,actions}/`.
- Permission entries registered via `modules/organizations/permissions.ts` into `server/permissions/`.

## Frontend Changes

- Department management pages under `app/organization/` (route shells only).
- Feature UI (department list, create/edit dialog, archive confirmation) in
  `modules/organizations/components/`, reusing shared table/dialog/empty-state primitives from
  `src/components/`.

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List departments for the caller's organization (paginated, filterable by active/archived).
- Create a department.
- Update (rename/describe) a department.
- Archive / restore a department.

## Permissions

- Create/update/archive: Organization Admin only, within their own organization.
- Read: Organization Admin; **Assumption:** faculty may read the department list (needed if
  classrooms reference departments). Super Admin access follows Phase 1 platform rules.

## Validation Rules

- Name: required, trimmed, bounded length, unique within the organization (case-insensitive).
- Organization scope always taken from server-side session context, never from client input.
- Archive is soft — departments are never hard-deleted (analytics may reference them, PRD §15).

## Edge Cases & Failure Scenarios

- Duplicate name (including case/whitespace variants) → rejected with a typed error.
- Archiving a department referenced by classrooms → behavior undecided until the relationship
  question is resolved; spec placeholder: block or warn.
- Cross-tenant access attempt (department id from another organization) → not-found semantics.
- Concurrent rename to the same name → uniqueness enforced at the data layer, surfaced as conflict.

## Acceptance Criteria

- An org admin can create, rename, archive, and restore departments in their organization.
- Department names are unique per organization; violations produce a clear validation error.
- No user can see or mutate another organization's departments.
- Archived departments are excluded from default lists but retrievable via filter.
