# F2.2.2 — Objective question types

**Epic:** E2.2 — Question Bank · **Complexity:** Medium

## Description

Authoring support for the four auto-gradable question types (PRD §4.3, §10): Multiple Choice
(one correct option), Multiple Select (one or more correct options), True/False, and Numerical.
Each type's payload includes the complete answer key at authoring time so Phase 3 auto-grading
(PRD §10 "Automatically graded") is a pure consumer — no grading logic is built in this phase.

## Dependencies

- F2.2.1: the question model, editor shell, and type-payload validation seam.

## Database Changes (conceptual)

Per-type payloads on the F2.2.1 question concept:

- **MCQ / Multiple Select:** an ordered list of options (text; bounded count), with exactly one
  (MCQ) or at least one (multiple select) marked correct. Option identity must be stable
  (per-option identifiers) so option shuffle (F2.3.3) and Phase 3 answers reference options, not
  display positions.
- **True/False:** the correct boolean.
- **Numerical:** the correct numeric value; **Assumption:** an optional acceptance tolerance
  (exact-match otherwise), since the PRD is silent — **Needs clarification.**

## Backend Changes

- `modules/questions/validators/`: Zod schemas per objective type (option counts, correct-answer
  presence, numeric constraints).
- `modules/questions/services/`: type-aware create/update paths; answer keys never leave the
  module unredacted (see Permissions).
- DTOs split "author view" (includes keys) from any future "taker view" (excludes keys) so Phase 3
  cannot accidentally leak answers.

## Frontend Changes

- Type-specific editor bodies in `modules/questions/components/`: option list editor with
  correct-answer marking (MCQ/multi-select), boolean picker (true/false), numeric answer +
  tolerance input (numerical). Surfaced through the F2.2.1 editor under `app/faculty/`.

## Socket Events

None this phase.

## AI Components

None this phase — objective grading is deterministic (PRD §10); AI evaluation targets subjective
types only (PRD §11).

## API Endpoints (purpose-level)

- Covered by F2.2.1's create/update/get endpoints; this feature defines the objective payloads
  those endpoints accept and return.

## Permissions

- Same as F2.2.1 (owner-scoped authoring). Answer keys are visible only to users with authoring
  permission on the question — never in any student-facing DTO.

## Validation Rules

- MCQ: ≥2 options, exactly one correct.
- Multiple Select: ≥2 options, ≥1 correct; **Needs clarification:** may *all* options be correct?
- Option text: required, bounded length; duplicate option text rejected within a question.
- True/False: correct value required; options are fixed, not editable.
- Numerical: correct value required, finite; tolerance (if adopted) non-negative.

## Edge Cases & Failure Scenarios

- Author unmarks every correct option → save rejected with a field-level error.
- Reordering options must not change which options are correct (identity, not position).
- Numerical values with high precision or scientific notation → define accepted numeric formats.
- Editing an option on a question used in a published quiz → governed by the F2.3.1 snapshot rule.

## Acceptance Criteria

- Faculty can author and edit all four objective types with complete answer keys.
- A question missing its key cannot be saved as authorable-complete.
- Answer keys never appear in non-authoring DTOs.
- Option identity survives reordering, satisfying the F2.3.3 shuffle contract.

## Definition of Done

- Meets `rules/definition-of-done.md`; Zod validation per type in `modules/questions/validators/`;
  docs updated per `CLAUDE.md`.

## Open Questions

- Numerical tolerance/units semantics (PRD silent).
- Multiple-select partial-credit intent — grading is Phase 3, but does the key need weighting data now?
- Maximum option count per question.
