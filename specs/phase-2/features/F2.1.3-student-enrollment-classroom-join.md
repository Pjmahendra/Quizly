# F2.1.3 — Student enrollment & classroom join

**Epic:** E2.1 — Departments & Classrooms · **Complexity:** Medium

## Description

Students become members of classrooms two ways: (a) enrollment by an org admin or the classroom's
faculty (PRD §4.2 "Manage Students"), and (b) self-service "Join Classroom" from the student portal
(PRD §4.4). The roster produced here is what quiz invitations (F2.3.5) and every later classroom
feature (announcements, resources — PRD §5) operate on.

**Needs clarification:** the PRD names "Join Classroom" but never specifies the mechanism — join
code? emailed invite link? open directory with approval? For planning, **Assumption:** a
per-classroom join code that faculty can view/regenerate, with an optional faculty-approval step —
to be validated before implementation.

## Dependencies

- Phase 1: tenancy, RBAC (Student, Faculty, Org Admin), user management.
- F2.1.2: classrooms must exist.

## Database Changes (conceptual)

An enrollment concept linking a student to a classroom within one organization, carrying status
(e.g., active, pending-approval if approval is adopted, removed), how it originated (admin/faculty
enrollment vs. student join), who performed it, and timestamps. One active enrollment per
student-classroom pair. If the join-code assumption holds, the classroom concept gains a
regenerable join code.

## Backend Changes

- Extends `modules/classrooms/`: enrollment services (enroll, remove, join-by-code, approve/reject
  if applicable, roster listing), repository additions, Zod validators, DTOs, server actions.
- Permission entries via `modules/classrooms/permissions.ts`.

## Frontend Changes

- Faculty/org-admin roster management (add students, remove, view roster, join-code display) in
  `modules/classrooms/components/`, surfaced under `app/faculty/`, `app/organization/`, `app/classroom/`.
- Student "Join Classroom" entry point and "my classrooms" list under `app/student/` (shells only).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Enroll student(s) into a classroom (faculty/org admin; supports selecting existing org students).
- Remove a student from a classroom.
- Student joins a classroom (via the clarified mechanism).
- Approve/reject a pending join (only if the approval step is confirmed).
- List classroom roster (paginated); list a student's own classrooms.

## Permissions

- Enroll/remove/approve: classroom's assigned faculty and Org Admin.
- Join: Student role, same organization as the classroom only.
- Roster read: faculty/org admin; students see membership of their own classrooms only per
  organization privacy policy (PRD §4.4 peer visibility caveat) — **Needs clarification.**

## Validation Rules

- Student and classroom must share the same organization (tenant rule; server-side context).
- No duplicate active enrollment; re-enrollment after removal creates a fresh active record.
- Only users with the Student role can be enrolled as students.
- Join code (if adopted): validated server-side, rate-limited (guidelines: rate limiting, PRD §16).

## Edge Cases & Failure Scenarios

- Join attempt on an archived classroom or with a stale/regenerated code → clear failure.
- Bulk enrollment where some students are invalid → partial-failure reporting, no silent drops.
- Student deactivated in Phase 1 → enrollment retained but inert (excluded from invitations).
- Removed student rejoining via code → allowed or blocked (**Needs clarification**).

## Acceptance Criteria

- Faculty/org admin can enroll and remove students; the roster reflects changes immediately.
- A student can join a classroom via the clarified mechanism and then sees it in their portal.
- No student can join or view a classroom in another organization.
- Duplicate enrollments are impossible; removal is auditable.
