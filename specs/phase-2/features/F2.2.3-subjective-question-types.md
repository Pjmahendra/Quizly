# F2.2.3 — Subjective question types

**Epic:** E2.2 — Question Bank · **Complexity:** Medium

## Description

Authoring support for the two subjective MVP types (PRD §4.3): Short Answer and Long Descriptive
Answer. These are manually evaluated in the MVP (PRD §10 "Manual Evaluation") and AI-evaluated in
Phase 7 (PRD §11). The critical deliverable is capturing the **expected/reference answer per
question at authoring time** — PRD §11 step 2 ("Faculty provides the expected answer(s) for each
question") makes this the data precondition for AI evaluation; skipping it now blocks Phase 7.

## Dependencies

- F2.2.1: question model, editor shell, type-payload validation seam.

## Database Changes (conceptual)

Per-type payloads on the F2.2.1 question concept: for both short-answer and long-descriptive
questions, the prompt plus one or more faculty-written expected/reference answers (PRD §11 says
"answer(s)" — plural supported). **Assumption:** an optional expected-length hint (e.g., short vs.
extended response) distinguishes the two types for the student UI; no grading configuration
(fuzzy matching, rubrics, keyword weights — PRD §11) is modeled now, but the reference-answer slot
must not preclude attaching that configuration in Phase 7.

## Backend Changes

- `modules/questions/validators/`: Zod schemas for short-answer and long-descriptive payloads,
  including at least one reference answer.
- `modules/questions/services/`: type-aware create/update; reference answers treated with the same
  secrecy as objective answer keys (author-only DTOs).

## Frontend Changes

- Editor bodies in `modules/questions/components/`: prompt editor plus a reference-answer editor
  supporting multiple reference answers, with clear labeling that these feed later AI evaluation.
  Surfaced through the F2.2.1 editor under `app/faculty/`.

## Socket Events

None this phase.

## AI Components

None this phase — **but this feature is Phase 7's feedstock:** the reference answers captured here
are exactly what the AI evaluator compares student responses against (PRD §11 steps 2 and 7).
Store them as first-class, retrievable-per-question data so `modules/evaluations/` and
`modules/ai/` (folder-structure §11) can consume them without migration.

## API Endpoints (purpose-level)

- Covered by F2.2.1's create/update/get endpoints; this feature defines the subjective payloads,
  including reference answers, that those endpoints accept and return.

## Permissions

- Same as F2.2.1 (owner-scoped authoring). Reference answers are author-visible only; they must
  never appear in any student-facing DTO in any phase.

## Validation Rules

- Prompt: required, bounded length.
- At least one reference answer required to mark the question authorable-complete;
  **Needs clarification:** whether a subjective question may be saved (draft) without one — planning
  position: draft yes, usable-in-published-quiz no.
- Reference answers: bounded count and length; non-empty after trimming.

## Edge Cases & Failure Scenarios

- Faculty saves a subjective question without a reference answer, then a quiz using it is published
  → publish-time validation in F2.3.2 must catch it (the Phase 7 blocking risk).
- Very long reference answers (essay-length) → generous but bounded limits; no truncation on read.
- Multiple reference answers with contradictory content → allowed (faculty judgment); flagged in UI copy.
- Editing reference answers after a quiz is published → governed by the F2.3.1 snapshot rule; the
  snapshot must include reference answers so Phase 7 evaluates against what was current at publish.

## Acceptance Criteria

- Faculty can author short-answer and long-descriptive questions with one or more reference answers.
- A question without a reference answer cannot be part of a published quiz (enforced with F2.3.2).
- Reference answers are stored per question, retrievable by authorized backend consumers, and
  absent from all student-facing shapes.
