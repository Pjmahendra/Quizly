# F2.3.5 — Student quiz invitations

**Epic:** E2.3 — Quiz Authoring & Scheduling · **Complexity:** Medium

## Description

Faculty invite students to a published/scheduled quiz — the "Invite Students" step of the quiz
workflow (PRD §8), sitting between Schedule and Students Join. This phase persists who is invited
and shows students their upcoming quizzes; actually joining and taking the quiz is Phase 3.

**Needs clarification:** the PRD never says whether a quiz is automatically visible to the whole
classroom roster or requires explicit invitation. PRD §8 names an explicit invite step, while
PRD §5 says enrolled students "automatically receive updates". Planning position (**Assumption**):
default classroom-wide invitation (invite-all from the roster) with optional per-student selection —
to be validated before implementation.

## Dependencies

- F2.3.2: a quiz must be scheduled/published before invitations make sense (**Assumption:**
  inviting from a draft is rejected).
- F2.1.3: the classroom roster is the invitation source.

## Database Changes (conceptual)

An invitation concept linking a quiz to a student within one organization: status (invited,
revoked; Phase 3 adds joined/attempted), who invited, and timestamps. One active invitation per
quiz-student pair. If the classroom-wide assumption holds, an invite-all action materializes
per-student records (**Needs clarification:** materialized rows vs. a roster-wide flag — planning
position: materialized rows, so later-enrolled students are an explicit decision).

## Backend Changes

- `modules/quizzes/`: invitation services (invite selected, invite roster, revoke, list), reading
  rosters via `modules/classrooms/index.ts` only; repositories, Zod validators, DTOs, actions;
  permission entries via `modules/quizzes/permissions.ts`.

## Frontend Changes

- Invitation management panel on the quiz (roster picker, invite-all, revoke, status list) in
  `modules/quizzes/components/`, under `app/faculty/`.
- Student "upcoming quizzes" list (invited, not yet startable) under `app/student/` (shell only).

## Socket Events

None this phase.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Invite selected students / invite the full classroom roster to a quiz.
- Revoke an invitation.
- List a quiz's invitations with status (faculty; paginated).
- List the calling student's quiz invitations (their upcoming quizzes).

## Permissions

- Invite/revoke/list-invitations: the quiz's owning faculty member (co-faculty question inherited
  from F2.3.1); org admin read per Phase 1 oversight.
- Students read only their own invitations. All operations tenant-scoped server-side.

## Validation Rules

- Only students enrolled (active) in the quiz's classroom may be invited (tenant + roster check).
- No duplicate active invitations; re-inviting after revoke reactivates or recreates cleanly.
- Invitations only against quizzes in an invitable state (scheduled/published — per F2.3.2 semantics).

## Edge Cases & Failure Scenarios

- Student enrolled after invite-all → not auto-invited under the materialized-rows position;
  faculty sees a "not yet invited" indicator (**Needs clarification** whether auto-invite is wanted).
- Student removed from the classroom after being invited → invitation auto-revoked or flagged
  (**Needs clarification**).
- Quiz rescheduled after invitations sent → invitations persist; notification of the change is the
  notifications phase's concern (PRD §14 "Quiz Scheduled").
- Bulk invite with partial failures (deactivated students) → per-row outcome report, no silent drops.
- Email/notification delivery (PRD §14) is explicitly out of scope — invitations are visible
  in-portal only this phase (**Assumption**).

## Acceptance Criteria

- Faculty can invite individual students or the whole roster to a scheduled/published quiz, and revoke.
- Students see their invited quizzes in their portal; nothing is startable (Phase 3).
- Only enrolled, same-organization students can ever hold an invitation to a quiz.
- Invitation state changes are auditable.

## Definition of Done

- Meets `rules/definition-of-done.md`; cross-module access via `index.ts` seams only; permissions
  registered centrally; docs updated per `CLAUDE.md`.

## Open Questions

- Classroom-wide visibility vs. explicit invitation (blocking — see Description).
- Auto-invite behavior for late enrollees and auto-revoke for removals.
- Whether invitation implies anything about attempts allowed (Phase 3 boundary).
