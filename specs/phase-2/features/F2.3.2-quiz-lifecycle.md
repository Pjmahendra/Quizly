# F2.3.2 — Quiz lifecycle

**Epic:** E2.3 — Quiz Authoring & Scheduling · **Complexity:** Large

## Description

The state machine around a quiz (PRD §4.3 "Schedule Quiz", "Publish Quiz", "Clone Quiz",
"Delete Quiz"; PRD §8 Create → Schedule → Invite): scheduling a quiz for a time window, publishing
it (making it real for students), cloning an existing quiz into a fresh draft, and deleting.
Later-phase states (live, paused, ended — PRD §4.3 live controls) are Phase 4; this phase defines
the authoring-side states and transitions only.

**Needs clarification:** timezone semantics for scheduling — the PRD is silent. Are times entered
in the faculty member's timezone, an organization timezone, or UTC-with-display-conversion?
Planning position (**Assumption**): store UTC instants, display in the viewer's local timezone.

## Dependencies

- F2.3.1: a complete quiz definition to schedule/publish; publish runs F2.3.1's completeness check.
- F2.2.3: publish-time validation that subjective questions carry reference answers.

## Database Changes (conceptual)

Lifecycle state on the quiz concept — conceptually: Draft → Scheduled → Published (terminal for
this phase; Phase 4 adds live states). Scheduling data: start time, and **Needs clarification:**
whether an end time / join window is required now or is Phase 3 concern. Deletion is soft for
anything past Draft (audit trail, PRD §16 Audit Logs); **Assumption:** hard delete allowed for
never-published drafts. Clone produces a new Draft copying composition, marks, randomization, and
integrity policy — never attempts data.

**Needs clarification:** PRD §4.3 lists Schedule and Publish as distinct while §8 orders
Schedule before Invite; the exact meaning of "published but not yet scheduled" (or vice versa)
must be pinned down — planning position: schedule sets the window, publish makes it visible/binding.

## Backend Changes

- `modules/quizzes/services/`: explicit transition service (single authority for legal
  transitions), schedule/publish/clone/delete operations, publish-time validation pipeline.
- Repository additions; Zod validators for scheduling input; audit-log entries per transition
  (Phase 1 audit facility, PRD §16).

## Frontend Changes

- Lifecycle controls (schedule dialog with timezone-explicit picker, publish confirmation with
  validation results, clone, delete) in `modules/quizzes/components/`, under `app/faculty/`.
- Student-visible "upcoming quizzes" listing is F2.3.5's concern.

## Socket Events

None this phase.

## AI Components

None this phase — publish-time validation enforces the reference-answer precondition that Phase 7
depends on (PRD §11 step 2).

## API Endpoints (purpose-level)

- Schedule (set/adjust the time window) a quiz.
- Publish a quiz (runs completeness validation; rejects with reasons).
- Clone a quiz into a new draft.
- Delete a quiz (soft/hard per state).
- List quizzes by state for the faculty member (paginated).

## Permissions

- All transitions: owning faculty member (same open co-faculty question as F2.3.1); org admin
  visibility per Phase 1 oversight. Tenant-scoped throughout.

## Validation Rules

- Schedule start must be in the future at the time of scheduling (with a small clock-skew grace).
- Publish requires: ≥1 question, all questions authorable-complete (keys/reference answers),
  marks assigned, schedule present (**Assumption** — publish without schedule rejected).
- Illegal transitions (e.g., delete a published quiz with invitations sent) rejected with typed errors.

## Edge Cases & Failure Scenarios

- Rescheduling a published quiz → allowed while not yet started (**Assumption**), invalidating
  nothing; Phase 3+ constraints noted for the future.
- Editing a published quiz → **Needs clarification:** locked vs. unpublish-edit-republish;
  planning position: content locked after publish (protects the F2.3.1 snapshot).
- Clone of a quiz whose source questions were since archived → clone succeeds from the snapshot.
- DST boundaries and cross-timezone faculty/students → resolved by the UTC storage assumption.

## Acceptance Criteria

- A complete quiz can be scheduled and published; an incomplete one is rejected with reasons.
- Cloning yields an independent draft preserving composition, marks, and configuration.
- Deletion behaves per state (hard for pure drafts, soft otherwise) and is audited.
- All transitions are permission-gated, tenant-isolated, and individually audited.

## Definition of Done

- Meets `rules/definition-of-done.md`; transition logic lives only in `modules/quizzes/services/`;
  docs updated per `CLAUDE.md`.
