# E2.3 — Quiz Authoring & Scheduling

## Purpose

Let faculty assemble complete quizzes from the question bank and carry them through the authoring
half of the quiz workflow: Create → Schedule → Invite (PRD §8), with draft support, cloning, and
deletion (PRD §4.3), randomization and option shuffle (PRD §4.3), and per-quiz integrity policy
configuration (PRD §13 — configuration only; enforcement is Phase 4). The epic's output is a
published, scheduled quiz with invited students — the exact input Phase 3 (quiz taking) consumes.

## Business Value

- Completes the faculty authoring loop: everything a quiz needs before delivery exists in one place.
- Draft + clone support (PRD §4.3) makes iteration and reuse cheap for faculty.
- Integrity policy captured per quiz now (PRD §13) means Phase 4 enforcement is a pure consumer of
  existing configuration.
- The randomization reproducibility contract (F2.3.3) protects Phase 3 from a class of unfixable
  "what did the student actually see?" disputes.

## Dependencies

- **Phase 1:** tenancy, RBAC, users.
- **E2.1:** classrooms and enrollment — quizzes need an audience and invitations need a roster.
- **E2.2:** the question bank — quizzes are composed from banked questions (F2.2.1–F2.2.3).
- Internal ordering: F2.3.1 (builder) first; F2.3.2–F2.3.5 layer lifecycle, randomization, policy,
  and invitations onto it and can proceed in parallel after F2.3.1.

## Acceptance Criteria

- Faculty can create a quiz, keep it in draft (PRD §4.3 "Draft Support"), compose it from banked
  questions of all MVP types, and set per-question marks.
- Faculty can schedule, publish, clone, and delete quizzes (PRD §4.3, §8); state transitions are
  explicit and invalid transitions are rejected.
- Faculty can enable randomized question order and option shuffle per quiz (PRD §4.3); the
  configuration defines a deterministic, per-attempt-reconstructable contract for Phase 3.
- Faculty can configure the integrity policy per quiz — enable/disable and warning threshold,
  default 3 (PRD §13) — with no enforcement behavior in this phase.
- Faculty can invite enrolled students to a quiz (PRD §8), pending the visibility model resolved
  in F2.3.5's **Needs clarification**.
- Domain code lives in `modules/quizzes/` per `rules/folder-structure.md` §4; a published quiz's
  content is frozen so attempts remain reproducible.
