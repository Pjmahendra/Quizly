# E6.2 — Organization Analytics & Reports

## Purpose

Deliver the Organization Admin's analytics responsibilities: institutional dashboards covering
Department Performance, Faculty Activity, Student Engagement, and Assessment Statistics (PRD §15
Organization; §4.2 "View Institutional Analytics"), plus report generation for org admins
(PRD §4.2 "Generate Reports") and report downloads for students (PRD §4.4 "Download Reports").
All org-wide aggregates are precomputed rollups scoped to the single tenant — never live
organization-wide scans on the request path (guidelines §2, §10).

## Business Value

Institutions buy Quisly at the organization level (PRD §1, §3); the org admin's dashboards and
reports are how the institution sees return on that investment — which departments perform, which
faculty are active, whether students engage. Downloadable reports give students and administrators
a portable record of outcomes, a standard expectation for educational institutions.

## Dependencies

- **Phase 3** results data (the substrate). The epic can start once Phase 3 lands and run in
  parallel with Phases 4–5.
- **Phases 4–5 enrich**: engagement/activity signals (Phase 4) deepen the Student Engagement and
  Faculty Activity dashboards; the Phase 5 storage platform is required for persisting generated
  report files (folder-structure §10) — F6.2.2's download path sequences after it.
- Departments and classrooms (Phase 2 per PRD §4.2/§18 Phase 1 scope) for the Department
  Performance dimension.
- BullMQ/Redis (PRD §17) for background report generation — subject to the serverless constraint
  flagged in Phase 4.
- E6.1's snapshot pipeline in `modules/analytics/` — org rollups build on the same conceptual
  mechanism, not a second one (guidelines §23 reuse check).

## Acceptance Criteria

- All four PRD §15 Organization metrics render in the Org Admin portal, scoped to the admin's own
  organization from trusted server context (guidelines §1.1) — cross-tenant reads are impossible.
- Dashboards read tenant-scoped precomputed rollups with visible freshness; no unbounded
  org-wide aggregate executes on the request path (guidelines §2, §10).
- An org admin can request a report, see its progress, and download the finished file; a student
  can download their own reports (PRD §4.2, §4.4).
- Report generation runs as background work and never blocks or times out a user-facing request
  (guidelines §10).
- Report formats/contents are clarified and recorded before F6.2.2 implementation (see its Open
  Questions); both features pass `rules/definition-of-done.md`.
