# E6.3 — Platform Analytics & Ops Surfaces

## Purpose

Give the Super Admin the platform-level views named in PRD §4.1 and §15 Platform: platform
analytics (Active Users, Active Institutions, Concurrent Quizzes), a System Health surface
("System Monitoring", PRD §4.1), and platform Feature Toggles (PRD §4.1). These are product
surfaces in the Super Admin portal — no infrastructure or deployment planning belongs here.
Platform metrics are inherently cross-tenant, so they are served exclusively from scheduled
platform-level snapshots computed off the request path (guidelines §2, §10) and gated behind
super-admin-only capabilities (guidelines §9).

## Business Value

The platform operator needs to know whether the product is alive and growing — how many users and
institutions are active, how much concurrent quiz load exists, and whether the system is healthy —
without querying the database by hand. Feature toggles give the operator a controlled way to roll
platform capabilities out or off (guidelines §22), reducing the blast radius of risky releases.

## Dependencies

- **Phase 3** results/attempt data for activity-derived metrics; Phase 2/4 live-quiz session data
  (Socket.IO presence, PRD §7, §12) is the natural source for Concurrent Quizzes — the metric
  definition must not require new tracking beyond what those phases record.
- Phase 1 RBAC: super-admin capabilities from the central catalog (guidelines §9); Phase 1 audit
  platform for logging toggle changes (PRD §4.1 Audit Logs; guidelines §8.8).
- The `modules/analytics/` snapshot pipeline (E6.1/E6.2) extended with platform-level rollups;
  monitoring signals for F6.3.2 depend on the Sentry/OpenTelemetry recommendation in PRD §17
  being adopted (**Needs clarification** — tracked in F6.3.2).
- Can start once Phase 3 lands; runs in parallel with Phases 4–5.

## Acceptance Criteria

- The Super Admin portal shows Active Users, Active Institutions, and Concurrent Quizzes
  (PRD §15 Platform) plus a System Health view (PRD §4.1, §15) — visible to super admins only.
- Platform metrics are read from precomputed platform snapshots; no live cross-tenant aggregate
  executes on the request path (guidelines §2, §10).
- Feature toggles can be viewed and changed by super admins; every change is audit-logged
  (guidelines §8.8) and takes effect without a redeploy.
- No feature in this epic exposes tenant-identifiable drill-downs beyond what the Super Admin's
  existing organization-management views already show (**Assumption** — see F6.3.1).
- All three features pass `rules/definition-of-done.md`.

> **Scheduling note:** F6.3.3 may be pulled earlier than the rest of Phase 6 if a risky rollout in
> a parallel phase needs flagging (guidelines §22).
