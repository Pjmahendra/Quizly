# F6.3.1 — Platform Analytics

**Epic:** E6.3 — Platform Analytics & Ops Surfaces · **Complexity:** Medium

## Description

The Super Admin's platform analytics dashboard (PRD §4.1 "Platform Analytics"): Active Users,
Active Institutions, and Concurrent Quizzes (PRD §15 Platform). These metrics are inherently
cross-tenant — exactly what guidelines §2/§10 forbid on the request path — so the dashboard reads
only scheduled platform-level snapshots computed in the background, gated behind a
super-admin-only capability. A read-only product surface in the Super Admin portal.

## Dependencies

- Phase 3 attempt/result activity for usage-derived metrics; Phase 2/4 live-quiz session presence
  (PRD §7, §12) as the source for Concurrent Quizzes — no new tracking is added here, only
  aggregation of what those phases already record.
- `modules/analytics/` snapshot pipeline (E6.1/E6.2) extended with platform-level rollups
  (guidelines §23); the background mechanism resolved in Phase 4.
- Phase 1 super-admin capabilities (guidelines §9).

## Database Changes (conceptual)

- A platform metrics snapshot concept: periodic platform-scoped counters (active users, active
  institutions, concurrent quiz sessions) with capture time, retained as a short history for
  recent-movement display. Prose only — no schema here.

## Backend Changes

- `modules/analytics/services/`: platform-analytics service reading snapshots; the snapshot
  producer runs on a schedule off the request path, aggregating across tenants only inside that
  background context (guidelines §10).
- `modules/analytics/repositories/` as the sole Prisma seam (folder-structure §8); Redis presence
  data may feed the concurrent-quizzes counter — read via the `server/cache/` adapter with
  catalog keys (folder-structure §9).

## Frontend Changes

- Super Admin route under `app/admin/` (thin shell) composing `modules/analytics/components/`:
  headline tiles for the three metrics, recent-trend charts from `src/components/charts/`
  (folder-structure §6), snapshot-time stamp.

## Socket Events

None — snapshot cadence suffices for an operator overview. **Assumption:** no live-updating
counter is required by the PRD.

## AI Components

None.

## API Endpoints (purpose-level)

- Fetch the current platform metrics snapshot (three headline figures plus capture time).
- Fetch recent snapshot history for trend display, bounded (guidelines §5).

## Permissions

- Super-admin-only capability to view platform analytics, registered via
  `modules/analytics/permissions.ts` into the central catalog (folder-structure §4, §9); no org
  role can hold it.

## Validation Rules

- History-range parameters bounded and validated; no tenant-identifying parameters accepted —
  the surface is platform-scoped by construction.

## Edge Cases & Failure Scenarios

- Snapshot job hasn't run yet (fresh deployment) → explicit "no snapshot yet" state.
- Snapshot producer failing silently → capture age always shown; stale beyond a threshold is
  flagged rather than presented as current.
- "Active" definitions drifting between metrics → definitions recorded once in the module and
  shown in the UI (see Open Questions).

## Acceptance Criteria

- Super Admin sees Active Users, Active Institutions, and Concurrent Quizzes (PRD §15 Platform)
  with the snapshot capture time visible.
- No live cross-tenant aggregate on the request path — reads hit only snapshot data
  (guidelines §2, §10).
- The surface is invisible and inaccessible to every non-super-admin role (guidelines §9).

## Definition of Done

- Passes `rules/definition-of-done.md`; documentation per `CLAUDE.md` (feature log entry;
  `docs/architecture.md` for the platform snapshot producer).

## Open Questions

- Per-organization drill-down here, or do existing org-management views cover it?
  **Assumption:** headline platform figures only, no new tenant drill-downs.
