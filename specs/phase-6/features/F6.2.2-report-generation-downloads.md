# F6.2.2 — Report Generation & Downloads

**Epic:** E6.2 — Organization Analytics & Reports · **Complexity:** Large

## Description

Report production and retrieval for two audiences: org admins "Generate Reports" (PRD §4.2) and
students "Download Reports" (PRD §4.4). An org admin requests a report, generation runs as
background work (heavy — guidelines §10; BullMQ in the stack, PRD §17), and the finished file is
stored and downloadable. Students download reports about their own results.
**Needs clarification:** report formats and exact contents are unspecified in the PRD.
**Assumption:** one downloadable file per report request; format TBD.

## Dependencies

- Phase 3 results data; F6.2.1's org rollups as report inputs where applicable.
- **Phase 5 storage platform** — files persist via `modules/storage/` with paths from
  `modules/storage/paths.ts` (folder-structure §9–§10); file persistence sequences after Phase 5
  even though the rest of Phase 6 runs in parallel.
- Background processing (BullMQ/Redis, PRD §17) — **subject to the serverless constraint flagged
  in Phase 4**; this spec assumes that resolution, conceptually.
- Phase 1 capabilities/tenancy and audit platform (generation/download logging).

## Database Changes (conceptual)

- A report-request record concept: requester, report type and parameters, lifecycle status
  (queued → generating → ready/failed), reference to the stored file, org scope, retention
  metadata. Prose only — no schema here.

## Backend Changes

- `modules/analytics/services/`: report service — validates the request, enqueues generation,
  tracks lifecycle, exposes status and download authorization; student reads limited to the
  requesting student's own data. File writes go through `modules/storage/` — never a hand-built
  blob path (folder-structure §9).
- **Assumption:** the blob layout gains a reports area under the org hierarchy, added to
  `modules/storage/paths.ts` as the single path authority (folder-structure §10).

## Frontend Changes

- Org Admin portal (`app/organization/`): report request form, report list with status and
  download action (shared table/pagination, guidelines §13).
- Student portal (`app/student/`): "Download Reports" surface listing the student's own reports
  (PRD §4.4). Both compose `modules/analytics/components/`; progress and failure states explicit.

## Socket Events

None — readiness shown via status polling/refetch. **Assumption:** an existing notification flow
may announce readiness; no new socket event is defined (catalog is the single source,
folder-structure §5).

## AI Components

None.

## API Endpoints (purpose-level)

- Request generation of a report (org admin; type + parameters).
- List report requests with status, paginated (guidelines §5).
- Download a finished report file (org admin; students for their own reports).

## Permissions

- Org-admin capability to generate/download organization reports; student capability to download
  their own reports — registered via `modules/analytics/permissions.ts` (folder-structure §4,
  §9). Download authorization re-checked at fetch time, never by URL secrecy.

## Validation Rules

- Report type/parameters validated (module Zod validators); parameter scope (departments,
  classrooms, periods) must belong to the caller's org from server context (guidelines §1.1);
  periods bounded to keep generation work finite.

## Edge Cases & Failure Scenarios

- Generation fails mid-job → status "failed", typed error, retry path (PRD §16); no orphaned
  half-written files.
- Report requested over data still being graded → content policy needed (see Open Questions).
- Duplicate rapid requests for the same report → idempotency/debounce (guidelines §4.5).
- Stored file missing at download → honest error, request flagged, never a silent broken download.

## Acceptance Criteria

- Org admin can request, monitor, and download a report; student can download their own reports
  (PRD §4.2, §4.4), tenant and self-scope enforced server-side (guidelines §1.1).
- Generation runs off the request path; the requesting UI returns promptly (guidelines §10).
- Files stored/retrieved only through the storage module's path catalog (folder-structure
  §9–§10); downloads permission-checked per fetch.
