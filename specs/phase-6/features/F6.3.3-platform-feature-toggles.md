# F6.3.3 — Platform Feature Toggles

**Epic:** E6.3 — Platform Analytics & Ops Surfaces · **Complexity:** Medium

## Description

Super Admin feature toggles (PRD §4.1 "Feature Toggles"): a managed catalog of platform feature
flags the Super Admin can view and switch without a redeploy, so risky capabilities roll out
gradually and turn off fast (guidelines §22). Every change is audit-logged. Scope is the platform
level named in the PRD; per-organization overrides are not invented here (see Open Questions).
**Scheduling note:** this feature may be pulled earlier than the rest of Phase 6 if a risky
rollout in a parallel phase needs flagging (guidelines §22).

## Dependencies

- Phase 1 super-admin capabilities and the audit platform (PRD §4.1 Audit Logs; guidelines §8.8).
- A home module: no existing module owns platform configuration. **Assumption:** a small new
  `modules/feature-flags/` module with the standard anatomy, justified via the reuse check
  (folder-structure §14; guidelines §23) — analytics does not own this concern.
- `server/cache/` for fast flag-evaluation reads (catalog keys, folder-structure §9).

## Database Changes (conceptual)

- A feature-flag record concept: flag key from a code-defined catalog, enabled state, description,
  and change metadata (who, when). Prose only — no schema here.

## Backend Changes

- `modules/feature-flags/services/`: flag catalog defined in code (unknown keys cannot be invented
  at runtime), an evaluation helper other modules call through the module's `index.ts`
  (folder-structure §4), and admin mutations that write audit entries (guidelines §8.8).
- Evaluation is read-heavy and cheap: cached via `server/cache/` with explicit invalidation on
  change — never a per-request table scan (guidelines §10). Repository as the sole Prisma seam
  (folder-structure §8).

## Frontend Changes

- Super Admin route under `app/admin/` (thin shell) composing `modules/feature-flags/components/`:
  flag list with state, description, last-changed metadata, a confirm-on-change control, and the
  flag's audit trail visible where the flag is viewed (guidelines §8.8).

## Socket Events

None. **Assumption:** changes propagate via cache invalidation and next-read pickup; no live push
required.

## AI Components

None.

## API Endpoints (purpose-level)

- List feature flags with current state and metadata (super admin).
- Change a flag's state (super admin; audited).

## Permissions

- Super-admin-only capability to view and manage feature toggles, registered via
  `modules/feature-flags/permissions.ts` into the central catalog (folder-structure §4, §9).

## Validation Rules

- Flag key must exist in the code-defined catalog — arbitrary keys rejected; state changes
  validated (module Zod validators) and idempotent (guidelines §4.5).

## Edge Cases & Failure Scenarios

- Flag referenced in code but missing from the store → safe default (off) plus a surfaced
  warning, never a crash.
- Cache invalidation failure after a change → bounded staleness via expiry; the admin UI shows
  the authoritative stored state (guidelines §10).
- Stale flags accumulating → each flag carries an owner/intent description; retiring a settled
  flag is part of its rollout's completion (guidelines §22 — an unread flag is dead code).

## Acceptance Criteria

- Super Admin can view all platform flags and switch them; changes take effect without a redeploy
  and are audit-logged with actor and time (guidelines §8.8, §22).
- Flag evaluation adds no meaningful request-path cost (cached, explicit invalidation —
  guidelines §10).
- No non-super-admin role can see or change toggles (guidelines §9).

## Definition of Done

- Passes `rules/definition-of-done.md`; documentation per `CLAUDE.md` (feature log entry;
  `docs/architecture.md` for the new module and its evaluation seam).

## Open Questions

- **Needs clarification:** which features are toggleable at launch? PRD §4.1 names the capability
  without a flag list.
- Platform-wide only, or per-organization overrides? **Assumption:** platform-wide only — per-org
  toggling is not in the PRD.
- Does a flipped-off feature hide UI, block the server action, or both? (Both, presumably —
  mirrored gates, guidelines §9 — but confirm.)
