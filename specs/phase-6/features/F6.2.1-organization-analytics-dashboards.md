# F6.2.1 — Organization Analytics Dashboards

**Epic:** E6.2 — Organization Analytics & Reports · **Complexity:** Large

## Description

The Org Admin portal's institutional dashboards (PRD §4.2 "View Institutional Analytics")
covering the four PRD §15 Organization metrics: Department Performance, Faculty Activity, Student
Engagement, and Assessment Statistics. All four are organization-wide aggregates, so every figure
is a tenant-scoped precomputed rollup refreshed off the request path — an org admin's dashboard
never triggers a live scan of the organization's attempts (guidelines §2, §10) and never sees
another tenant (guidelines §1.1).

## Dependencies

- Phase 3 results data (blocking substrate); departments/classrooms for the department dimension.
- **Phases 4–5 enrich in parallel:** engagement/activity signals (Phase 4) feed Student Engagement
  and Faculty Activity; dashboards must degrade gracefully while those signals don't exist yet.
- E6.1's `modules/analytics/` snapshot pipeline — org rollups extend the same mechanism
  (guidelines §23); Phase 1 tenancy/capabilities; chart primitives (folder-structure §6).

## Database Changes (conceptual)

- Organization-level rollup concepts, one per dimension: per-department performance aggregates,
  per-faculty activity counters, engagement summaries, and org-wide assessment statistics — each
  org-keyed, period-based, freshness-stamped, refreshed on schedule or data-change triggers.
  Prose only — no schema here.

## Backend Changes

- `modules/analytics/services/`: org-analytics services per dimension, authorizing the org-admin
  capability and scoping every read to the caller's organization from server context.
- `modules/analytics/repositories/` as the sole Prisma seam (folder-structure §8); scheduled
  refresh shares the background mechanism resolved in Phase 4; caching via `server/cache/`
  catalog keys (folder-structure §9).

## Frontend Changes

- Org Admin routes under `app/organization/` (thin shells) composing
  `modules/analytics/components/`: dashboard overview plus per-dimension views; shared tables
  with pagination/search for department and faculty breakdowns (guidelines §11, §13); freshness
  stamps; empty states for new organizations.

## Socket Events

None — periodic rollups; near-real-time org dashboards are not a PRD requirement.

## AI Components

None.

## API Endpoints (purpose-level)

- Fetch org dashboard summary (headline figures per dimension).
- Fetch department performance and faculty activity breakdowns, paginated (guidelines §5).
- Fetch student engagement and assessment statistics for a selected period.

## Permissions

- Org-admin capability to view institutional analytics for their own organization, registered via
  `modules/analytics/permissions.ts` (folder-structure §4, §9); frontend gates mirror the guard.

## Validation Rules

- Organization scope always from trusted server context — never a client-supplied org id
  (guidelines §1.1); department/faculty filter ids validated as belonging to that org; period
  selections bounded to defined windows.

## Edge Cases & Failure Scenarios

- New org with no departments/quizzes → empty states, not zeros presented as findings.
- Departments/faculty removed mid-period → rollups must not dangle; label or fold removed
  entities.
- Phase 4 engagement signals absent (parallel delivery) → engagement panel shows "data source not
  yet available" rather than fabricated metrics.
- Rollup refresh failure → serve last good rollup with staleness indicator (guidelines §7).

## Acceptance Criteria

- All four PRD §15 Organization metrics render in the Org Admin portal, scoped to the admin's own
  organization with isolation provable at the repository seam (guidelines §1.1).
- Every dashboard read hits precomputed rollups with visible freshness — no unbounded org-wide
  aggregate on the request path (guidelines §2, §10).
- Breakdowns paginate via shared components; empty/degraded states per guidelines §13.
