# F6.3.2 — System Health Surface

**Epic:** E6.3 — Platform Analytics & Ops Surfaces · **Complexity:** Medium

## Description

A System Health view in the Super Admin portal (PRD §4.1 "System Monitoring"; §15 Platform
"System Health"): a product surface telling the operator whether core dependencies and workloads
are healthy — conceptually: database, cache/queue, storage, realtime, email, and background-job
backlog/failure indicators. **Scope guard: product surface only** — it displays health signals;
it does not plan, provision, or configure monitoring infrastructure.

## Dependencies

- Phase 1 super-admin capabilities (guidelines §9).
- Health signal sources: the stack's own components (PostgreSQL, Redis, BullMQ, Azurite/Azure
  Blob, Socket.IO, Resend — PRD §17). Deeper telemetry depends on adopting the
  Sentry/OpenTelemetry recommendation (PRD §17) — **Needs clarification** (see Open Questions).
- Background-processing resolution from Phase 4 for queue-depth/failure signals.

## Database Changes (conceptual)

- Little to none. **Assumption:** health is computed on demand against each dependency with a
  short-lived cached status; if history is wanted, a small periodic health snapshot concept
  (component, status, checked-at) suffices. Prose only — no schema here.

## Backend Changes

- `modules/analytics/services/`: system-health service composing per-dependency status checks
  exposed by the infrastructure adapters (`server/db`, `server/cache`, `server/storage`,
  `server/socket`, `server/mail`) — modules depend on `server/` abstractions, never the reverse
  (folder-structure §5).
- Checks are cheap probes with timeouts, cached briefly via `server/cache/` catalog keys
  (folder-structure §9) so the dashboard never hammers dependencies.

## Frontend Changes

- Super Admin route under `app/admin/` (thin shell) composing `modules/analytics/components/`:
  per-component status list with checked-at times and queue backlog/failure indicators; degraded
  and unknown states visually distinct (guidelines §13).

## Socket Events

None — operator refresh/poll cadence suffices; a live health feed is not a PRD requirement.

## AI Components

None.

## API Endpoints (purpose-level)

- Fetch current system health status across components (super admin).
- Fetch background-job queue indicators (depth, recent failures), bounded.

## Permissions

- Super-admin-only capability to view system health, registered via
  `modules/analytics/permissions.ts` into the central catalog (folder-structure §4, §9).

## Validation Rules

- No client-supplied targets: the set of checked components is fixed server-side; any history
  range parameters bounded.

## Edge Cases & Failure Scenarios

- A dependency check hangs → probe timeouts so one sick component cannot stall the view; that
  component reports "unknown/timeout".
- The health surface's own dependency (e.g. Redis for caching) is the sick one → the surface
  still renders with that component marked down, not a blank page (guidelines §7).
- All-green false confidence → checked-at freshness always shown; stale checks flagged.

## Acceptance Criteria

- Super Admin sees a health status per core dependency plus background-job indicators, each with
  a checked-at time (PRD §4.1, §15).
- A failing dependency degrades only its own row; the surface never blocks on a single failure.
- Visible only to super admins (guidelines §9); probes bounded and cached — no expensive
  request-path work (guidelines §10).

## Definition of Done

- Passes `rules/definition-of-done.md`; documentation per `CLAUDE.md` (feature log entry;
  `docs/architecture.md` for health-check seams on the server adapters).

## Open Questions

- **Needs clarification:** what does "System Health" mean to the operator — dependency up/down
  plus queue state (assumed), or also error rates/latency requiring Sentry/OpenTelemetry
  (PRD §17, "Recommended") to be adopted first?
- Is a status history/uptime view needed, or is current status enough for MVP?
