# F6.1.2 — Student Progress & Trend Analytics

**Epic:** E6.1 — Faculty Analytics · **Complexity:** Medium

## Description

Faculty views of Student Progress and Performance Trends across quizzes (PRD §15 Faculty): how a
student's scores develop over the faculty member's quizzes, and how cohort performance trends over
time. Complements F6.1.1's per-quiz view with a per-student/over-time axis, served from rollups
precomputed off the request path (guidelines §2, §10). Post-hoc only — live in-quiz monitoring is
Phase 4's concern (PRD §12).

## Dependencies

- Phase 3 graded results across multiple quizzes — trends need history; blocking substrate.
- F6.1.1's snapshot pipeline in `modules/analytics/` — extended with a per-student dimension, not
  duplicated (guidelines §23).
- Phase 1 tenancy/capabilities; enrollment data defining "the faculty's students"; chart
  primitives in `src/components/charts/` (folder-structure §6).

## Database Changes (conceptual)

- A per-student performance rollup concept: per student, per faculty-owned quiz series — score
  history, completion status, trend-ready time series — org-scoped, freshness-stamped, recomputed
  as results finalize. Prose only — no schema here.

## Backend Changes

- `modules/analytics/services/`: student-progress service returning a student's history and trend
  series limited to quizzes the requesting faculty owns, plus cohort-trend reads.
- `modules/analytics/repositories/` as the only Prisma seam (folder-structure §8); rollups share
  F6.1.1's recompute mechanism; caching via `server/cache/` catalog keys (folder-structure §9).

## Frontend Changes

- Faculty portal surfaces under `app/faculty/` (thin shells) composing
  `modules/analytics/components/`: student list with shared search/pagination (guidelines §11,
  §13), per-student and cohort trend views, freshness stamp, insufficient-data states.

## Socket Events

None — post-hoc trends; live monitoring is Phase 4's concern (PRD §12).

## AI Components

None.

## API Endpoints (purpose-level)

- List students visible to the faculty member with summary progress, paginated (guidelines §5).
- Fetch one student's performance history/trend across the faculty's quizzes.
- Fetch cohort performance trend for the faculty's quiz series.

## Permissions

- Faculty capability to view progress of students in their own classrooms/quizzes, registered via
  `modules/analytics/permissions.ts` (folder-structure §4, §9); frontend gates mirror the guard.

## Validation Rules

- Student/quiz ids validated (module Zod validators); the student must belong to the faculty's
  classrooms/quizzes within the same organization, from server context (guidelines §1.1);
  time-range filters bounded.

## Edge Cases & Failure Scenarios

- Zero or one completed quiz → "not enough data for a trend" state, not a flat chart.
- Student unenrolled mid-series → history retention policy needed (see Open Questions).
- Disputes/regrades changing past scores → rollup recompute; trends update, freshness shown.
- Large cohorts → list/series reads paginated and bounded; no unbounded fan-out (guidelines §5, §10).

## Acceptance Criteria

- Faculty sees Student Progress and Performance Trends (PRD §15) only for students in their own
  classrooms/quizzes within their organization (guidelines §1.1).
- Trend reads come from precomputed rollups with visible freshness — no request-path scans of all
  attempts (guidelines §10).
- Insufficient-data and pending-grading states render honestly (guidelines §13).

## Definition of Done

- Passes `rules/definition-of-done.md`; documentation updated per `CLAUDE.md` (feature log entry;
  `docs/architecture.md` if the rollup pipeline changes shape).

## Open Questions

- What is a "trend" — score over time, over quiz sequence, or against class average?
  **Needs clarification** (PRD §15 names the metric only).
- Does progress include in-flight quizzes or completed ones only?
- Do trends persist for students who leave a classroom (retention/visibility policy)?
