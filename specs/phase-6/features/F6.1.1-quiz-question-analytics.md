# F6.1.1 — Quiz & Question Analytics

**Epic:** E6.1 — Faculty Analytics · **Complexity:** Medium

## Description

Per-quiz analytics for faculty: Average Score, Pass Rate, and Time Taken at quiz level, plus
Question Difficulty and per-question correctness/timing (PRD §15 Faculty), rendered in the Faculty
portal from Phase 3 graded results. All figures come from precomputed per-quiz snapshots recomputed
off the request path when results change — never live scans of attempts (guidelines §2, §10).

## Dependencies

- Phase 3 graded attempts (scores, per-question outcomes, timing) — blocking substrate.
- Phase 1 tenancy and capability catalog (guidelines §1.1, §9); new `modules/analytics/` module
  (folder-structure §4; reuse check guidelines §23); chart primitives (folder-structure §6).
- **Blocking clarification:** Pass Rate and Question Difficulty formulas (see Open Questions).

## Database Changes (conceptual)

- A per-quiz analytics snapshot concept: quiz-level aggregates (average score, pass counts, timing)
  and per-question rollups (correctness ratio, average time), freshness-stamped and org-scoped,
  recomputed when results finalize or regrade. Prose only — no schema here.

## Backend Changes

- `modules/analytics/services/`: quiz-analytics service — reads snapshots, requests recomputes on
  result finalization, enforces ownership/tenancy before returning data.
- `modules/analytics/repositories/` as the only Prisma seam (folder-structure §8). Recompute runs
  off the hot path (per Phase 4's background-processing resolution); payloads cached via
  `server/cache/` with catalog keys (folder-structure §9).

## Frontend Changes

- Faculty portal route under `app/faculty/` (thin shell, folder-structure §3) composing
  `modules/analytics/components/`: metric tiles, paginated per-question table (guidelines §13),
  time-taken visualization, freshness stamp, empty/pending states.

## Socket Events

None — snapshot-based dashboards; refetch on navigation suffices for post-hoc analytics.

## AI Components

None.

## API Endpoints (purpose-level)

- Fetch quiz-level analytics summary for a quiz the requesting faculty owns.
- Fetch per-question analytics (difficulty, correctness, timing), paginated (guidelines §5).

## Permissions

- Faculty capability to view analytics for their own quizzes, registered via
  `modules/analytics/permissions.ts` (folder-structure §4, §9), mirrored by frontend gates.

## Validation Rules

- Quiz id validated via module Zod validators; ownership and org scope from trusted server context
  (guidelines §1.1); any filters bounded and validated.

## Edge Cases & Failure Scenarios

- Zero attempts, or descriptive answers awaiting manual/AI grading → "not yet available" states;
  no divide-by-zero or misleading zeros.
- Regrade/dispute resolution after snapshot → recompute triggered; freshness always visible.
- Auto-terminated attempts (PRD §13) → inclusion policy undefined (see Open Questions).
- Recompute failure → last good snapshot with staleness shown; typed error if none (guidelines §7).

## Acceptance Criteria

- Faculty sees Average Score, Pass Rate, Question Difficulty, and Time Taken for their own quizzes
  only (PRD §15), tenant isolation intact (guidelines §1.1).
- No unbounded aggregate on the request path; snapshot freshness visible (guidelines §10).
- Empty and pending states render per guidelines §13.

## Definition of Done

- Passes `rules/definition-of-done.md`; documentation updated per `CLAUDE.md` (feature log entry
  and `docs/architecture.md` for the snapshot pipeline).

## Open Questions

- **Needs clarification:** Pass Rate formula — what defines "pass" (fixed threshold, per-quiz
  setting, org policy)? PRD §15 names the metric only.
- **Needs clarification:** Question Difficulty formula — correctness ratio, time-weighted, or
  another index? PRD §15 names the metric only.
- Do auto-terminated (integrity policy) attempts count toward Pass Rate and Difficulty?
