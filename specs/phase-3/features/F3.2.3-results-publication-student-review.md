# F3.2.3 — Results Publication & Student Review

**Epic:** E3.2 — Grading & Results · **Complexity:** Medium

## Description

Faculty publish a quiz's results; students then review their own result (PRD §4.4 "Review Results"; PRD §8 "Results Generated"). Until publication, students see only that their attempt was submitted — no scores. Publication is an explicit faculty action per quiz, gated on grading completeness (F3.2.2), and triggers the "Results Published" student notification (PRD §14). The review surface shows at minimum the total score; what more students may see is an open product decision (below).

## Dependencies

- F3.2.1 + F3.2.2 — complete scores; grading-completeness gating.
- Notification hook via `server/mail/` (Resend, PRD §14). **Assumption:** email and/or a simple in-app record this phase; Notification Center is a later phase.
- Code home: publication service in `modules/evaluations/`; student review UI in `modules/evaluations/components/` behind `app/student/` thin shells.

## Database Changes (conceptual)

A per-quiz publication state (published or not, by whom, when) governing result visibility. Publication is a visibility gate over existing evaluation data, not a copy of it. **Needs clarification (assumed no):** whether per-student publication is required — the PRD implies quiz-level only.

## Backend Changes

- Publish service: verify grading completeness (all submissions fully graded — **Assumption**, vs. publishing with exclusions), flip visibility, send notifications; idempotent — re-publishing is a no-op (guidelines §4.5).
- Student result query: own attempt result, shaped by the visibility policy, only when published; organization- and ownership-scoped in the service.
- Unpublish/republish support **Needs clarification** — needed for dispute-driven corrections (F3.3.1) unless corrections auto-flow to published results.

## Frontend Changes

- Faculty: publish control with gating feedback ("cannot publish — 3 submissions ungraded") and publication status on the quiz dashboard.
- Student: result review page — total score, submission metadata, and whatever per-question detail the visibility decision allows; entry from quiz history and notifications.

## Socket Events

None this phase — students learn of publication via notification/refresh; faculty dashboards poll. Phase 4 may push a results-published signal alongside live leaderboard work; the event catalog (`modules/socket/events/`) would gain it then.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Faculty: publish a quiz's results (gated, idempotent); fetch publication status/gating detail.
- Student: fetch own result for a published quiz.

## Permissions

- Faculty publish for quizzes they own. Students read only their own published results — enforced server-side, never by hiding UI alone. Declared in `modules/evaluations/permissions.ts` → `server/permissions/`.

## Validation Rules

- Publication rejected while any submission has ungraded subjective answers (per the gating assumption), with an actionable reason.
- Result reads rejected (typed error) when unpublished, not-own, or cross-organization.

## Edge Cases & Failure Scenarios

- Late submissions after publication (F3.1.3 sweep stragglers) → **Needs clarification:** auto-included once graded, or requiring republication?
- Notification send fails → publication still succeeds; failure logged and retryable (guidelines §4.5 — no fire-and-forget without an error handler).
- Marks changed after publication (dispute resolution) → student view reflects the correction; the change is audit-logged (guidelines §8.8).

## Acceptance Criteria

- No student can observe any score before publication, by UI or by API.
- Publication flips visibility for all graded submissions and notifies students (PRD §14 "Results Published").
- The review page renders correctly for submitted, auto-terminated, and fully-manual-graded attempts alike.

## Definition of Done

Meets `rules/definition-of-done.md`; visibility policy decision recorded before build; docs updated per CLAUDE.md (`docs/logs/` entry).

- Publication gating: strict all-graded, or publish-with-exclusions?
- Handling of late submissions and unpublish/republish (above).
