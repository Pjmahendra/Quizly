# F3.1.1 — Quiz Join & Eligibility

**Epic:** E3.1 — Attempt Lifecycle · **Complexity:** Medium

## Description

A student joins a published, currently-active quiz they are eligible for (PRD §4.4 "Join Quiz", §8 "Students Join"). Eligibility means: the quiz is published and within its scheduled window; the student is enrolled in the quiz's classroom or explicitly invited (PRD §8 "Invite Students"); and student and quiz belong to the same organization. Joining either creates a new attempt in `not-started` state or resumes the student's existing attempt (handing off to F3.1.2). The student portal lists joinable quizzes.

## Dependencies

- Phase 2: published quizzes with schedule windows; classroom enrollment; quiz invitations (**Assumption:** invitations exist from Phase 2 per PRD §8; if not, enrollment-only this phase).
- Code home: `modules/attempts/` (join service, eligibility rules), consuming `modules/quizzes/` and `modules/classrooms/` via `index.ts` only. UI: entry surfaces in `app/student/` and `app/quiz/` as thin shells (folder-structure §3).

## Database Changes (conceptual)

An attempt record is created at join (owned by F3.1.2's model): student, quiz, organization, state, join timestamp. Eligibility is derived from existing publication/schedule/enrollment/invitation data. "One active attempt per student per quiz" is a persistence-level invariant.

## Backend Changes

- Eligibility service in `modules/attempts/services/`: publication state, schedule window, enrollment/invitation, tenancy — returning typed, distinguishable refusal reasons (`lib/errors.ts` base + module error types, folder-structure §9).
- Join action in `modules/attempts/actions/` (transport only), idempotent — repeated joins return the same attempt (guidelines §4.5). Repository queries scoped by organization from server-side context, never client input (CLAUDE.md multi-tenancy rule).

## Frontend Changes

- Student portal: "available quizzes" list with schedule/status and a join entry point.
- Quiz entry screen in `app/quiz/`: title, duration, instructions, "Start" handoff to F3.1.2. Feature UI lives in `modules/attempts/components/` (folder-structure §4).

## Socket Events

None this phase — the available-quizzes list refreshes by polling/reload. Phase 4 upgrades join to establish the live Socket.IO connection and emit `student-connected` from the event catalog (`modules/socket/events/`, folder-structure §5).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List quizzes the student can currently join (published, in-window, eligible).
- Join a quiz — validate eligibility, create or return the student's attempt.
- Fetch join/eligibility status for one quiz (drives the entry screen).

## Permissions

- Student capability to view/join eligible quizzes; declared in `modules/attempts/permissions.ts` and registered into `server/permissions/` (guidelines §9 — no strings at call sites).

## Validation Rules

- Quiz published and within its scheduled window at join time (server clock).
- Student enrolled/invited and in the same organization as the quiz.
- Reject joining a quiz already submitted by the student (subject to F3.1.2's retake decision).

## Edge Cases & Failure Scenarios

- Two simultaneous join requests → exactly one attempt (race guarded, guidelines §4.5).
- Quiz unpublished or window closed between listing and join → typed refusal.
- Student removed from classroom after joining → attempt continues; **Assumption:** eligibility is checked at join, not continuously.
- **Needs clarification:** late-join policy — may a student join after the window opens, and does joining late shrink their duration or keep it (interacts with F3.1.3)?

## Acceptance Criteria

- Eligible student joins and lands in an attempt; each ineligibility case yields a distinct, student-comprehensible refusal.
- Re-joining resumes the same attempt; no duplicate attempts under any request pattern.
- Join events are audit-logged (guidelines §8.8).
