# F3.2.1 — Objective Grading Engine

**Epic:** E3.2 — Grading & Results · **Complexity:** Large

## Description

Automatic grading for the objective question types in PRD §10: MCQ, Multiple Select, True/False, and Numerical. Grading runs when an attempt is submitted (student submit or auto-submit, F3.1.2/F3.1.3), scoring each objective answer against the question's answer key and producing per-question scores and an attempt-level objective subtotal. Grading is deterministic and idempotent: regrading the same attempt yields identical results. Subjective types route to F3.2.2, not here. **Assumption:** Short Answer is manually graded this phase (PRD §10 lists it under neither heading explicitly; PRD §11 targets it with AI later).

## Dependencies

- F3.1.2 — submitted attempts, answers, and the presentation snapshot (option identity resolved through the snapshot so shuffled options grade correctly).
- Phase 2 — questions carry answer keys and maximum marks (**Assumption**; gap closes first).
- Code home: `modules/evaluations/` — pure grading logic in services/utils, persistence via its repositories (folder-structure §4, §8).

## Database Changes (conceptual)

Per-question evaluation records (score awarded, max marks, derivation: auto vs. manual) and an attempt-level result aggregate (objective subtotal, pending-manual flag, total once complete). Evaluation data is owned by `modules/evaluations/`, referencing attempts — not embedded in them.

## Backend Changes

- Grading functions per question type as pure, side-effect-free logic: exact-match MCQ/True-False; set comparison for Multiple Select; numeric comparison for Numerical.
- Orchestration service: on submission, grade objective answers via the snapshot, store per-question scores, compute the subtotal, mark the attempt "pending manual grading" if subjective questions exist, else "fully graded".
- Idempotency: triggered on submit exactly once per attempt; a re-trigger overwrites with identical values, never double-counts (guidelines §4.5).

## Frontend Changes

- None student-facing (results hidden until F3.2.3 publication).
- Faculty: per-quiz grading-status summary (auto-graded vs. awaiting manual) feeding the F3.2.2 workbench entry point, in `modules/evaluations/components/`.

## Socket Events

None this phase — grading completes synchronously with submission; faculty status views poll. Phase 4 may surface completion via the live dashboard (completion events from the event catalog, `modules/socket/events/`); this feature itself requires no new event.

## AI Components

None this phase. (AI never grades objective types in any phase; PRD §11's scope is subjective.)

## API Endpoints (purpose-level)

- (Internal) grade attempt on submission — service-to-service, not client-invoked.
- Faculty: fetch grading status/summary for a quiz's submissions.
- Faculty: re-run objective grading for an attempt (recovery; idempotent).

## Permissions

- No student capability. Faculty view grading status and re-run grading for owned quizzes only; declared in `modules/evaluations/permissions.ts` → `server/permissions/`.

## Validation Rules

- A question is gradable only if its answer key exists and is well-formed; otherwise the attempt is flagged for faculty attention rather than silently scored zero.
- Scores bounded: 0 ≤ awarded ≤ question max; subtotals equal the sum of parts.
- **Needs clarification:** Multiple Select partial credit — all-or-nothing vs. partial (do wrong selections deduct?). **Needs clarification:** Numerical tolerance — exact vs. configured tolerance/precision, and who configures it (per question?).

## Edge Cases & Failure Scenarios

- Unanswered objective question → scored zero, recorded as unanswered (distinct from answered-wrong, for analytics and disputes).
- Answer key edited after submissions exist → existing scores unchanged until an explicit faculty regrade; the change is audit-logged. **Assumption:** no silent retroactive regrade.
- One malformed question → other questions still grade; the failure is isolated, flagged, and recoverable via re-run.

## Acceptance Criteria

- All four objective types grade correctly against their keys, including shuffled-option quizzes (via the snapshot), immediately on submit and auto-submit.
- Regrade of an unchanged attempt is a no-op; totals always equal the sum of per-question scores.
- Attempts with subjective questions are correctly marked pending-manual.
