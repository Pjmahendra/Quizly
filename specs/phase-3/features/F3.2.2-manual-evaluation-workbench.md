# F3.2.2 — Manual Evaluation Workbench

**Epic:** E3.2 — Grading & Results · **Complexity:** Large

## Description

The faculty surface for manually grading subjective answers — descriptive answers, essays, long answers (PRD §10 "Manual Evaluation"), plus short answers per the F3.2.1 assumption. Faculty open a quiz's submissions, step through each student's subjective answers (rendered against the question and, where authored, the expected answer), and record per-question marks and written feedback. The workbench tracks grading completeness per attempt and per quiz, gating results publication (F3.2.3).

**This is the exact surface Phase 7 AI evaluation extends** (PRD §11): "Evaluate with AI", suggested score/feedback/reasoning, and accept/modify/override all attach to this same per-response review UI in `modules/evaluations/components/` (folder-structure §11). Nothing here may assume a human is the only producer of a suggested score.

## Dependencies

- F3.2.1 — evaluation records and pending-manual status; F3.1.2 — submitted answers and snapshot.
- Code home: `modules/evaluations/` (services, repositories, validators, `components/` faculty review UI); routes under `app/faculty/` as thin shells (folder-structure §3).

## Database Changes (conceptual)

Manual evaluation extends the per-question evaluation record: marks awarded, feedback text, grader identity, timestamps, and revision history for mark changes (needed by disputes, F3.3.1). The record already distinguishes auto vs. manual derivation (F3.2.1); Phase 7 adds an AI-suggested variant to the same concept.

## Backend Changes

- Grading service: record/update marks + feedback for one answer; recompute the attempt total and grading-completeness state atomically with the mark write.
- Queries: a quiz's submissions with grading progress; one attempt's subjective answers with question context, expected answer, and current marks.
- Mark updates idempotent, last-write-wins per grader action, revision history preserved (guidelines §4.5, §8.8).

## Frontend Changes

- Workbench in `app/faculty/` + `modules/evaluations/components/`: submission list with progress; per-student view stepping question-by-question; marks input bounded by max; feedback field; per-quiz completeness summary. Layout reserves space for Phase 7's AI suggestion panel so that addition is additive, not a redesign.

## Socket Events

None this phase — submission lists and progress refresh by polling. Phase 4 may push "new submission arrived" via the live dashboard; Phase 7's post-MVP batch AI evaluation (PRD §11) would add queue-progress events. Neither changes this feature's contract.

## AI Components

None this phase. Phase 7 (PRD §11) adds per-response AI evaluation on top of this exact surface; this feature must not block that (see Description).

## API Endpoints (purpose-level)

- List a quiz's submissions with grading status (paginated per guidelines).
- Fetch one attempt's subjective answers with question context for grading.
- Record/update marks and feedback for one answer.

## Permissions

- Faculty grade only quizzes they own; org-admin visibility **Needs clarification** (PRD §4.2 gives admins reports, not grading). Declared in `modules/evaluations/permissions.ts` → `server/permissions/`. Students have no access.

## Validation Rules

- Marks numeric, 0 ≤ marks ≤ question max; feedback length bounded; Zod schemas in `modules/evaluations/validators/`.
- Grading allowed only on submitted/auto-terminated attempts, never in-progress ones.
- Mark changes after publication (F3.2.3) go through the dispute/regrade path, never silent edits — every post-publication change is audit-logged.

## Edge Cases & Failure Scenarios

- Two graders (or tabs) grade the same answer concurrently → deterministic last write; revision history keeps both entries.
- Auto-terminated attempts with unanswered subjective questions → shown as unanswered, gradable as zero without free-text input.
- Quiz with no subjective questions → "nothing to grade"; publication gating trivially satisfied.

## Acceptance Criteria

- Faculty can fully grade every subjective answer of every submission for their quiz; marks and feedback persist and totals recompute immediately.
- Grading completeness is accurate per attempt and per quiz, and is what F3.2.3 gates on.
- All grading actions are attributable and audit-logged (guidelines §8.8).
