# F3.1.3 — Server-Authoritative Timing & Auto-Submit

**Epic:** E3.1 — Attempt Lifecycle · **Complexity:** Large

## Description

Quiz duration is enforced by the server, never the client. When an attempt starts, the server computes an authoritative deadline from the quiz's duration and schedule; the client timer is a display convenience only. At the deadline, an unsubmitted attempt is auto-submitted: remaining questions are recorded as unanswered and the attempt marked auto-submitted for time expiry — deliberately reusing the termination semantics defined in PRD §13 ("remaining questions are submitted as unanswered"), while integrity *enforcement* itself stays in Phase 4. Timing must tolerate client clock drift, disconnects, and clients that never return.

## Dependencies

- F3.1.2 — the deadline gates saves/submits; auto-submit is a transition of that state machine.
- Phase 2 quiz scheduling data (window, duration).
- Code home: timing service in `modules/attempts/`; background enforcement uses BullMQ/Redis (PRD §17). **Assumption:** a queue adapter is introduced under `server/` this phase (folder-structure §5 lists no queue home yet — placement confirmed before implementation).

## Database Changes (conceptual)

The attempt carries its server-computed deadline (set once at start) and, on auto-submit, a submission reason distinguishing time expiry from student submission and from Phase 4 integrity termination. The deadline is immutable this phase — faculty "Extend Time" (PRD §4.3) is a Phase 4 live-quiz control.

## Backend Changes

- Deadline computation at start: min(start + duration, quiz window end). **Needs clarification:** whether window end truncates duration for late joiners — tied to F3.1.1's late-join policy.
- **Lazy enforcement (primary):** every save/submit re-checks the deadline against the server clock; past-deadline saves are rejected and the attempt auto-submitted on touch.
- **Background sweep (safety net):** a scheduled/delayed job auto-submits overdue attempts whose clients never returned, so grading is never blocked by an absent client; idempotent with lazy enforcement and racing student submits (guidelines §4.5).
- Auto-submit reuses F3.1.2's submit path (one code path, one set of invariants) and triggers objective grading (F3.2.1).

## Frontend Changes

- Countdown timer driven by server-provided deadline + periodic remaining-time re-sync to correct drift; warning states near the deadline.
- Deadline-reached UX: inputs lock; student told the attempt was auto-submitted.

## Socket Events

None this phase — the client re-syncs remaining time by polling; auto-submit is discovered on the next request. Phase 4 upgrades to `timer-sync` and `student-auto-submitted` from the event catalog (`modules/socket/events/`), plus faculty pause/resume/extend-time controls (PRD §4.3, §7).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Fetch authoritative remaining time / deadline for the student's attempt (timer sync).
- (Internal) auto-submit overdue attempt — invoked by lazy enforcement and the sweep, never directly by clients.

## Permissions

- Students read only their own attempt's timing; auto-submit is system-initiated, no user-facing mutation. Registered via `modules/attempts/permissions.ts` → `server/permissions/`.

## Validation Rules

- Deadline derives exclusively from server-side quiz data and server clock — no client-supplied timestamp is trusted anywhere in the flow.
- A small, named grace tolerance for network latency on last-second saves. **Needs clarification:** grace duration — product/fairness decision.

## Edge Cases & Failure Scenarios

- Client clock hours off → display may differ; enforcement unaffected.
- Student disconnects and never returns → sweep auto-submits; results flow proceeds.
- Save in flight at the boundary → accepted within grace, else rejected; deterministic, never partially applied. Sweep racing a student submit → exactly one submission wins.
- Sweep outage → lazy enforcement still guarantees correctness on next touch; overdue attempts caught up when the sweep resumes.

## Acceptance Criteria

- No save or submit is accepted past deadline + grace, regardless of client behavior.
- Every overdue attempt reaches its terminal state without client participation, remaining questions recorded as unanswered (PRD §13 semantics).
- Auto-submissions are audit-logged with reason = time expiry (guidelines §8.8).
