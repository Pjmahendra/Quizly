# F3.3.2 — Quiz Concerns

**Epic:** E3.3 — Disputes & Concerns · **Complexity:** Small

## Description

Students raise a concern about a quiz itself (PRD §4.4 "Raise Quiz Concerns") — distinct from result disputes (F3.3.1): a concern targets the quiz or a question (wrong/ambiguous question, typo, delivery problem such as timing), not the grading of the student's own result, and does not require published results — it can be raised after submitting, before results exist. Faculty are notified ("Student Concern Raised", PRD §14), review, respond, and close.

**Needs clarification:** as with disputes, the PRD defines no resolution workflow. A minimal lifecycle (open → closed with a faculty response) is **assumed**; anything richer (categories, escalation, converting a concern into a mass regrade) needs a product decision.

## Dependencies

- F3.1.2 — concerns reference the student's attempt/snapshot as context where relevant.
- F3.3.1 — shares the `modules/disputes/` module home, notification hook, and faculty inbox (**Assumption:** one module owns both; they remain separate concepts within it, not one record type with a flag — the distinction is product-visible per PRD §4.4).
- Notification via `server/mail/` (Resend, PRD §14).

## Database Changes (conceptual)

A concern record: raising student, target quiz (optionally a specific question), description, status, faculty response, timestamps, organization scope. Deliberately not attached to a result.

## Backend Changes

- Raise service: validate the student actually participated in (or was eligible for) the quiz, create the concern, notify the quiz owner; idempotent against duplicate submission (guidelines §4.5).
- Respond/close service and role-scoped, paginated list queries — mirroring F3.3.1's shape so the faculty inbox presents both streams consistently.

## Frontend Changes

- Student: "raise concern" entry from the attempt/quiz context (post-submission this phase) and a tracked list alongside disputes.
- Faculty: concerns appear in the same inbox surface as disputes, visually distinguished.

## Socket Events

None this phase — notification + polling suffice. Phase 4's live quiz experience is where *during-quiz* concern raising becomes meaningful (surfacing on the live faculty dashboard); the event would then be added to the catalog in `modules/socket/events/`.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Student: raise a concern on a quiz/question; list own concerns.
- Faculty: list concerns for owned quizzes; respond/close.

## Permissions

- Student: raise/read own concerns for quizzes they participated in. Faculty: manage concerns on owned quizzes. Declared in `modules/disputes/permissions.ts` → `server/permissions/`.

## Validation Rules

- Description required and length-bounded; a referenced question must belong to the quiz as the student saw it (snapshot check).
- Duplicate open concern by the same student on the same target → rejected, pointed to existing.
- **Needs clarification:** may a student raise a concern mid-attempt this phase, or only after submission? (Assumed post-submission until Phase 4.)

## Edge Cases & Failure Scenarios

- Concern about a question that faculty subsequently edits or removes → concern retains snapshot context and remains reviewable.
- Concern proves a question invalid after results are published → any grading remedy flows through F3.2.2/F3.3.1 mechanics, not the concern record itself.
- Notification failure → creation still succeeds; delivery retried/logged (guidelines §4.5).

## Acceptance Criteria

- A student who took a quiz can raise a concern before or after results exist, distinct from a dispute, and track its status and the faculty response.
- Quiz-owning faculty are notified (PRD §14) and can close concerns with a response.
- Tenancy and ownership scoping hold for every read and write; actions are audit-logged (guidelines §8.8).

## Definition of Done

Meets `rules/definition-of-done.md`; the concern-vs-dispute distinction documented; docs updated per CLAUDE.md (`docs/logs/` entry).

## Open Questions

- Resolution workflow depth, categories, org-admin visibility (shared with F3.3.1).
- Mid-attempt raising this phase (above).
