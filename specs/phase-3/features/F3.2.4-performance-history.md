# F3.2.4 — Performance History

**Epic:** E3.2 — Grading & Results · **Complexity:** Small

## Description

A student's own performance history (PRD §4.4 "View Performance History"): a chronological list of their quiz attempts with status (in-progress, submitted, auto-submitted, awaiting grading, published) and, for published results, the score. Each published entry links into the F3.2.3 review page. This is a personal read model, deliberately distinct from faculty/organization analytics (PRD §15, `modules/analytics/`, later phase) — no aggregates beyond the student's own rows this phase.

## Dependencies

- F3.1.2 (attempts) and F3.2.3 (published results + visibility policy).
- Code home — **Assumption:** read services in `modules/attempts/` composing published-result data from `modules/evaluations/` via its `index.ts` (folder-structure §14); history UI in `modules/attempts/components/` behind `app/student/` thin shells. Revisit if `modules/analytics/` proves the better owner when it lands.

## Database Changes (conceptual)

None expected — history is a query over existing attempt and evaluation data. If retakes are allowed (F3.1.2 open question), history must represent multiple attempts per quiz and which one counts; that decision shapes the query, not new concepts.

## Backend Changes

- History query service: the student's attempts across quizzes, newest first, paginated (guidelines: every list is paginated), each row carrying quiz title, date, status, and score when published — score omitted entirely otherwise.
- Ownership/tenancy enforced in the service from server-side context.

## Frontend Changes

- Student portal history page: paginated list with status badges, empty state for new students, link-through to result review (published) or the attempt (in-progress). Shared table/pagination primitives from `src/components/` (folder-structure §6); feature specifics stay in the module.

## Socket Events

None this phase — history is request/response and refreshes on load. Phase 4's live features do not change this surface; no upgrade planned beyond fresher data arriving sooner.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Student: fetch own attempt/result history (paginated, filter by classroom or quiz optional).

## Permissions

- Student reads own history only. Faculty/peer visibility is **out of scope** — "View Peer Profiles" (PRD §4.4) is privacy-policy-gated and not this feature. Declared via `modules/attempts/permissions.ts` → `server/permissions/`.

## Validation Rules

- Pagination inputs validated (page size bounds); filters restricted to entities the student can see; unpublished scores never serialized into any response shape.

## Edge Cases & Failure Scenarios

- Attempt for a quiz later deleted/archived by faculty → history row must not break; **Needs clarification:** display policy for deleted quizzes (retain row with quiz snapshot data?). Interacts with Phase 2 quiz-deletion semantics.
- Mixed states across many quizzes → statuses must be mutually exclusive and accurate; an auto-submitted attempt reads distinctly from a student-submitted one.
- Retakes (if allowed) → all attempts listed, counting attempt clearly marked.

## Acceptance Criteria

- A student sees every attempt they have made, with accurate status, and scores only for published results.
- No path exposes another student's history or an unpublished score.
- List paginates and renders an empty state per guidelines §13.
