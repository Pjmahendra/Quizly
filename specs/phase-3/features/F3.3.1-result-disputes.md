# F3.3.1 — Result Disputes

**Epic:** E3.3 — Disputes & Concerns · **Complexity:** Medium

## Description

Students raise a dispute against their own published result (PRD §4.4 "Raise Result Disputes"), optionally targeting a specific question, with a written justification. Faculty are notified ("Student Concern Raised", PRD §14), review the dispute alongside the student's recorded answer and the what-they-saw snapshot (F3.1.2), respond, and resolve — with or without a mark change. Students track status and outcome. Mark changes flow through the evaluations module (F3.2.2's revision-tracked mark update), never through a dispute-side edit.

**Needs clarification:** the PRD names the capability but gives no resolution workflow — status set, SLAs, escalation (org admin?), and student reply/appeal are all open. A minimal lifecycle (open → resolved, single faculty response) is **assumed** this phase.

## Dependencies

- F3.2.3 — disputes attach to published results only; F3.2.2 — mark-correction path with revision history. Notification hook via `server/mail/` (Resend, PRD §14).
- Code home — **Assumption:** new `modules/disputes/` module (standard anatomy; new-domain rule, folder-structure §14), consuming attempts/evaluations via `index.ts` only. UI behind `app/student/` and `app/faculty/` thin shells.

## Database Changes (conceptual)

A dispute record: raising student, target attempt/result (optionally a specific question), justification, status, faculty response, resolution outcome, timestamps, organization scope. Linked one-way to evaluation revisions when a correction results. Prose only.

## Backend Changes

- Raise service: validate eligibility (own result, published, within any raise-window), create dispute, notify quiz-owning faculty; duplicate submissions idempotent (guidelines §4.5).
- Resolve service: record response/outcome; mark changes delegate to the evaluations mark update (totals re-derive there); then close the dispute.
- List/track queries for both roles, organization-scoped, paginated.

## Frontend Changes

- Student: "raise dispute" entry on the result review page (F3.2.3); dispute list with status and outcome in the student portal.
- Faculty: dispute inbox per quiz with student answer + snapshot context and a resolve flow.

## Socket Events

None this phase — faculty learn of disputes via notification and inbox polling. Phase 4 can add a real-time toast on the faculty dashboard; the event would then join the catalog in `modules/socket/events/` (folder-structure §5), never a literal at the emit site.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Student: raise a dispute; list own disputes with status/outcome.
- Faculty: list disputes for owned quizzes; fetch one with context; respond/resolve.

## Permissions

- Student: raise/read own disputes only. Faculty: manage disputes on owned quizzes. **Needs clarification:** org-admin visibility/escalation. Declared in `modules/disputes/permissions.ts` → `server/permissions/` (guidelines §9).

## Validation Rules

- Only own, published results are disputable; justification required and length-bounded (Zod in `modules/disputes/validators/`).
- One open dispute per (student, result, question) — duplicates rejected, pointed to existing.
- **Needs clarification:** raise window (e.g. N days after publication) — none assumed absent a decision.

## Edge Cases & Failure Scenarios

- Dispute raised, then faculty independently corrects marks → resolution references the correction, never double-applies it.
- Faculty account deactivated with open disputes → disputes must remain reachable (**Needs clarification:** reassignment path — ties to the org-admin question).
- Notification failure → dispute creation still succeeds; delivery retried/logged (guidelines §4.5 — no unhandled fire-and-forget).

## Acceptance Criteria

- A student can raise, see, and track a dispute on their published result; the faculty owner is notified and can resolve it with an explanation, with or without a mark change.
- A mark-changing resolution updates the student's visible result and is fully audit-logged (guidelines §8.8), attributable end-to-end.
- No cross-tenant or cross-student visibility under any query.
