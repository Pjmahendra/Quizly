# F3.1.2 — Attempt State Machine & Answer Persistence

**Epic:** E3.1 — Attempt Lifecycle · **Complexity:** Extra Large

## Description

The core attempt model: a state machine (`not-started` → `in-progress` → `submitted` | `auto-terminated`) with idempotent answer persistence. Every answer save and the final submit must be safe under retries, double-clicks, flaky networks, and concurrent tabs (guidelines §4.5). The attempt also records **exactly what the student saw** — question order and per-question option order produced by randomization/shuffle (PRD §4.3) — so grading, review, and disputes can reproduce the student's view. This is the correctness heart of the product; Phase 4's live experience and Phase 7's AI evaluation both build on this record.

## Dependencies

- F3.1.1 creates the attempt this feature manages; Phase 2 question/quiz models (types, randomization settings).
- Code home: `modules/attempts/` — services (transitions, orchestration), repositories (only Prisma call sites, folder-structure §8), validators, `components/` for the answering UI.

## Database Changes (conceptual)

An attempt (student, quiz, organization, state, start/submit timestamps, submission reason); a per-attempt presentation snapshot (ordered question IDs, per-question option ordering, captured at start); per-question answer records typed by question type with save metadata supporting idempotency and last-write ordering. One active attempt per student per quiz enforced at the persistence level. Prose only — schema design happens at implementation.

## Backend Changes

- State-machine service: legal transitions only; illegal/repeated transitions rejected with typed errors; transitions race-guarded (transactions/optimistic concurrency — guidelines §4.5 requires naming the temporal coupling in the design).
- Answer-save service: idempotent upsert per (attempt, question); a stale or duplicate save never resurrects an older answer; saves rejected once the attempt leaves `in-progress`.
- Submit service: idempotent — duplicate submits return the already-submitted outcome; first submit triggers objective grading (F3.2.1).
- Snapshot service: generates and persists the randomized presentation at start; all reads (student UI, grading, review) go through the snapshot, never re-randomize.

## Frontend Changes

- Attempt-taking UI in `app/quiz/` (thin shell) + `modules/attempts/components/`: navigation per snapshot order, inputs for all MVP question types (PRD §4.3), answered/unanswered indicators, save-state feedback, explicit submit with confirmation.
- Resume: reload/reopen restores state and saved answers from the server.

## Socket Events

None this phase — answer saves are plain requests; progress needs no broadcast yet. Phase 4 upgrades: `answer-submitted` and `student-auto-submitted` from the event catalog (`modules/socket/events/`) feed the faculty monitoring dashboard (PRD §12).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Start attempt (transition to `in-progress`, generate snapshot).
- Fetch attempt state + snapshot + saved answers (resume/render).
- Save/replace an answer for one question (idempotent).
- Submit attempt (idempotent, terminal).

## Permissions

- Student: operate only on their **own** attempt (ownership enforced in the service, not the route). Faculty: read attempts for owned quizzes (feeds F3.2.2). Declared in `modules/attempts/permissions.ts` → `server/permissions/`.

## Validation Rules

- Answer payloads validated per question type via Zod in `modules/attempts/validators/` — e.g. selected options must exist in the snapshot for that question.
- Transitions validated against the state machine; saves only while `in-progress` and before the F3.1.3 deadline.

## Edge Cases & Failure Scenarios

- Concurrent saves to one question (two tabs) → deterministic last-write outcome; concurrent save + submit → save lands before submit or is rejected — never corruption.
- Double-submit / retried submit → single submission record, same response.
- Disconnect mid-attempt → saved answers intact on resume; unsaved local input lost (acceptable pre-Phase 4).
- **Needs clarification:** attempt limits/retakes are unspecified in the PRD — single attempt? faculty-configured N? which attempt counts (best/latest)? Blocks F3.1.1 join and F3.2.4 rules.

## Acceptance Criteria

- All MVP question types can be answered, saved, revised, and submitted; resume restores exact state; no request pattern loses an answer, duplicates an attempt, or double-submits.
- The persisted snapshot reproduces the student's question/option ordering exactly for grading and review; transitions and submissions are audit-logged (guidelines §8.8).
