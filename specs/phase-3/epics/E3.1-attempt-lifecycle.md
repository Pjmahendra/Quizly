# E3.1 — Attempt Lifecycle

## Purpose

Give students a complete, correct quiz-taking flow without real-time features: join an eligible
published quiz (PRD §4.4 "Join Quiz", §8), progress through an attempt whose state and answers are
persisted server-side, and be submitted — by the student or automatically at the server-computed
deadline. The attempt record must capture exactly what each student saw (question order, option
order) so grading and disputes are reproducible against randomized quizzes (PRD §4.3).

## Business Value

This epic is the correctness heart of the product: every downstream capability — grading, results,
disputes, analytics, and the Phase 4 live experience — is only as trustworthy as the attempt
record. A student-visible correctness failure here (lost answers, double submission, wrong
deadline) is reputationally unrecoverable for an assessment platform. Landing it now completes the
PRD Phase 1 milestone loop (PRD §18) and gives Phase 4 a stable state machine to attach Socket.IO
events to, rather than building state and transport at the same time.

## Dependencies

- Phase 1: authentication, RBAC, organization tenancy scoping.
- Phase 2: quiz creation/scheduling/publishing, question bank and question types, randomization
  and shuffle settings, classroom enrollment (PRD §4.3, §5, §8).
- `modules/attempts/` is the code home (folder-structure §4); it consumes `modules/quizzes/` and
  `modules/classrooms/` via their `index.ts` only (folder-structure §14).
- Within this epic: F3.1.2 depends on F3.1.1; F3.1.3 depends on F3.1.2.

## Features

| ID | Feature | Complexity |
|---|---|---|
| F3.1.1 | Quiz join & eligibility | Medium |
| F3.1.2 | Attempt state machine & answer persistence | Extra Large |
| F3.1.3 | Server-authoritative timing & auto-submit | Large |

## Acceptance Criteria

- An eligible student (published/active quiz, enrolled/invited, same organization) can join and
  receive an attempt; ineligible students are rejected with a typed, explicit reason.
- Attempt state transitions follow the defined machine (not-started → in-progress → submitted /
  auto-terminated) and reject invalid or concurrent conflicting transitions.
- Answer saves and attempt submission are idempotent (guidelines §4.5): retries, double-clicks,
  and duplicate requests never lose data, duplicate answers, or double-submit.
- Each attempt records the exact question and option ordering the student saw; a grader or
  reviewer can reconstruct the student's view of the quiz.
- The deadline is computed and enforced server-side; a student past the deadline can no longer
  save answers, and an unsubmitted attempt is auto-submitted with remaining questions unanswered
  (PRD §13 termination semantics reused for time expiry).
- All join/submit/auto-submit events are captured in the audit trail (guidelines §8.8) via
  `modules/audit/`.
