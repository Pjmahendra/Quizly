# E0.3 — Design System Foundation

## Purpose

Encode the visual language of `rules/design-specification.html` ("Quizly — Architectural Design
System 2026") as the project's single styling primitive: named tokens for color, typography,
spacing, radius, elevation, motion, and z-index, with light/dark theming applied through token
overrides at a single root (guidelines §12); then seed the shared, feature-agnostic UI primitive
library (`components/ui`, `components/feedback`, `components/tables`, `components/forms` per
folder-structure §6) that guidelines §11/§13 require every future surface to reuse.

## Business Value

Design consistency is a feature and every deviation is debt (guidelines §12). Seeding tokens and
primitives before any portal UI exists guarantees that all four PRD portals (PRD §4) are composed
from one visual system rather than converging on it later at high cost. The shared table,
pagination, dialog, empty-state, skeleton, and toast primitives are mandated abstractions
(guidelines §11 "abstract every reusable pattern once", §13) — building them once here prevents
per-screen hand-rolling across every later phase.

## Acceptance Criteria

- All token categories (color with semantic roles, typography scale, spacing base unit, radius, elevation, motion, z-index) are defined as named tokens traceable to `rules/design-specification.html`; no raw hex/pixel/z-index values appear in component code (guidelines §12).
- Light and dark themes are produced by token overrides at a single root — never by forked components — and honor the user's reduced-motion preference (guidelines §12, §14).
- The seed primitives (button, input, dialog, table, pagination, empty state, skeleton, toast) exist in their folder-structure §6 homes, are feature-agnostic, and render correctly in both themes.
- Every interactive primitive expresses its full interaction-state set (rest, hover, focus, active, selected, disabled, loading) and meets WCAG 2.1 AA (guidelines §12, §14).
