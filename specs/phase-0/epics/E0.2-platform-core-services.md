# E0.2 — Platform Core Services

## Purpose

Build the cross-cutting infrastructure seams that every future module depends on, in their
canonical homes per `rules/folder-structure.md` §5, §7, §9: the single pooled Prisma client and
repository conventions (`server/db`), the shared structured logger with correlation IDs
(`server/logger`), the typed error framework and standard response envelope (`lib/errors.ts`),
the Redis cache core with its cache-key catalog (`server/cache`), and a health/readiness endpoint
that reports real dependency readiness (guidelines §18).

## Business Value

These services are the contract that keeps fifteen future domain modules consistent: one way to
reach the database, one logging surface, one error vocabulary, one response envelope, one cache-key
catalog. Establishing them before any product code exists means no module ever invents a parallel
path (guidelines §23), and production issues are diagnosable from day one via correlated,
structured logs and honest health signals — directly serving the reliability and performance
non-functional requirements (PRD §16).

## Dependencies

- E0.1 — requires the scaffold (F0.1.1), validated env (F0.1.2), and local service wiring (F0.1.3).

## Acceptance Criteria

- Exactly one pooled Prisma client exists (`server/db`); repository conventions are documented and no code path touches Prisma outside a repository (folder-structure §8; guidelines §6).
- All application logging flows through `server/logger` with module identifiers and correlation IDs; no `console` output in application code (guidelines §18).
- Typed error classes in `lib/errors.ts` implement the status mapping table of guidelines §7 with stable machine-readable codes; success and error responses use the standard envelope helpers (guidelines §5).
- `server/cache` exposes the Redis client, helpers, and a `keys.ts` catalog seeded with the key families of folder-structure §9; no cache key is built at a call site.
- The health/readiness endpoint reports ready only when required dependencies are reachable; optional dependencies degrade rather than fail readiness (guidelines §18) — and it exercises the full spine (env → config → adapters → errors/envelope → logger) as the phase's walking skeleton.
