# E0.1 — Project Scaffold & Tooling

## Purpose

Establish the strict, gated repository baseline: a Next.js 15 + TypeScript application laid out
exactly per `rules/folder-structure.md` §2, with zero-warning lint/type/format gates and a
pre-commit gate (guidelines §4.1, §20), validated environment configuration (guidelines §8.6), and
typed connection wiring for the local Dockerized PostgreSQL/Redis/Azurite services mandated by
`CLAUDE.md`. This epic produces the ground every later feature is built on — and nothing else.

## Business Value

Every Phase 1+ feature (PRD §18) lands in a prepared seam instead of forcing structural decisions
mid-feature. Strict gates from day one mean the codebase never accumulates a backlog of type
errors, lint debt, or configuration guesswork — the cheapest moment to enforce standards is before
the first product line of code exists. Validated configuration removes the classic "works on my
machine" class of failures across the team.

## Dependencies

None — this is the first epic of the first phase. Requires only a local Docker runtime
(per `CLAUDE.md`).

## Acceptance Criteria

- A fresh clone boots the dev server using documented steps, rendering a placeholder page.
- The `src/` tree matches `rules/folder-structure.md` §2; route files contain composition only (folder-structure §3).
- TypeScript strict mode with no per-file opt-outs; lint/format pass at zero warnings; the pre-commit gate blocks violations rather than warning (guidelines §4.1, §20).
- `lib/env.ts` validates all environment variables at startup and fails fast on missing required secrets; `.env.example` documents every key by name and purpose, never value (guidelines §8.6).
- Typed `config/` modules are the only place configuration shapes are defined (folder-structure §7), and local-dev connection values target the Dockerized services on `localhost` only (`CLAUDE.md`).
- No auth, RBAC, socket, AI, or domain-module code exists anywhere in the scaffold.
