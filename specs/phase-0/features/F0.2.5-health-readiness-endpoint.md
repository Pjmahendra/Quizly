# F0.2.5 — Health & readiness endpoint

**Epic:** E0.2 — Platform Core Services · **Complexity:** Small

## Description

Expose a health/readiness endpoint that reflects real readiness (guidelines §18): the service
reports ready only when its required dependencies are reachable, and optional dependencies degrade
— not fail — readiness. It probes the three local services through their seams (PostgreSQL via
`server/db`, Redis via `server/cache`, Azurite via a minimal connectivity probe), responds using
the standard envelope (F0.2.3), and logs with correlation IDs (F0.2.2). This endpoint is the
walking-skeleton proof of the phase: the first request that traverses env → config → adapters →
typed errors/envelope → structured logs end to end. It also lays the groundwork for platform
"System Health" visibility (PRD §15).

## Dependencies

- F0.1.3 — connection wiring for the three services.
- F0.2.1 — DB connectivity probe; F0.2.4 — Redis connectivity probe.
- F0.2.2 — correlated logging; F0.2.3 — envelope and typed errors.

## Database Changes

None — the DB probe is a trivial connectivity check against the entity-less baseline schema.

## Backend Changes

- A thin route handler under `app/api/` (routes are shells — folder-structure §3) that composes per-dependency readiness probes and applies the required-vs-optional classification.
- Probe functions exposed from `server/db` and `server/cache`; a minimal Azurite connectivity probe (**Assumption:** the full `server/storage` adapter and `modules/storage` are deferred to the resource-sharing phase, PRD §9, §18 — Phase 0 ships only a connectivity check).
- Bounded probe timeouts so a slow dependency cannot hang the endpoint.

## Frontend Changes

None.

## Socket Events

None.

## AI Components

None.

## API Endpoints

- Health/readiness check — accepts no input; returns overall readiness plus per-dependency status (ready/degraded/unready), using the standard envelope; ready only when all required dependencies are reachable (guidelines §18).

## Permissions

Unauthenticated (**Assumption:** unavoidable — no auth exists in Phase 0).

## Validation Rules

- No request input is trusted or required; the endpoint is a pure read (guidelines §5 verb safety).
- The response never includes connection strings, hostnames beyond a dependency label, credentials, or stack traces (guidelines §7, §8.7).
- Per-dependency status uses stable machine-readable status codes, not free-form text (guidelines §18).

## Edge Cases & Failure Scenarios

- A required dependency down → overall unready, with the failing dependency identified by label only; details go to the server log with a correlation ID.
- An optional dependency down → overall ready-but-degraded, never a hard failure (guidelines §18).
- A probe that hangs → bounded timeout converts it to an unready/degraded result instead of a hanging request.
- A probe throwing unexpectedly → normalized through the typed-error boundary (F0.2.3); the endpoint itself never 500s with a raw error.

## Acceptance Criteria

- With all containers up, the endpoint reports ready with all dependencies healthy; stopping the required dependency flips readiness to unready; stopping an optional dependency yields degraded-but-ready (guidelines §18).
- The response uses the standard envelope, and each request produces correlated structured log entries.
- Probe timeouts are bounded and demonstrated.

## Definition of Done

Passes `rules/definition-of-done.md` and the guidelines §4.1/§20 gates at zero warnings; plus: required/optional semantics demonstrated against live containers, and docs updated per `CLAUDE.md`.
