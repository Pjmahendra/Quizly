# F0.2.4 — Redis cache core & key catalog

**Epic:** E0.2 — Platform Core Services · **Complexity:** Medium

## Description

Create the Redis cache core in `server/cache` (folder-structure §5): `redis.ts` (the single client
instance), `cache.ts` (shared helpers), and `keys.ts` — the cache-key catalog that is the only
place cache keys are built (folder-structure §9; guidelines §5 "name cache keys from a shared
catalog, never hardcode at call sites"). The catalog is seeded with the key families from
folder-structure §9: `quiz:{quizId}`, `student:{studentId}`, `leaderboard:{quizId}`,
`active-users:{quizId}`, `warnings:{attemptId}`, `socket-room:{quizId}` — exposed as typed key
builders. Redis and efficient caching are mandated by the stack and NFRs (PRD §16, §17); Phase 0
seeds the seam so later phases (sessions, leaderboards, warnings) consume it without inventing
parallel paths (guidelines §23).

## Dependencies

- F0.1.2 — Redis configuration shape from `config/redis.ts`.
- F0.1.3 — local Dockerized Redis to connect to.

## Database Changes

None.

## Backend Changes

- `server/cache/redis.ts` — single Redis client, configured only from `config/redis.ts`, module-singleton like the DB client.
- `server/cache/keys.ts` — the typed key catalog seeded with the six folder-structure §9 families; each family is a function typed by its identifier.
- `server/cache/cache.ts` — thin shared helpers (get/set/delete with TTL) with a documented convention that cached sensitive data always carries an explicit expiry (guidelines §8.7).
- A connectivity probe consumable by the health endpoint (F0.2.5).

## Validation Rules

- No cache key is constructed outside `keys.ts` (folder-structure §9, §14); key builders require their typed identifier.
- Only one Redis client instance exists, configured from `config/redis.ts`.
- Helpers require an explicit TTL decision (a value or a deliberate no-expiry) rather than an implicit default silently applied.

## Edge Cases & Failure Scenarios

- Redis unavailable — cache helpers fail gracefully and callers must be able to treat cache as degradable, never a hard dependency for correctness (PRD §16 reliability; readiness classification handled in F0.2.5).
- Reconnection after a Redis restart — the client recovers without app restart (PRD §16 "automatic reconnect" ethos).
- Dev hot reload — no client/connection leakage (same concern as F0.2.1).
- Two features wanting the same key family — the catalog is the single home; duplicates are a review-time defect (guidelines §4.6).

## Acceptance Criteria

- `server/cache` exists with `redis.ts`, `keys.ts`, `cache.ts` per folder-structure §5.
- `keys.ts` contains exactly the six seeded folder-structure §9 families as typed builders; a demonstration lookup round-trips through the helpers against local Redis.
- No cache-key string literal exists at any call site.

Passes `rules/definition-of-done.md` and the guidelines §4.1/§20 gates at zero warnings; plus: catalog seeded and verified against folder-structure §9, degradable-failure behavior demonstrated, and docs updated per `CLAUDE.md`.
