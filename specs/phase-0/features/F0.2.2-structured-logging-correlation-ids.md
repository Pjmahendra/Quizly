# F0.2.2 — Structured logging & correlation IDs

**Epic:** E0.2 — Platform Core Services · **Complexity:** Medium

## Description

Create the shared structured logger in `server/logger` — the only logging surface in the
application (folder-structure §5). Log entries are structured, never string-concatenated, and every
entry carries a module/component identifier and a correlation ID (guidelines §18). A correlation ID
is generated (or adopted) per request and propagated so that a generic client-facing error can be
tied to its detailed server-side entry (guidelines §7). Raw console output in application code is
banned (guidelines §18), and sensitive values are never logged — identifiers and codes only
(guidelines §8.7). This gives every later phase — including live-quiz reliability work (PRD §16) —
a diagnosable footing from day one.

## Backend Changes

- `server/logger/` — the shared structured logger; child/scoped loggers carry a module identifier so every entry names its origin (guidelines §18).
- Request-scoped correlation-ID plumbing for route handlers and server actions: adopt an inbound ID if present, otherwise generate one; expose it to error translation (F0.2.3).
- A gate banning `console.*` in application code (tooling-enforced), with the logger as the sanctioned alternative.

## Permissions

None — no auth exists in Phase 0. The structure anticipates security events (auth failures, denials) becoming first-class log events in Phase 1 (guidelines §18).

## Validation Rules

- Every log entry carries: level, timestamp, module identifier, correlation ID, and a stable machine-readable code for errors (guidelines §18).
- No `console.*` in application code — gate-enforced.
- No sensitive values in any entry: no credentials, tokens, personal identifiers, or payload bodies; identifiers and codes only (guidelines §8.7).

## Edge Cases & Failure Scenarios

- Logging outside a request context (boot, background) — entries still carry a module identifier; correlation ID marked absent rather than fabricated per-entry.
- Logger misconfiguration or transport failure must never crash the request path — logging is non-critical-path (guidelines §7 background-failure rule).
- Inbound correlation header malformed or hostile — treated as absent; a fresh ID is generated (never trust client input, guidelines §8.1).
- Very large objects passed to the logger — entries log identifiers, not payloads (guidelines §8.7).

## Acceptance Criteria

- All server-side logging flows through `server/logger`; the `console.*` gate fails a violating build.
- A request producing an error yields a server log entry whose correlation ID matches the one surfaced by the response envelope (with F0.2.3/F0.2.5 as the demonstration path).
- Entries are structured with module identifiers; no sensitive values appear in logs.
