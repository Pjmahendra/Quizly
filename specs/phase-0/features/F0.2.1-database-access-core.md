# F0.2.1 — Database access core

**Epic:** E0.2 — Platform Core Services · **Complexity:** Medium

## Description

Create the single pooled Prisma client in `server/db` (folder-structure §5) — the only place a
database client is ever instantiated, with managed connection pooling (guidelines §6). Establish
the repository conventions that all future domain modules implement (folder-structure §8): data
access flows page/action → service (`modules/*/services`) → repository (`modules/*/repositories`)
→ Prisma (`server/db`), and repositories are the only Prisma call sites — keeping the ORM
replaceable and giving tenant scoping a single enforcement point per domain (a foundational concern
per `CLAUDE.md` multi-tenancy rules and PRD §16 security).

## Dependencies

- F0.1.2 — database configuration shape from `config/database.ts`.
- F0.1.3 — local Dockerized PostgreSQL to connect to.
- F0.2.3 (soft) — repository error-translation convention targets the typed error classes.

## Database Changes

Conceptual only: a baseline Prisma schema pointing at the local PostgreSQL exists so the client can
be generated, but it contains **no domain entities** — product entities begin in Phase 1 (PRD §18).

## Backend Changes

- `server/db/` — the pooled Prisma client as a module singleton; nothing else instantiates a client (folder-structure §5; guidelines §6).
- `prisma/` folder seams per folder-structure §8 (`schema.prisma`, empty `migrations/`, placeholder `seed.ts`, `factories/`, `fixtures/`).
- Repository base conventions documented in `docs/architecture.md` (per `CLAUDE.md`): naming, use of the shared client, translation of persistence errors to typed transport errors (guidelines §6), and the service → repository → Prisma flow (folder-structure §8).
- Enforcement of "no Prisma outside repositories/`server/db`" via an import-boundary gate.

## Validation Rules

- Exactly one Prisma client instance exists at runtime, configured only from `config/database.ts`.
- No import of the Prisma client or generated types outside `server/db` and (future) `modules/*/repositories`.
- Raw driver/ORM errors never cross a repository boundary untranslated (guidelines §6).

## Edge Cases & Failure Scenarios

- Dev-server hot reload re-evaluating modules — must not leak additional clients/connections (singleton survives reload).
- Database unreachable at boot — the app starts but readiness fails (F0.2.5); no crash loop and no secret leakage in the error output.
- Connection-pool exhaustion — pool limits come from config, with behavior documented.
- Concurrent local tooling (migrations, seed) sharing the same database — conventions note isolation expectations.

## Acceptance Criteria

- The app connects to the Dockerized PostgreSQL through the single `server/db` client; a trivial connectivity check succeeds (consumed by F0.2.5).
- The import-boundary gate fails when Prisma is referenced outside its allowed homes.
- Repository conventions are documented in `docs/architecture.md` and reference the folder-structure §8 flow.
- The Prisma schema contains no domain entities.

## Definition of Done

Passes `rules/definition-of-done.md` and the guidelines §4.1/§20 gates at zero warnings; plus: single-client + boundary enforcement demonstrated, conventions documented, and a `docs/logs/` entry created per `CLAUDE.md`.
