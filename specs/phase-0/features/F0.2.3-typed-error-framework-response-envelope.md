# F0.2.3 — Typed error framework & response envelope

**Epic:** E0.2 — Platform Core Services · **Complexity:** Medium

## Description

Create the typed error foundation in `lib/errors.ts` (folder-structure §7, §9): domain error base
classes mapped to transport statuses per the guidelines §7 table — not-found 404, forbidden 403,
invalid input/business rule 400, conflict 409, missing/invalid credentials 401, valid input but
invalid state 422 — with every error carrying a stable machine-readable code as part of the
contract. Alongside it, standard success/error response envelope helpers per guidelines §5: one
consistent envelope for success and metadata, errors signaled through status codes plus a
structured error body, and standardized helpers (created, no-content, paginated) so no endpoint
ever hand-rolls the envelope. Client errors stay generic; server logs stay detailed; the two are
linked by the correlation ID (guidelines §7; F0.2.2).

## Validation Rules

- Every error crossing a transport boundary is a typed error with a stable code; a bare generic error surfacing as a 500 is a defect (guidelines §7).
- Client-facing errors never contain stack traces, raw queries, paths, env values, or secret material (guidelines §7).
- Success and error responses always use the shared envelope helpers — no hand-rolled shapes (guidelines §5).

## Edge Cases & Failure Scenarios

- Non-`Error` values thrown (strings, objects) — normalized into the generic server fault path.
- An error thrown while translating/serializing an error — a minimal hardcoded fallback response guarantees the boundary never hangs or leaks.
- Double-wrapping (a typed error wrapped in another) — translation unwraps to the most specific typed error.
- Persistence errors reaching the boundary raw — must already be translated at the repository seam (guidelines §6; F0.2.1).

## Acceptance Criteria

- The error base classes cover the full guidelines §7 status table, each with a stable, documented code.
- The envelope helpers exist and produce consistent success/error shapes; F0.2.5 demonstrates them live.
- An unknown thrown error results in a generic client response plus a detailed, correlated server log entry.
