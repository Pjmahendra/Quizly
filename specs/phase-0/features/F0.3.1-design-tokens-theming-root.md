# F0.3.1 — Design tokens & theming root

**Epic:** E0.3 — Design System Foundation · **Complexity:** Medium

## Description

Encode the visual language of `rules/design-specification.html` ("Quizly — Architectural Design
System 2026") as named design tokens — the only styling primitive (guidelines §12): a brand color
ramp plus semantic roles (primary, success, warning, danger, info, muted, surface, on-surface) and
a neutral scale; a small fixed typography scale; a single spacing base unit; radius; elevation;
motion easing/duration tokens; and a z-index scale. Light and dark themes are produced through
token overrides at a single root — never by forking components (guidelines §12). Tokens are wired
in `src/styles/` (folder-structure §2) and consumed through Tailwind CSS (PRD §17), so no feature
code ever hardcodes a hex value, raw pixel size, palette class, or numeric z-index.

## Validation Rules

- Every color usage goes through a semantic role, never a raw ramp value in feature code (guidelines §12).
- Every themable token defines both a light and a dark value — a missing dark value is a defect, not a fallback.
- Text/UI contrast meets WCAG 2.1 AA in both themes: 4.5:1 body text, 3:1 large text and meaningful UI components (guidelines §14).
- No raw hex, pixel size, or numeric z-index in component code — enforced by review and, where feasible, a style gate.

## Edge Cases & Failure Scenarios

- First paint before theme resolution — no flash of the wrong theme; the resolved theme must be applied before content renders.
- OS theme preference changing while the app is open — the root responds without reload when following system preference.
- A token referenced before it is defined (typo/missing) — must fail visibly in development rather than silently rendering defaults.
- Reduced-motion users — motion tokens resolve to non-animated equivalents (guidelines §14).

## Acceptance Criteria

- All token categories exist, named per the design specification, and drive Tailwind-consumed styling.
- Toggling the root theme switches every tokenized surface between light and dark with no component-level forks (guidelines §12).
- Contrast checks pass in both themes (guidelines §14).
- A spot check of the scaffold UI shows zero hardcoded style values.

## Definition of Done

Passes `rules/definition-of-done.md` (esp. §6 Layout & Visual Hierarchy, §12 Accessibility Experience, §17 Quality Gates) and the guidelines §4.1/§20 gates at zero warnings; plus: tokens traceable to `rules/design-specification.html`, both themes verified, and docs updated per `CLAUDE.md`.
