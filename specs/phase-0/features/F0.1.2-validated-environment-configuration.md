# F0.1.2 — Validated environment configuration

**Epic:** E0.1 — Project Scaffold & Tooling · **Complexity:** Small

## Description

Create `lib/env.ts` as the single validated entry point for environment variables: every variable
is parsed against a Zod schema (PRD §17) at startup, with types derived from the validated shape so
inference and runtime agree (guidelines §4.1). The application fails fast on a missing or malformed
required secret (guidelines §8.6). Typed `config/` modules (folder-structure §7) consume the
validated env and are the only place configuration shapes are defined. A committed `.env.example`
documents every key's name and purpose — never a value (guidelines §8.6).

## Backend Changes

- `lib/env.ts` — schema-validated environment access; the only module that reads raw process env.
- `config/database.ts`, `config/redis.ts`, `config/storage.ts` — typed config shapes for the Phase 0 services (folder-structure §7). Other config modules (`auth`, `mail`, `socket`, `ai`, `security`) are deferred to the phases that consume them, to avoid over-building.
- Startup fail-fast path: boot aborts with the offending key's name (never its value) when a required variable is missing or invalid.
- `.env.example` at the repo root documenting each key: name, purpose, required/optional.

## Validation Rules

- Every environment variable used anywhere must be declared in the `lib/env.ts` schema; direct raw env reads elsewhere are a defect.
- Required keys missing/invalid → startup failure naming the key, never the value (guidelines §8.6).
- Structured values (URLs, ports) are validated for shape at boot, not at first use.
- `.env.example` lists names and purposes only — committing a real value anywhere is a violation (guidelines §8.6).

## Edge Cases & Failure Scenarios

- Optional key absent — a documented default applies; the default is defined in the schema, not scattered at call sites.
- Lint/type/format gates and builds that run without a populated env (e.g. CI) must not require real secrets to pass.
- Error output during fail-fast must never echo secret values (guidelines §8.7).
- The same key needed with different values per environment — the schema stays environment-agnostic; values differ per env file.

## Acceptance Criteria

- Booting without a required key fails immediately with a clear message naming the key.
- All configuration shapes live in `config/` modules typed from validated env (folder-structure §7); no other module defines config shapes.
- `.env.example` covers every declared key with name and purpose, and contains no values.
- No raw process-env access exists outside `lib/env.ts`.
