# F5.1.2 — Resource Upload & Management

**Epic:** E5.1 — File Storage & Resources · **Complexity:** Large

## Description

Faculty upload learning resources to their classrooms and manage them over time (PRD §9): PDF
documents, PowerPoint presentations, images, Word documents, external links, course notes, and
recorded lectures. Faculty can replace a resource's file and delete resources. External links are
resources without blobs. **Assumption:** course notes and recorded lectures are plain file uploads
(no note editor, no streaming pipeline — see phase exclusions). Domain home is
`modules/resources/` (folder-structure §4), storing blobs via `modules/storage` under
`organizations/{orgId}/classrooms/{classroomId}/resources/` (folder-structure §10).

## Dependencies

- F5.1.1 (storage provider + path catalog).
- Phase 1 (permissions, audit platform); Phase 2 (classrooms, faculty assignment).
- E5.4 consumes this feature's "resource added" domain event for notifications (F5.4.2).

## Database Changes (conceptual)

A classroom resource entity: org- and classroom-scoped, with type (document kind vs. external
link), title/description, blob reference or URL, original filename, size, content type, uploader,
timestamps. Soft-delete only if the audit requirement justifies retention (guidelines §6) —
otherwise hard delete with blob cleanup. A bookmark relation arrives in F5.1.3.

## Backend Changes

- `modules/resources/`: actions (I/O tier), services (authorization + orchestration),
  repositories (only Prisma call sites), validators (Zod), dto, `permissions.ts`, `constants.ts`.
- Upload flow: validate → authorize (faculty of this classroom) → store blob via `modules/storage`
  → persist metadata → emit domain event → audit log (guidelines §8.8).
- Replace: new blob then metadata swap then old-blob cleanup; delete: metadata removal then blob
  deletion, tolerant of partial failure (orphaned blobs reconciled, never orphaned records).
- Quota seam: a single service-level check point reserved for per-org storage quotas.

## Frontend Changes

- Faculty resource manager in `modules/resources/components/` composed into `app/classroom/`:
  upload (with progress), list with pagination/search, replace, delete with confirmation dialog
  (shared dialog/table primitives from `src/components/`, guidelines §13).

## Socket Events

- `resource-added` (existing catalog event, folder-structure §5) emitted to the classroom's room
  on successful upload; live consumption behavior is specified in F5.1.3.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Create/upload a classroom resource (file or external link).
- Update resource metadata; replace a resource's file.
- Delete a resource.
- List a classroom's resources (paginated, filter by type) — shared with F5.1.3.

## Permissions

- Registered via `modules/resources/permissions.ts` into `server/permissions/`: resource create,
  update, replace, delete — granted to faculty for classrooms they are assigned to; org admins per
  role templates (PRD §4.2–4.3; guidelines §9). Frontend gates mirror server guards.

## Validation Rules

- Content-type **allow-list** matching PRD §9 types, extension/MIME pairing checks, and bounded
  file size (guidelines §8.4). **Needs clarification:** size/type limits and per-org quotas are
  unspecified in the PRD — ship conservative configurable defaults.
- Filename sanitized for display only — never used in the blob path (F5.1.1).
- External links validated as http(s) URLs; titles/descriptions length-bounded.
- **Needs clarification:** malware scanning — no scanning service exists in the stack (PRD §17).

## Edge Cases & Failure Scenarios

- Blob stored but metadata write fails → compensating blob delete; no ghost files.
- Delete while a student is mid-download; replace mid-download → old blob served to completion or
  a clean failure, never a corrupt mix.
- Duplicate titles/filenames in one classroom; upload interrupted by disconnect (resumable is out
  of scope — clean retry).
- Classroom archived/deleted → resource lifecycle follows the classroom (Phase 2 semantics).

## Acceptance Criteria

- Faculty of a classroom can upload every PRD §9 resource type, replace files, and delete; other
  faculty/students cannot mutate (guidelines §8.2).
- All mutations are audit-logged and every list paginates (guidelines §5, §8.8).
- Invalid type/oversized uploads are rejected server-side with typed errors (guidelines §8.4).

## Definition of Done

Meets `rules/definition-of-done.md`; documentation per CLAUDE.md (`docs/logs/` entry; architecture
doc updated with the resources module); quota clarification tracked before GA.
