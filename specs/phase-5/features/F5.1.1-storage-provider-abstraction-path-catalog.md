# F5.1.1 — Storage Provider Abstraction & Path Catalog

**Epic:** E5.1 — File Storage & Resources · **Complexity:** Medium

## Description

The storage platform every file feature builds on (PRD §9 "Storage Strategy"). A provider
interface in `modules/storage/providers/` runs against Azurite today and Azure Blob Storage later
with no caller changes; the low-level SDK client lives in `server/storage/` (folder-structure §5,
§10). `modules/storage/paths.ts` is the **only** code that builds a blob path, implementing the
org-scoped hierarchy: `organizations/{orgId}/` → `faculty/{facultyId}/`,
`classrooms/{classroomId}/`, `quizzes/{quizId}/`, `students/{studentId}/` with their sub-nodes
(folder-structure §10). Upload, download, and delete operations (`upload.ts`, `download.ts`,
`delete.ts`) plus orchestration in `services/` complete the module.

## Dependencies

- Phase 1 tenant scoping context (every path is rooted at an `orgId` from server-side context).
- Azurite in local Docker (CLAUDE.md); `config/storage.ts` for typed, env-derived configuration
  (folder-structure §7).
- No dependency on Phases 2–4 for the platform itself; consumers arrive in F5.1.2+.

## Database Changes (conceptual)

None. This feature is infrastructure; blob metadata records belong to consuming features
(F5.1.2). The blob store itself is not a database and holds no authoritative domain state.

## Backend Changes

- `server/storage/`: Azurite/Azure Blob client instantiation — the single connection point.
- `modules/storage/providers/`: provider interface + Azurite implementation (put, get/stream,
  delete, exists); Azure Blob arrives later as a second implementation of the same interface.
- `modules/storage/paths.ts`: typed path builders per hierarchy node; accepts only trusted ids.
- `modules/storage/services/` + `upload.ts` / `download.ts` / `delete.ts`; typed storage errors
  extending `lib/errors.ts` (folder-structure §9).
- Container/root bootstrap on startup for local development.

## Frontend Changes

None. This module has no UI; consuming features own their surfaces.

## Socket Events

None.

## AI Components

None this phase.

## API Endpoints (purpose-level)

None public. The module is a server-side seam consumed by other modules via its `index.ts`
(folder-structure §4); F5.1.2/F5.1.3 expose the user-facing endpoints.

## Permissions

None of its own. Authorization is enforced by calling modules before any storage operation; the
storage module never decides who may access a blob (it has no domain context to do so).

## Validation Rules

- Path builders accept only server-side trusted ids — never client-supplied strings; no path
  separators, `..`, or encoding tricks can enter a path (path-traversal defense, guidelines §8.4).
- Every upload is size-bounded at this seam as defense-in-depth; per-type limits are the calling
  feature's policy (F5.1.2).
- Provider configuration validated at startup via `lib/env.ts` / `config/storage.ts`.

## Edge Cases & Failure Scenarios

- Azurite unreachable or container missing → typed storage error mapped to a graceful failure,
  never a raw driver error (guidelines §6, §7).
- Partial upload failure → no orphan metadata: callers write metadata only after a confirmed put.
- Delete of a missing blob → idempotent success (guidelines §4.5).
- Oversized stream despite client checks → aborted server-side at the bound.

## Acceptance Criteria

- Grepping the codebase finds no blob-path construction outside `paths.ts` and no storage SDK
  usage outside `server/storage/` + `modules/storage/providers/` (folder-structure §9–10).
- Swapping the provider implementation requires zero changes in any caller (PRD §9).
- All storage failures surface as typed errors with the shared logger recording context
  (folder-structure §5).
