# F5.2.1 — Classroom Announcements

**Epic:** E5.2 — Classroom Communication · **Complexity:** Medium

## Description

Faculty post announcements inside their classrooms and every enrolled student receives them
(PRD §5 — "Students enrolled in the classroom automatically receive updates"). Announcements
appear on a classroom surface (newest first, paginated) and produce the "Classroom Announcement"
notification (PRD §14) once E5.4 is live. Announcements are classroom domain: they live in
`modules/classrooms/` (folder-structure §4). Optional file attachments reuse the storage platform
under `organizations/{orgId}/classrooms/{classroomId}/announcements/` (folder-structure §10).

## Dependencies

- Phase 1 (permissions, audit), Phase 2 (classrooms, enrollment, faculty assignment).
- F5.1.1/F5.1.2 for attachments (**Assumption:** attachments are in scope because the path
  catalog reserves an `announcements/` node; PRD §5 does not name them explicitly).
- F5.4.1/F5.4.2 for notification delivery — the in-classroom surface ships independently first.

## Database Changes (conceptual)

An announcement entity: org- and classroom-scoped, author (faculty), title/body, optional
attachment references, created/edited timestamps. Body is rich-text-lite or plain text —
**Needs clarification:** formatting expectations are unspecified. Lifecycle: announcements persist
with the classroom; a retention policy is defined with the classroom's lifecycle (guidelines §6).

## Backend Changes

- `modules/classrooms/`: announcement actions/services/repositories/validators/dto additions;
  authorization = faculty assigned to the classroom (services layer).
- Post/edit/delete mutations audit-logged (guidelines §8.8).
- On post: emit a domain event consumed by F5.4.2 for fan-out — the announcement service never
  fans out itself (single fan-out path, guidelines §10).

## Frontend Changes

- Announcement composer + feed in `modules/classrooms/components/`, composed into
  `app/classroom/`: faculty create/edit/delete; students read-only, newest first, paginated,
  with shared empty state (guidelines §13).

## Socket Events

None of its own. Live delivery to students rides the notification platform's
**Proposed addition to catalog:** `notification-created` (defined in F5.4.1) rather than a
dedicated announcement event — one delivery path, no new catalog entry here.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Create an announcement (optionally with attachments); edit; delete.
- List a classroom's announcements (paginated) for enrolled members.

## Permissions

- Registered via `modules/classrooms/permissions.ts`: announcement create/edit/delete for the
  classroom's assigned faculty (and org admin per role templates); read for enrolled students and
  classroom faculty (guidelines §9). Frontend gates mirror the guards.

## Validation Rules

- Title/body required with bounded lengths; attachments follow F5.1.2's allow-list and size
  bounds (guidelines §8.4).
- Body sanitized/encoded for output — XSS defense if any formatting is allowed (guidelines §8.4).
- Classroom scope resolved from server-side context, never client input (guidelines §1.1).

## Edge Cases & Failure Scenarios

- Announcement posted to a classroom with zero enrolled students → succeeds; no fan-out work.
- Edit/delete after students have already been notified → surface shows current state;
  notifications are not retracted (**Assumption**; retraction is unspecified).
- Attachment upload fails mid-post → announcement not published half-formed (one transaction for
  metadata, compensation for blobs — guidelines §6).
- Very large classrooms → fan-out cost belongs to F5.4.2's background path, not this request.

## Acceptance Criteria

- Assigned faculty can post/edit/delete; enrolled students see announcements newest-first with
  pagination; non-members see nothing (PRD §5; guidelines §8.2).
- Posting produces the "Classroom Announcement" notification for all enrolled students once E5.4
  is live (PRD §14).
- Mutations are audit-logged; denial and empty states use distinct copy (guidelines §8.8, §13).
