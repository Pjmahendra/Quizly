# F5.3.3 — Discussion Shared Resources

**Epic:** E5.3 — Discussion Rooms · **Complexity:** Medium

## Description

Room members share resources with each other (PRD §6 lists "Shared Resources"): upload files into
the room and post external links, reusing the E5.1 storage platform end-to-end — provider,
validation policy, and path catalog. Members view and download what was shared, gated by room
membership. The blob hierarchy in folder-structure §10 has no discussions node, so this feature
**extends the path catalog** with `organizations/{orgId}/discussions/{roomId}/resources/` — added
in `modules/storage/paths.ts` only, the single home for paths (folder-structure §9).

## Dependencies

- F5.1.1 (storage platform + path catalog) and F5.1.2 (upload validation policy to reuse).
- F5.3.1 (rooms and membership — the access boundary).
- **Assumption:** room resources are a lightweight share (file/link + title), not the full
  classroom resource entity with replace/bookmark semantics — PRD §6 gives no management detail.

## Database Changes (conceptual)

A room-resource entity: org- and room-scoped, sharer, type (file or external link), title, blob
reference or URL, size/content type, created timestamp. Deleting follows room lifecycle (room
archive → read-only; room deletion → content deletion per F5.3.1's policy, guidelines §6).

## Backend Changes

- `modules/discussions/` resource services/repositories/validators: share (upload via
  `modules/storage`), list, download brokering with membership check, delete (sharer or room
  owner).
- `modules/storage/paths.ts`: the new discussions path node — no other file touches paths.
- Domain event on share, consumed by F5.4.2 (**Assumption:** room shares notify room members
  under the "Resource Added" trigger family; PRD §14 does not distinguish classroom vs. room —
  **Needs clarification**).

## Frontend Changes

- Shared-resources panel in the room page (`modules/discussions/components/`): upload/link
  composer, paginated list with sharer and time, download, delete for sharer/owner; shared
  empty-state and dialog primitives (guidelines §13).

## Socket Events

- `resource-added` (existing catalog event, folder-structure §5) emitted to the room's socket
  room on a successful share, so an open room updates live — same event the classroom surface
  uses, scoped to a different room.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Share a resource into a room (file upload or external link).
- List a room's shared resources (paginated); download a shared file.
- Delete a shared resource (sharer or room owner).

## Permissions

- Share/view/download for room members; delete for the sharer and room owner — registered via
  `modules/discussions/permissions.ts` (guidelines §9). No access for non-members, including
  uninvited faculty (PRD §6).

## Validation Rules

- Identical file policy to F5.1.2: content-type allow-list, extension/MIME pairing, bounded size
  (guidelines §8.4) — one policy, defined once and imported, not duplicated.
- External links validated as http(s) URLs; titles length-bounded.
- Room scope and membership resolved server-side (guidelines §1.1, §8.2).
- Room shares count toward the org storage quota seam reserved in F5.1.2
  (**Needs clarification:** quotas unspecified).

## Edge Cases & Failure Scenarios

- Blob stored but record write fails → compensating delete (no orphan blobs).
- Sharer leaves or is removed from the room → shared items remain (**Assumption:** content
  belongs to the room once shared).
- Download after room archive → allowed (read-only); after room deletion → gone with the room.
- Member removed mid-download → current stream completes; new requests denied.

## Acceptance Criteria

- Members can share files and links, see the shared list live (`resource-added`), and download;
  non-members cannot reach the list or any blob (PRD §6; guidelines §8.2).
- All room-resource blobs live under the discussions path node built exclusively by `paths.ts`
  (folder-structure §9–10).
- Lists paginate; invalid uploads are rejected server-side with typed errors (guidelines §5, §8.4).
