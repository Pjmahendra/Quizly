# F5.2.2 — Classroom Quiz History

**Epic:** E5.2 — Classroom Communication · **Complexity:** Small

## Description

A quiz-history surface inside the classroom (PRD §5 lists "Quiz History" as classroom content):
members see the classroom's past and scheduled quizzes in one place — title, schedule/completion
timing, and status — with role-appropriate depth (faculty see management context; students see
their own participation/results context). This is a **read-only composition surface**: the
classroom module renders data owned by the quizzes domain, imported only via `modules/quizzes`'
public `index.ts` (folder-structure §4). No quiz logic is built here.

## Dependencies

- Phase 2 (classrooms, enrollment).
- Phases 3–4 (quiz creation/scheduling/execution/results) own the underlying data. This is the
  one E5.1–E5.3 feature that touches quiz phases: the surface is built against the quizzes
  module's public read interface and renders the shared empty state until quiz data exists, so it
  does not block parallel delivery of this phase.

## Database Changes (conceptual)

None. All data is owned by the quizzes/attempts domains; this feature adds no entities.
(If the quizzes module lacks a classroom-scoped read query, that query is added inside
`modules/quizzes/` — not here.)

## Backend Changes

- A classroom-scoped, viewer-aware quiz-history read exposed by `modules/quizzes` (service +
  repository there), consumed by `modules/classrooms/` composition (folder-structure §4, §8).
- Authorization: enrollment/assignment to the classroom, plus the quizzes domain's own visibility
  rules (e.g., students never see unpublished drafts — Phases 3–4 semantics govern).

## Frontend Changes

- Quiz-history list in `modules/classrooms/components/` composed into `app/classroom/`: paginated
  table (shared table primitives), status indicators, links into quiz/result surfaces owned by
  the quiz phases, shared empty state before data exists (guidelines §11, §13).

## Socket Events

None. History is not a live surface this phase; live quiz state belongs to the quiz phases.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List a classroom's quiz history for the current viewer (paginated, role-aware) — served by the
  quizzes module's read interface.

## Permissions

- No new capabilities. Reuses classroom membership (Phase 2) plus quiz visibility capabilities
  registered by `modules/quizzes/permissions.ts` (guidelines §9); this surface introduces no
  permission strings of its own.

## Validation Rules

- Classroom scope from trusted server context (guidelines §1.1); pagination parameters bounded
  with sane defaults and a hard maximum (guidelines §5).

## Edge Cases & Failure Scenarios

- Quiz phases not yet delivered → the surface renders the shared empty state, not an error.
- Quiz deleted or unpublished after appearing in history → respects the quiz domain's visibility
  rules on next load; links degrade to not-found handled with typed errors (guidelines §7).
- Student enrolled after past quizzes ran → sees history per quiz-domain visibility rules
  (**Needs clarification:** whether pre-enrollment quizzes are visible is a quiz-phase decision).

## Acceptance Criteria

- Enrolled students and assigned faculty see the classroom's quiz history inside the classroom,
  paginated, with role-appropriate fields (PRD §5).
- No quiz business logic or Prisma access exists in `modules/classrooms/` for this surface —
  everything flows through `modules/quizzes`' public interface (folder-structure §4, §14).
- Empty state shown before quiz phases deliver data; non-members get a permission denial
  (guidelines §8.2, §13).

## Definition of Done

Meets `rules/definition-of-done.md`; documentation per CLAUDE.md (feature log entry); composition
boundary (no deep imports) verified in review.

## Open Questions

- Which exact fields per role (does a student see class-wide stats or only their own attempt
  status)? Depends on Phases 3–4 result-visibility decisions.
- Does "history" include upcoming scheduled quizzes or only completed ones? PRD §5 is silent —
  **Assumption:** both, clearly grouped.
