# F5.1.3 — Resource Consumption

**Epic:** E5.1 — File Storage & Resources · **Complexity:** Medium

## Description

The student side of resources (PRD §9): enrolled students view a classroom's resources, download
files (or follow external links), and bookmark resources for quick retrieval across classrooms.
Where a live classroom surface is open, newly added resources appear without refresh via the
catalog `resource-added` event (folder-structure §5). Domain home remains `modules/resources/`;
downloads are brokered through `modules/storage` so enrollment is checked on every fetch.

## Dependencies

- F5.1.1 (storage platform) and F5.1.2 (resource entity and lists).
- Phase 2 enrollment (the access boundary); socket infrastructure for the live update path.

## Database Changes (conceptual)

A bookmark relation between a user and a resource (org-scoped, unique per user+resource, with a
created timestamp). Optionally a download-count on the resource for faculty insight —
**Assumption:** simple counter only; per-student download tracking is not required by the PRD.

## Backend Changes

- `modules/resources/` read services: enrollment-authorized resource listing, resource detail,
  download brokering (stream via provider; content-type and content-disposition set server-side).
- Bookmark services: add/remove bookmark; list "my bookmarks" across classrooms.
- Repository-level tenant + enrollment scoping as the single enforcement seam
  (folder-structure §8; guidelines §8.2).

## Frontend Changes

- Student classroom resource view in `modules/resources/components/` composed into
  `app/classroom/` and `app/student/`: paginated list with type filter and search, download
  action, bookmark toggle, and a "My bookmarks" surface in the student portal.
- Live insert of new resources when `resource-added` arrives on an open classroom surface, using
  the shared feedback primitives (`src/components/feedback/`).

## Socket Events

- `resource-added` (existing catalog event, folder-structure §5): consumed by open classroom
  surfaces; clients join the classroom room via `server/socket/` room primitives, event name
  imported from `modules/socket/events/` — never a string literal (folder-structure §5).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List classroom resources for an enrolled student (paginated, filtered) — shared with F5.1.2.
- Download/stream a resource file; resolve an external-link resource.
- Add/remove a bookmark; list the current user's bookmarks (paginated).

## Permissions

- Resource view/download and bookmark capabilities registered via
  `modules/resources/permissions.ts`, granted to students for classrooms they are enrolled in and
  to the classroom's faculty (guidelines §9). No public/anonymous access to any blob.

## Validation Rules

- Resource id validated and resolved within the caller's org and enrollment scope from server
  context — never from client-supplied org/classroom ids (guidelines §1.1, §8.2).
- Bookmark operations idempotent (double-add / double-remove are safe, guidelines §4.5).
- Download responses set safe headers (no content-sniffing execution; attachment disposition for
  document types) per guidelines §8.4–8.5.

## Edge Cases & Failure Scenarios

- Resource deleted or replaced between list and download → typed not-found/refreshed metadata,
  never a raw storage error (guidelines §7).
- Student unenrolled from the classroom → resource and bookmark access ends; bookmarks to
  now-inaccessible resources render as inert entries or are pruned (**Assumption:** shown as
  unavailable, not silently deleted).
- Blob missing for an existing record (orphaned metadata) → graceful failure + logged for
  reconciliation.
- `resource-added` arriving for a surface the user has scrolled/paginated → no duplicate rows.

## Acceptance Criteria

- An enrolled student can view, download, and bookmark resources; a non-member gets a permission
  denial distinct from a load failure (PRD §9; guidelines §9, §13).
- New uploads appear live on open classroom surfaces via the catalog event (folder-structure §5).
- All lists paginate with totals (guidelines §5); downloads stream without buffering whole files
  in memory (guidelines §10).

## Definition of Done

Meets `rules/definition-of-done.md`; documentation per CLAUDE.md (feature log entry; student-portal
docs updated); accessibility of list/download/bookmark controls per guidelines §14.
