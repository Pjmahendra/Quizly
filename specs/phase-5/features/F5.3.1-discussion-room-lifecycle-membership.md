# F5.3.1 — Discussion Room Lifecycle & Membership

**Epic:** E5.3 — Discussion Rooms · **Complexity:** Large

## Description

Students independently create discussion rooms for group study, topic discussions, peer learning,
and collaborative preparation (PRD §6). The creator invites students from their organization;
invitees accept or decline; members can leave; the creator manages the room (rename, remove
members, archive/delete). Faculty are not mandatory participants but **may join if invited**
(PRD §6) — they can never enter uninvited. Membership is the sole access boundary for everything
in E5.3. Domain home is `modules/discussions/` (folder-structure §4).

## Dependencies

- Phase 1 (tenancy, permissions — rooms are org-scoped; invitees must be same-org users).
- Phase 2 (the student population and portal surfaces). No dependency on Phases 3–4.
- F5.4.1 for invite visibility (**Assumption:** invitations surface in the notification center —
  PRD §6 names "Invite Students" but not the delivery mechanism); a plain invites list in the
  room UI works before E5.4 lands.

## Database Changes (conceptual)

A discussion-room entity (org-scoped, creator, name/topic, status active/archived, timestamps)
and a membership relation (user, role member/owner, joined-at) with an invitation state
(pending/accepted/declined) — one row per user per room. Faculty membership is the same relation;
their portal role distinguishes them. Lifecycle per guidelines §6: archived rooms are read-only;
deletion policy for rooms and their content defined here.

## Backend Changes

- `modules/discussions/`: actions/services/repositories/validators/dto/`permissions.ts`.
- Services: create room, invite (same-org users only), accept/decline, leave, remove member,
  rename, archive, delete — all authorization at the service layer, tenant scope at the
  repository seam (folder-structure §8).
- Domain events for invites (consumed by E5.4) and membership changes (consumed by F5.3.2's
  socket-room access checks).

## Frontend Changes

- Room list ("My rooms" + pending invites), create dialog, invite picker (searching same-org
  students, optionally faculty), member management, archive/delete confirmations — in
  `modules/discussions/components/`, composed into `app/student/` (faculty reach invited rooms
  from `app/faculty/`). Shared dialog/table/empty-state primitives (guidelines §13).

## Socket Events

None of its own. Membership gates who may join the room's socket room (F5.3.2); membership-change
push to live clients is **not** required this phase (presence is excluded — see phase exclusions).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Create / rename / archive / delete a room.
- Invite users; accept or decline an invitation; list my invitations.
- List my rooms (paginated); list a room's members; leave; remove a member (owner only).

## Permissions

- Registered via `modules/discussions/permissions.ts`: room create (students; **Needs
  clarification:** whether faculty/admins may also create — PRD §6 says "students can
  independently create"), invite/manage (room owner), participate (members). All checks are
  membership-based capabilities, never role-name matching (guidelines §9).

## Validation Rules

- Room name/topic required, length-bounded; invitees must exist in the caller's org (server-side
  resolution — guidelines §1.1, §8.2).
- Invitation operations idempotent: re-invite of a pending invitee, double-accept, and leave of a
  non-member are safe no-ops (guidelines §4.5).
- **Needs clarification:** maximum members per room and rooms per student are unspecified — ship
  configurable conservative bounds.

## Edge Cases & Failure Scenarios

- Owner leaves or their account is deactivated → ownership transfer or archive policy required
  (**Needs clarification** — no PRD guidance; **Assumption:** room archives if the sole owner
  departs).
- Invitee deactivated before accepting → invitation invalidated.
- Concurrent remove + accept; invite to an archived room → rejected with typed conflict errors.
- Cross-org invitation attempts → impossible by construction at the repository seam.

## Acceptance Criteria

- A student can create a room, invite peers (and optionally faculty), and manage membership;
  invitees can accept/decline; members can leave (PRD §6).
- Faculty can participate only in rooms they were invited to; no uninvited access by any route
  (PRD §6; guidelines §8.2).
- Room and invite lists paginate; membership mutations are audit-visible where the room is viewed
  (guidelines §5, §8.8).
