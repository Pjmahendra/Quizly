# F5.3.2 — Discussion Chat & Threads

**Epic:** E5.3 — Discussion Rooms · **Complexity:** Large

## Description

Real-time conversation inside a discussion room (PRD §6 lists "Chat" and "Discussion Threads" as
distinct features). **Assumption:** the room has one main chat stream, and members can additionally
create named discussion threads — topic-scoped message streams within the room — since the PRD
does not define the chat/thread relationship. Messages deliver live over Socket.IO via the
existing catalog event `discussion-message` (folder-structure §5); history is always paginated
(guidelines §5) and message data has a defined lifecycle (guidelines §6). Thread replies feed the
"Discussion Replies" notification trigger (PRD §14) via F5.4.2.

## Dependencies

- F5.3.1 (rooms and membership — the access boundary for every message).
- `server/socket/` infrastructure (bootstrap, auth middleware, room primitives) and
  `modules/socket/` handlers/event catalog (folder-structure §5).
- F5.4.2 consumes reply domain events for notifications; chat works before E5.4 lands.

## Database Changes (conceptual)

A message entity (org- and room-scoped, author, body, timestamps, optional parent container) and a
thread entity (room-scoped, title, creator, timestamps). Messages belong to either the room's main
stream or a thread. Indexed for reverse-chronological paging (guidelines §6). Retention/archival
policy for messages defined with the room lifecycle — **Needs clarification:** no retention
requirement exists in the PRD.

## Backend Changes

- `modules/discussions/`: message/thread services and repositories; persistence-first flow —
  validate → authorize membership → persist → emit socket event → emit domain event for
  notifications.
- `modules/socket/handlers/`: a discussions handler joining members to the room's socket room
  (named via the module's room-naming scheme) and broadcasting `discussion-message`; membership
  checked at join and on every send (guidelines §8.2).

## Frontend Changes

- Chat surface in `modules/discussions/components/` composed into the room page: main stream,
  thread list, thread view, composer; infinite scroll backed by paginated history; optimistic send
  with failure rollback; reconnect recovers missed messages via history fetch (guidelines §13, §15).

## Socket Events

- `discussion-message` (existing catalog event, folder-structure §5): server → room members on
  each new message/thread reply; name imported from `modules/socket/events/`, never a literal.
- No typing/presence/read-receipt events — excluded this phase (see phase exclusions).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Send a message (main stream or thread); create a thread.
- Fetch paginated message history (main stream; per thread); list a room's threads (paginated).
- **Assumption:** delete-own-message within a grace window; no editing — unspecified in PRD §6.

## Permissions

- Message send/read and thread create/read for room members only, registered via
  `modules/discussions/permissions.ts` (guidelines §9). Socket joins enforce the same capability
  as HTTP reads — one authorization source.

## Validation Rules

- Message body required, length-bounded, output-encoded on render (XSS defense, guidelines §8.4);
  thread titles length-bounded.
- Rate limiting on sends per user (abuse prevention, guidelines §8.8).
- Pagination params bounded with hard maximums; membership resolved server-side (guidelines §1.1).

## Edge Cases & Failure Scenarios

- Send while disconnected → optimistic UI rolls back or retries; duplicates prevented by
  client-generated idempotency on the send path (guidelines §4.5).
- Member removed mid-session → socket room eviction; further sends rejected.
- Burst traffic in a large room → broadcast fan-out is O(room members) only; history reads never
  unbounded (guidelines §10).
- Archived room → read-only history; sends rejected with a typed conflict.

## Acceptance Criteria

- Members exchange messages live in the main stream and in threads; non-members can neither read
  nor send by any route (PRD §6; guidelines §8.2).
- History paginates with totals; reconnect shows no gaps or duplicates (guidelines §5).
- Thread replies produce "Discussion Replies" notifications for thread participants once E5.4 is
  live (PRD §14).
