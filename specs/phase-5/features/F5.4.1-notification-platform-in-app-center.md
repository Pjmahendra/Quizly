# F5.4.1 — Notification Platform & In-App Center

**Epic:** E5.4 — Notifications · **Complexity:** Large

## Description

The notification backbone (PRD §14) and the user-facing Notification Center (PRD §4.4): a
persistent, org-scoped notification entity with read/unread state; a center UI with unread badge,
paginated history, mark-read and mark-all-read; and live in-app delivery to connected users over
Socket.IO. This feature builds the platform and its delivery path; producing notifications from
domain events is F5.4.2, and the email channel is F5.4.3. Domain home is `modules/notifications/`
(folder-structure §4).

## Dependencies

- Phase 1 (users, tenancy — recipients are org-scoped users; platform-wide sends span orgs,
  PRD §4.1).
- `server/socket/` infrastructure for live delivery; `server/cache/` for the unread-count key.
- No dependency on Phases 3–4; consumed by F5.4.2/F5.4.3.

## Database Changes (conceptual)

A notification entity: recipient, org scope (nullable for platform-wide, PRD §4.1), type (from a
typed trigger vocabulary), title/body, a reference to the originating entity (for deep-linking),
read flag with read-at, created-at. Indexed for per-recipient reverse-chronological paging and
unread counting. Data lifecycle defined up front — notifications grow unbounded (guidelines §6):
**Assumption:** retention window with archival/deletion, exact window an open question.

## Backend Changes

- `modules/notifications/`: actions/services/repositories/validators/dto/`permissions.ts`.
- Creation API (internal, called by F5.4.2's fan-out): persist → push live if recipient connected
  → invalidate unread-count cache.
- Read services: paginated list, unread count, mark-read, mark-all-read.
- New cache key in `server/cache/keys.ts` (the single key catalog, folder-structure §9) for
  per-user unread counts — **addition to the cache-key catalog**.

## Frontend Changes

- Notification center in `modules/notifications/components/`: bell + unread badge in the shared
  navigation, dropdown/panel with paginated history, per-item and mark-all read actions,
  deep-links to the originating surface — composed into **every** portal layout (students and
  faculty both receive notifications, PRD §14; Super Admin/org portals included).
- Live badge/list update on socket delivery; graceful degradation to fetch-on-load when the
  socket is unavailable (guidelines §13).

## Socket Events

- **Proposed addition to catalog:** `notification-created` — server → a user-scoped socket room,
  carrying the new notification for live in-app display. Registered in `modules/socket/events/`
  with a handler in `modules/socket/handlers/`; user-room join uses `server/socket/` primitives
  (folder-structure §5). No existing catalog event covers user-directed delivery.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- List my notifications (paginated, filter unread); get unread count.
- Mark one notification read; mark all read.

## Permissions

- Recipients read and mutate only their own notifications — enforced at the repository seam
  (guidelines §8.2); capabilities registered via `modules/notifications/permissions.ts`
  (guidelines §9). Creation is internal-only this feature (Super Admin sends arrive in F5.4.2).

## Validation Rules

- Notification type from the typed vocabulary only; title/body length-bounded; reference ids
  validated on creation (guidelines §8.4).
- Pagination bounded with hard maximums (guidelines §5); mark-read idempotent (guidelines §4.5).

## Edge Cases & Failure Scenarios

- Recipient offline at creation → no socket delivery; the center shows it on next load (persist
  first, push second — push failure never loses a notification).
- Mark-all-read racing new arrivals → only notifications existing at request time are marked.
- Unread-count cache stale/unavailable → recomputed from the source of truth (guidelines §10).
- Deep-link target since deleted (resource removed, room gone) → typed not-found surface, not a
  crash (guidelines §7).

## Acceptance Criteria

- Every portal shows the center with an accurate unread badge; history paginates; read state
  persists across sessions and devices (PRD §4.4; guidelines §5).
- A connected user sees a new notification live via `notification-created` without refresh; an
  offline user sees it on next load.
- No user can read or mutate another user's notifications (guidelines §8.2).
