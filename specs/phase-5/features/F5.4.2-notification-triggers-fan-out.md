# F5.4.2 — Notification Triggers & Fan-Out

**Epic:** E5.4 — Notifications · **Complexity:** Large

## Description

Wire every PRD §14 trigger into the F5.4.1 platform. Student triggers: Quiz Scheduled, Quiz
Starting Soon, Quiz Published, Results Published, Resource Added, Classroom Announcement,
Discussion Replies. Faculty triggers: Student Joined, Quiz Completed, Evaluation Finished, Student
Concern Raised. Plus Super Admin **platform-wide notifications** (PRD §4.1). A single trigger
registry maps domain events → audience resolution → notification creation, with fan-out to large
audiences running in background jobs (BullMQ, PRD §17; guidelines §10). Triggers reference domain
events from Phases 2–4 **where they exist**: this phase's own events (resource added,
announcement, discussion replies) wire immediately; quiz/evaluation/join/concern triggers are
registered in the registry and connected as their producing phases land.

## Dependencies

- F5.4.1 (platform). E5.1–E5.3 domain events for this phase's triggers.
- Phases 2–4 domain events for classroom-join, quiz, results, evaluation, and concern triggers —
  wiring deferred per event until its producer exists.
- BullMQ/Redis (PRD §17). **Assumption:** a `server/queue/` adapter (BullMQ) is added alongside
  the existing server adapters — folder-structure §5 does not yet list a queue home; job names
  come from a shared catalog (guidelines §5).

## Database Changes (conceptual)

The typed trigger vocabulary (extends F5.4.1's notification type). Platform-wide sends need a
broadcast record authored by the Super Admin (content, audience = all, sent-at, sender) from
which per-user notifications materialize. No other new entities — audiences resolve from existing
enrollment/membership relations.

## Backend Changes

- `modules/notifications/`: trigger registry (event type → audience resolver → notification
  template), enqueue-based fan-out worker, idempotent job handling (a retried job never
  double-notifies — guidelines §4.5).
- Audience resolvers consult owning modules via their public interfaces: enrollment from
  `modules/classrooms`, thread participants from `modules/discussions` (folder-structure §4).
- "Quiz Starting Soon" requires **scheduled** (delayed) jobs keyed to quiz schedule times —
  registered now, activated with the quiz phases.
- Super Admin broadcast service with chunked fan-out across orgs.

## Frontend Changes

- Super Admin: platform-wide notification composer + send history in
  `modules/notifications/components/`, composed into `app/admin/` (PRD §4.1).
- Recipient display reuses the F5.4.1 center; each trigger type gets copy and a deep-link target.

## Socket Events

- Delivery rides F5.4.1's **proposed** `notification-created`; no further catalog additions.
  Fan-out jobs must not emit per-recipient events synchronously on the producing request path
  (guidelines §10).

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Super Admin: create a platform-wide notification; list past broadcasts (paginated).
- All other triggers are internal event-driven flows — no public endpoints.

## Permissions

- Platform-wide broadcast capability for Super Admin only, registered via
  `modules/notifications/permissions.ts` (PRD §4.1; guidelines §9). Trigger execution is
  system-internal; audience resolution enforces tenancy — a notification never crosses org
  boundaries except explicit platform-wide broadcasts (guidelines §8.2).

## Validation Rules

- Trigger payloads validated against per-type schemas before fan-out (guidelines §5, §8.4).
- Broadcast content length-bounded and output-encoded.
- Fan-out jobs carry idempotency keys; audience queries are paginated/chunked — never one
  unbounded read (guidelines §6, §10).

## Edge Cases & Failure Scenarios

- Live-quiz bursts (many quiz-completed events at once) → queue absorbs the spike; workers rate
  themselves; the producing request never blocks on fan-out (guidelines §10).
- Job retry after partial fan-out → idempotency prevents duplicate notifications.
- Audience shrinks between event and job execution (student unenrolled) → resolver works from
  execution-time state.
- Producing event exists but E5.4 deployed later → no retroactive notifications
  (**Assumption**).

## Acceptance Criteria

- Every PRD §14 trigger whose producer exists creates notifications for exactly its audience;
  each trigger type deep-links to the right surface.
- Super Admin broadcasts reach all users, chunked through the queue (PRD §4.1).
- A live-quiz-scale burst produces no request-path slowdown and no duplicate or cross-tenant
  notifications (guidelines §8.2, §10).
