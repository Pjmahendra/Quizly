# F5.4.3 — Email Delivery via Resend

**Epic:** E5.4 — Notifications · **Complexity:** Medium

## Description

The email channel for notifications: "Emails are delivered using Resend" (PRD §14, §17). Selected
triggers from F5.4.2 additionally send an email through the `server/mail/` adapter (`resend.ts`,
`mail.ts`, templates under `templates/{faculty,student,system}/` — folder-structure §5).
**Needs clarification:** PRD §14 does not say which triggers are email-worthy vs. in-app only —
the trigger→channel mapping ships configurable, with a conservative proposed default (scheduling,
results, and announcements email; chat-level noise does not). **Assumption:** a minimal per-user
opt-out — one "email notifications on/off" preference — since preference controls are unspecified.
Templates cover only the PRD §14 trigger list (see phase exclusions); auth emails remain Phase 1's.

## Dependencies

- F5.4.1 (platform) and F5.4.2 (triggers — email is a second channel on the same fan-out).
- Resend account/API key via `lib/env.ts` / `config/mail.ts` (folder-structure §7); the queue
  adapter from F5.4.2 for asynchronous sending.
- Phase 1 profile preferences home for the opt-out toggle (F1.4.5 surface).

## Database Changes (conceptual)

A per-user email opt-out preference (default: opted in — **Assumption**). A per-notification email
delivery status (queued/sent/failed) for observability and retry bookkeeping — prose-level; exact
shape decided at implementation.

## Backend Changes

- `server/mail/`: Resend client (`resend.ts`), send facade (`mail.ts`), one template per PRD §14
  trigger in the audience-appropriate folder; `server/` never imports `modules/`
  (folder-structure §5) — the notifications module calls the mail facade, not vice versa.
- `modules/notifications/`: channel resolution (trigger→channel mapping + user opt-out) inside
  the fan-out flow; email sends enqueued as background jobs with retry/backoff — never on the
  request path (guidelines §10; PRD §16 retry mechanisms).

## Frontend Changes

- Email opt-out toggle on the profile/preferences surface (Phase 1's profile management,
  `modules/users/components/`) — a single switch this phase.

## Socket Events

None. Email is asynchronous by nature; in-app delivery is F5.4.1's concern.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Read/update my email notification preference (rides the Phase 1 profile-preferences surface).
- No public send endpoints — sending is internal to fan-out.

## Permissions

- Users mutate only their own preference (guidelines §8.2). No new capabilities beyond profile
  self-management; send authority is system-internal.

## Validation Rules

- Emails send only to verified addresses from the user record (Phase 1 verification) — never to
  client-supplied addresses (guidelines §1.1).
- Template variables encoded for HTML email context (injection defense, guidelines §8.4).
- Trigger→channel mapping validated at startup from typed config; opt-out honored at enqueue time
  **and** re-checked at send time.

## Edge Cases & Failure Scenarios

- Resend outage/rate limit → jobs retry with backoff; permanent failure marks delivery failed and
  is logged — the in-app notification is unaffected (channels are independent).
- User opts out between enqueue and send → send-time re-check suppresses it.
- Burst fan-out emailing a large classroom → batched/paced through the queue to respect provider
  limits (guidelines §10).
- Bounces/complaints → out of MVP scope beyond logging (**Assumption**); revisit with
  clarification.

## Acceptance Criteria

- Triggers mapped to email produce a Resend delivery to opted-in recipients, using the correct
  audience template; in-app delivery is never blocked by email failures (PRD §14).
- Opted-out users receive no notification email but still receive auth/system emails owned by
  Phase 1.
- No email send occurs on a user-facing request path (guidelines §10).
