# F5.3.4 — Peer Profile Visibility

**Epic:** E5.3 — Discussion Rooms · **Complexity:** Small

## Description

Students can view peer profiles "subject to organization privacy policies" (PRD §4.4). In
collaboration surfaces — discussion room member lists, chat messages, classroom rosters where they
exist — a peer's name links to a small profile view. **Needs clarification:** the privacy-policy
model is entirely unspecified in the PRD — no policy entity, no field-level rules, no defaults.
Until it is defined, this feature ships a deliberately minimal MVP. **Assumption:** the MVP
exposes only display name and portal role to users who share a context (same discussion room or
same classroom), controlled by a single org-level on/off toggle standing in for the future policy
model. Profile data lives in `modules/users/`; the surface composes it via its `index.ts`.

## Dependencies

- Phase 1 (user identity/profiles, org settings home for the toggle).
- F5.3.1 (shared-room context is the primary consumer); Phase 2 classroom rosters where present.
- Profile images are excluded (Phase 1 exclusion — they arrive on the storage platform later; the
  MVP profile shows an initials avatar).

## Database Changes (conceptual)

An org-level setting for peer-profile visibility (the stand-in toggle). No new profile fields.
The full privacy-policy model — whatever it becomes — is explicitly deferred.

## Backend Changes

- `modules/users/`: a peer-profile read service returning the minimal field set, authorized by
  shared-context membership (same room or classroom) plus the org toggle — evaluated
  server-side, never trusted from the client (guidelines §8.2).
- The shared-context check consults `modules/discussions` / `modules/classrooms` via their public
  interfaces (folder-structure §4).

## Frontend Changes

- Peer-profile popover/card in `modules/users/components/`, invoked from member lists and chat
  author names in discussion and classroom surfaces; hidden entirely when the toggle is off.

## Socket Events

None.

## AI Components

None this phase.

## API Endpoints (purpose-level)

- Fetch a peer's minimal profile (authorized by shared context + org toggle).

## Permissions

- A peer-profile view capability registered via `modules/users/permissions.ts` (guidelines §9);
  granted through shared-context membership, gated by the org setting. Self-profile management
  remains Phase 1's F1.4.5 — untouched here.

## Validation Rules

- Target user id resolved within the caller's org (guidelines §1.1); shared context verified
  server-side on every request; response contains only the allow-listed minimal fields — nothing
  else serializes (guidelines §8.7).

## Edge Cases & Failure Scenarios

- Toggle disabled after profiles were viewed → subsequent requests denied; no cached leakage.
- Shared context dissolves (leaves room / unenrolled) → access ends with it.
- Deactivated peer → profile request returns not-found, not stale data.
- Denial rendered as "profiles unavailable", distinct from load failure (guidelines §9, §13).

## Acceptance Criteria

- With the toggle on, a student can open a minimal profile for peers sharing a room or classroom;
  with it off, no peer profile is reachable by any route (PRD §4.4; guidelines §8.2).
- Only the allow-listed fields ever appear in responses — verified at the contract (guidelines §5).
- No cross-org or no-shared-context profile access is possible.

## Definition of Done

Meets `rules/definition-of-done.md`; documentation per CLAUDE.md (feature log entry) including an
explicit record that the MVP is a placeholder for the unspecified policy model.

## Open Questions

- **Needs clarification (blocking full build):** the organization privacy-policy model — policy
  granularity (org/role/field level), who administers it, defaults, and whether students can
  restrict their own visibility.
- Should faculty viewing student profiles follow the same rules or a separate capability?
