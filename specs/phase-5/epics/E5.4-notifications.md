# E5.4 — Notifications

## Purpose

Build the platform's notification backbone (PRD §14): an org-scoped notification entity with
read/unread state and a notification center UI (PRD §4.4); trigger wiring so every PRD §14 event —
student triggers (Quiz Scheduled, Quiz Starting Soon, Quiz Published, Results Published, Resource
Added, Classroom Announcement, Discussion Replies) and faculty triggers (Student Joined, Quiz
Completed, Evaluation Finished, Student Concern Raised) — plus Super Admin platform-wide
notifications (PRD §4.1) produce in-app notifications; and an email channel delivered via Resend
(PRD §14, §17). Code homes: `modules/notifications/` for the domain, `server/mail/` for the Resend
adapter and templates (folder-structure §4–5).

## Business Value

Notifications are the connective tissue of every other feature: resources, announcements,
discussions, and (in later phases) quizzes only feel alive if their events reach users. PRD §4.4
names the Notification Center as a student portal feature and PRD §14 enumerates the contract this
epic must satisfy. A single platform with one fan-out path prevents each domain from growing its
own ad-hoc delivery code.

## Dependencies

- Phase 1: tenancy, permissions, and user identity (recipients, roles, Super Admin).
- Phase 2: classrooms/enrollment (audience resolution for classroom-scoped triggers).
- E5.1–E5.3 in this phase produce the Resource Added, Classroom Announcement, and Discussion
  Replies triggers; **quiz and evaluation triggers reference Phases 2–4 domain events where they
  exist** — wiring for not-yet-built events lands with those phases (F5.4.2).
- Socket infrastructure for live in-app delivery; a new catalog event is required
  (**Proposed addition to catalog:** `notification-created`, see F5.4.1).
- BullMQ/Redis (PRD §17) for background fan-out; Resend account/config for F5.4.3.

## Acceptance Criteria

- Every user has a notification center with unread badge, read/unread and mark-all-read,
  paginated history, and live delivery while connected (PRD §4.4; guidelines §5).
- Every PRD §14 trigger whose producing event exists creates in-app notifications for exactly its
  intended audience — never cross-tenant, never to non-members (guidelines §8.2).
- Super Admin can publish platform-wide notifications reaching all users (PRD §4.1).
- Fan-out to classroom-sized audiences runs off the request path via background jobs
  (guidelines §10); notification volume has a defined lifecycle/retention (guidelines §6).
- Emails are delivered via Resend through `server/mail/` for the triggers designated as email
  (pending **Needs clarification** on the email/in-app split), honoring the minimal opt-out
  (F5.4.3 **Assumption**).
