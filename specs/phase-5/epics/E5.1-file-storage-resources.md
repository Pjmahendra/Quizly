# E5.1 — File Storage & Resources

## Purpose

Deliver the file-storage platform (PRD §9) and the classroom resource lifecycle built on it:
a provider abstraction that runs against Azurite today and Azure Blob Storage later without caller
changes, a single path catalog implementing the org-scoped blob hierarchy, faculty upload and
management of learning resources, and student consumption (view / download / bookmark). Code homes
are `modules/storage/` and `modules/resources/`, with the low-level client in `server/storage/`
(folder-structure §5, §10).

## Business Value

Resource sharing is a named pillar of the product (PRD §1, §3, §5, §9): classrooms only become a
daily destination when course material lives there. The provider abstraction protects the
production migration path (Azurite → Azure Blob, PRD §9 "Storage Strategy") and the path catalog
gives organization-level isolation, cleaner permissions, and straightforward backup/migration —
the exact benefits PRD §9 claims for the hierarchy. Later phases (profile images, quiz
attachments, submissions) reuse this platform instead of rebuilding it.

## Dependencies

- Phase 1: tenant scoping context and permission catalog (all resources are org-scoped).
- Phase 2: classrooms and enrollment (resources attach to classrooms; consumption is
  enrollment-gated).
- Azurite running in local Docker (CLAUDE.md); no dependency on Phases 3–4.
- Within the epic: F5.1.2 and F5.1.3 depend on F5.1.1; F5.1.3 depends on F5.1.2.

## Acceptance Criteria

- Every blob path in the system is built by `modules/storage/paths.ts` — no other code constructs
  a path (folder-structure §9–10); swapping Azurite for Azure Blob changes a provider, never a
  caller (PRD §9).
- Faculty can upload, replace, and delete all PRD §9 resource types (PDF, PPT, images, Word
  documents, external links, course notes, recorded lectures) in their classrooms, with allow-list
  type validation and bounded sizes (guidelines §8.4).
- Students in an enrolled classroom can view, download, and bookmark resources; non-members cannot
  reach a resource or its blob by any route (guidelines §8.2).
- Open classroom surfaces receive new resources live via the catalog `resource-added` event
  (folder-structure §5).
- Uploads, replacements, and deletions are audit-logged (guidelines §8.8); resource lists paginate
  (guidelines §5).
- Storage quota enforcement has a single reserved seam, pending quota definition
  (**Needs clarification**, see F5.1.2).
