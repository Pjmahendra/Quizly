# E5.3 — Discussion Rooms

## Purpose

Give students independently created collaboration spaces (PRD §6): room lifecycle and
invitation-based membership (faculty join only if invited), chat with discussion threads and
paginated history, resources shared inside the room (reusing E5.1), and peer profile visibility
subject to organization privacy policies (PRD §4.4). Code home is `modules/discussions/`, with
real-time delivery through `modules/socket/` handlers over the `server/socket/` primitives
(folder-structure §4–5).

## Business Value

Discussion rooms serve group study, topic discussions, peer learning, and collaborative
preparation (PRD §6) — the "classroom collaboration" differentiator of PRD §1. Student-owned
spaces increase daily engagement beyond scheduled assessments, and "Discussion Replies" is a named
student notification trigger (PRD §14), feeding the E5.4 platform.

## Dependencies

- Phase 1: tenancy and permissions (rooms are org-scoped; membership is the access boundary).
- Phase 2: the student body and portal surfaces students already inhabit.
- Socket infrastructure (`server/socket/` bootstrap, auth middleware, room primitives) from the
  real-time phase; F5.3.2 uses the existing catalog event `discussion-message`
  (folder-structure §5). No dependency on quiz Phases 3–4.
- F5.3.3 reuses E5.1 (F5.1.1/F5.1.2) for storage; it extends the path catalog with a discussions
  node via `paths.ts` only.
- F5.3.4 is gated on an unspecified privacy-policy model (**Needs clarification**) and ships a
  minimal-assumption MVP.

## Features

| ID | Feature | Complexity |
|---|---|---|
| F5.3.1 | Discussion room lifecycle & membership | Large |
| F5.3.2 | Discussion chat & threads | Large |
| F5.3.3 | Discussion shared resources | Medium |
| F5.3.4 | Peer profile visibility | Small |

## Acceptance Criteria

- A student can create a room, invite students, and manage membership; faculty can participate
  only when invited; non-members can reach nothing of the room (PRD §6; guidelines §8.2).
- Members exchange chat messages and thread replies delivered live via the catalog
  `discussion-message` event, with paginated history and a defined retention/lifecycle policy
  (guidelines §5, §6).
- Members share resources into the room, stored under the room's path-catalog node and gated by
  membership (PRD §6).
- Peer profiles expose only the minimal assumed field set until the org privacy-policy model is
  clarified (PRD §4.4).
- Thread replies produce "Discussion Replies" notifications once E5.4 is live (PRD §14).
